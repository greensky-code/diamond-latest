Member Conditions, How DIAMOND Thinks

DIAMOND logic supports two types of member conditions:

  • Privacy - a keyword MCOND (Member Conditions) member Condition Type of Privacy represents "privacy by diagnosis" functionality

  • Pre-Existing Condition - a keyword MCOND member Condition Type of Pre-Existing, Rated-up, or Ridered-out represents "pre-existing condition" functionality

DIAMOND evaluates each type of member condition record at different times and in a different manner. For information see the following two sections:

  • Evaluating Privacy Member Condition Records

  • Evaluating Pre-Existing Condition Member Condition Records

Evaluating Privacy Member Condition Records

DIAMOND logic evaluates privacy member condition records during:

  • Institutional claims processing

  • Professional claims processing

  • Dental claims processing

  • Authorizations processing

Evaluating Privacy Member Condition Records in Institutional Claims

For online claims processing, DIAMOND logic evaluates privacy member condition records when you:

  • Save the claim header for the first time

  • Change one or more diagnosis codes on the claim header and save, but only if no claim detail lines have a Proc Status equal to F - Final (posted), P - Paid, N - Paid (no check), or C - Check Issued (paid)

  • Use Special menu command Change Header to change the Member ID on the claim header and save

  • Use Topic menu command UB92 to change one or more diagnosis codes and save

For DDS API processing, DIAMOND logic evaluates privacy member condition records when you:

  • Post a claim

During privacy member condition record evaluation, DIAMOND logic compares the claim header data to each privacy member condition record looking for the first record to satisfy all of the following:

  • Claim header Member ID matches the privacy member condition record Subscriber ID + Person No.

  • Either the claim header Admit Date or Disch Date are within the Effective Date/Term Date range of the privacy member condition record

  • One of the diagnosis codes on the claim header or Topic menu command UB92 falls within the privacy member condition record From Value/Thru Value diagnosis code range

Based on the results of the evaluation, DIAMOND logic proceeds as follows:

  • When a privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the claim header Privacy Applies box to Y and saves the seq_memb_cond_id to the claim header table.

  • If no privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the claim header Privacy Applies box to N.

Evaluating privacy member condition Records in Professional Claims

For online claims processing, DIAMOND logic evaluates privacy member condition records when you:

  • Save the claim header for the first time

  • Change one or more diagnosis codes on the claim header and save, but only if no claim detail lines have a Proc Status equal to F - Final (posted), P - Paid, N - Paid (no check), or C - Check Issued (paid)

  • Use Special menu command Change Header to change the Member ID on the claim header and save

  • After pricing, medical definitions evaluation, and adjudication of a claim have been completed

For DDS API processing, DIAMOND logic evaluates privacy member condition records when you:

  • Post a claim

During privacy member condition record evaluation, DIAMOND logic compares the claim header data to each privacy member condition record looking for the first record to satisfy all of the following:

  • Claim header Member ID matches the privacy member condition record Subscriber ID + Person No.

  • Either the claim header Svc Date or Thru Date are within the Effective Date/Term Date range of the privacy member condition record

  • One of the diagnosis codes on the claim header falls within the privacy member condition record From Value/Thru Value diagnosis code range

Based on the results of the evaluation, DIAMOND logic proceeds as follows:

  • When a privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the claim header Privacy Applies box to Y and saves the seq_memb_cond_id to the claim header table.

  • If no privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the claim header Privacy Applies box to N.

Evaluating privacy member condition Records in Dental Claims

For online claims processing, DIAMOND logic evaluates privacy member condition records when you:

  • Save the claim header for the first time

  • Change one or more diagnosis codes on the claim header and save, but only if no claim detail lines have a Proc Status equal to F - Final (posted), P - Paid, N - Paid (no check), or C - Check Issued (paid).

  • Use Special menu command Change Header to change the Member ID on the claim header and save

  • Use Topic menu command Other Claim Info to change one or more diagnosis codes and save

  • After pricing, medical definitions evaluation, and adjudication of a claim have been completed

For DDS API processing, DIAMOND logic evaluates privacy member condition records when you:

  • Post a claim

During privacy member condition record evaluation, DIAMOND logic compares the claim header data to each privacy member condition record looking for the first record to satisfy all of the following:

  • Claim header Member ID matches the privacy member condition record Subscriber ID + Person No.

  • Either the claim header Svc Date or Thru Date are within the Effective Date/Term Date range of the privacy member condition record

  • One of the diagnosis codes on the claim header or Topic menu command Other Claim Info falls within the privacy member condition record From Value/Thru Value diagnosis code range

Based on the results of the evaluation, DIAMOND logic proceeds as follows:

  • When a privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the claim header Privacy Applies box to Y and saves the seq_memb_cond_id to the claim header table.

  • If no privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the claim header Privacy Applies box to N.

Evaluating privacy member condition Records in Authorizations

For online and DDS API authorization processing, DIAMOND logic evaluates privacy member condition records when you:

  • Save an authorization for the first time

  • Change the Member ID or Req Date on an authorization and save

During privacy member condition record evaluation, DIAMOND logic compares the authorization data to each privacy member condition record looking for the first record to satisfy all of the following:

  • Authorization Member ID matches the privacy member condition record Subscriber ID + Person No.

  • For online authorization processing, the authorization Req Date falls within the Effective Date/Term Date range of the privacy member condition record

  • For DDS API inpatient authorization processing, the authorization Admit Date falls within the Effective Date/Term Date range of the privacy member condition record

  • For DDS API non-inpatient authorization processing, the authorization Request Date falls within the Effective Date/Term Date range of the privacy member condition record

DIAMOND logic determines whether an authorization is inpatient or non-inpatient based on the authorization type. You maintain authorization types using keyword AUTYP (Authorization Type). For more information, see AUTYP Field Definitions.

Based on the results of the evaluation, DIAMOND logic proceeds as follows:

  • When a privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the authorization Privacy Exists box to Y.

  • If no privacy member condition record is found that satisfies all the criteria, DIAMOND logic sets the authorization Privacy Exists box to N.

Using the Privacy Address for Pay to Subscriber and Pay to Dependent Claims

During accounts payable processing, DIAMOND logic determines whether to use the privacy address as the vendor address for a claim payment. The privacy address is only used for Pay to Subscriber or Pay to Dependent claims and only under certain conditions.

You enter a privacy address for a member using keyword ALIAS (Alias/Responsible Party/Privacy). For more information, see Adding a member privacy address.

Pay to Subscriber claims where member is the subscriber

DIAMOND uses the member's privacy address as the vendor address for Pay to Subscriber claims if all of the following are true:

  • Member is the Subscriber

  • Claim header Pay Sub box = Y

  • Claim header Privacy Applies box = Y

  • A privacy address exists for the member

If the other criteria are met but no privacy address exists for the member, DIAMOND logic uses the member's address from the Member Master table. If no member address exists, the vendor address is blank.

Pay to Subscriber claims where member is a dependent

DIAMOND does not use the privacy address as the vendor address for Pay to Subscriber claims if the following are true:

  • Claim header Pay Sub box = Y

  • Claim header Privacy Applies box = Y

  • A privacy address exists for the member or subscriber

but

  • Member is a dependent, not the subscriber

Here DIAMOND logic instead uses the subscriber's member address as the vendor address for this Pay to Subscriber claim. The subscriber's privacy address is reserved for the subscriber's own Pay to Subscriber claims where privacy applies. It is not used for any dependent's Pay to Subscriber claims.

Pay to Dependent claims

DIAMOND uses the member's privacy address as the vendor address for Pay to Dependent claims if all of the following are true:

  • Member is a dependent

  • Claim header Pay Dep box = Y

  • Claim header Privacy Applies box = Y

  • A privacy address exists for the member

If the other criteria are met but no privacy address exists for the member, then DIAMOND logic uses the member's address from the Member Master table. If no member address exists, the vendor address is blank.
