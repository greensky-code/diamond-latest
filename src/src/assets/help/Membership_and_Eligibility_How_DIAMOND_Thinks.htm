Changes to member records

It is critical that changes to member demographic and eligibility records be made using the appropriate DIAMOND feature. In most cases, changes to eligibility must be made using one of the many commands on the menu bar found in MEMBR. DIAMOND does include changes made either in MEMBR or MELIG in all processing, including Capitation and Premium Billing, but some limitations apply to MELIG changes.

Specifically, if changes are made in directly in MELIG rather than using commands on the Special menu in keyword MEMBR the following limitations apply:

  • Changes made to a subscriber never ripple through to dependents, regardless of the Dependent Edit flag on line of business (LINBS).

  • Changes are made to DIAMOND‘s eligibility file only. Other records containing eligibility information, such as authorizations or claims, are not updated.

  • Member ID cards are not generated automatically as a result of the change.

  • Provider changes made using MELIG are not validated against DIAMOND’s provider/member enrollment functionality (PROVE) records.

There are, however, some changes which must be made using MELIG:

  • Correcting the r desubscriber or dependent Plan Code because the Plan Code is wrong.

  • Updating the pre-existing condition waived check box (PEC Waived)

  • Updating the pre-existing condition end date (PEC End Date)

  • Updating the pre-existing condition reason (Reason)

  • Updating the usefined fields (UserDef 1-3)

Enforcement of subscriber/dependent eligibility consistency

You may elect to relax the standard edits DIAMOND uses to make sure that dependents cannot have eligibility and benefits which are outside of or different from the subscribers. In most commercial health plans, this consistency is critical. In some situations and programs (e.g. Medicaid, often) the eligibility of dependents can be different from that of the head of household or subscriber. Therefore, by line of business you may choose to eliminate this editing.

When consistency tests are enforced, Diamond requires that all dependents must be enrolled in the same group and plan as the subscriber. In addition, the dependents’ effective and termination dates must fall within the eligibility period of the subscriber; i.e., dependents are eligible for coverage based on the subscriber’s eligibility dates and the dependents will not be eligible after the subscriber is terminated.

If you choose to relax the edits, consistency tests are not enforced. In addition, be aware that:

  • Premium Billing (GPBIL) cannot be used for this line of business, since billing logic assumes that all members of a family are enrolled in the same benefit plan and that the eligibility periods of the dependents always fall within the eligibility period of the subscriber.

  • Family unit-oriented benefit rules (such a family deductibles) cannot be used for this line of business.

  • MEMBR > Special > Member Address Ripple cannot be used for this line of business.

To manage which kinds of business require consistency, and which do not, use the LINBS record, Dep Edit field.

Overlapping Eligibility Periods Not Allowed

DIAMOND does not allow overlapping eligibility periods for a member.

Online, you cannot enter a member eligibility period with an Effective Date or Term Date that falls within an existing eligibility period for that member.

In Member Batch, if you submit a file that contains two overlapping eligibility records for the same member, DIAMOND processes them without creating overlapping periods. For processing details, see Member Batch, How DIAMOND Thinks>Submitting Multiple Member Records in One Transaction user documentation.
