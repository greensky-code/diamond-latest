In addition to storing the codes and descriptions, you may use PROCD to accomplish a variety of edits and prompts with the entries for each procedure. These additional functions include:

  • Automatically placing a claim detail line item on hold.

  • Allowing or not allowing the procedure based on the member’s age and/or gender.

  • Counting the procedure as a day for benefit calculation and authorization purposes.

  • Warning claims processors than an alternative procedure exists for asterisked procedures and other implementation specific situations.

  • Prompting claims processors to enter elapsed time when an anesthesia code is entered.

  • Grouping procedures into categories for reporting purposes.

  • Entering procedure codes associated with a revenue code.

In DIAMOND, each procedure code is stored as one procedure code record in PROCD. To vary pricing based on price schedule, price region, etc. use keyword PROCP (Procedure Price).

Using Effective and Termination Dates

If you use the effective/termination dates for a procedure DIAMOND applies the following edits:

  • In claims, DIAMOND verifies that the date of service entered on the claim (PSCLM, Detail — Date, INCLM, Detail — Date, DNCLM, Detail — Date) is on or after the procedure code effective date entered and before the procedure code termination date entered in these fields.

  • In authorizations, DIAMOND verifies that the "from" date on the authorization (AUTHS, Surgical Procedures window) on or after the procedure code effective date and before the procedure code termination date entered in these fields.

Allowing Procedures Based on Member’s Age

You can allow or not allow a specific procedure based on the member’s age. You do this by specifying a From Age and To Age for the procedure in PROCD.

Then, when you enter the procedure code on a claim or authorization (INCLM, PSCLM, DNCLM, and AUTHS), DIAMOND validates the age of the member against the age you enter in this field. DIAMOND displays an error message if you attempt to enter a procedure code for a member who does not meet the age specification.

DIAMOND performs the age validation as follows*:

If the PROCD From Age and To Age are whole numbers (for example, 0.0, 2.0, 5.0):

The Claim header primary date of service must fall in the range from (member’s date of birth + number of years in From Age) through (member’s date of birth + number of years in To Age - one day).

That is, the claim header primary date of service must be greater than or equal to (member’s date of birth + number of years in From Age) and less than or equal to (member’s date of birth + number of years in To Age - one day).

If either PROCD From Age or To Age are decimal numbers (for example 4.9, 5.1):

The member’s age as of the claim header primary date of service, calculated to the nearest tenth of a year, must fall in the range from the PROCD From Age through the PROCD To Age.

That is, the member’s age as of the claim header primary date of service, calculated to the nearest tenth of a year, must be greater than or equal to the PROCD From Age and less than or equal to the PROCD To Age.

*Exception: Professional Services, Dental Services, and Institutional claims perform this age validation as detailed above for online processing. Transactions submitted to DIAMOND using DDS (DIAMOND Data Services) may be processed differently. For more information, see the DDS API Guide for the specific claims transaction.

Counting Procedures as Days

Certain benefits, such as coverage for inpatient mental health services, may be limited according to a maximum number of days. Similarly, some authorizations limit the approved services in terms of days, such as a defined number of inpatient days for a particular surgery.

In order for DIAMOND to accumulate rendered services to such benefit and authorized maximums the system must recognize when a procedure code represents a day. This is accomplished by marking those procedure codes in the Count As Days check box in keyword PROCD.

For more information about accumulating days, see reference documentation for AUTHS, Days/Visits field, and BRULE, Type 20, specifically the Limit Type field.

Target Revenue Codes

Some hospital reimbursement may require both a revenue code and its associated CPT code be entered on a claim detail line in order for the line to be priced accurately. In PROCD there are three fields that must be entered to enable this kind of pricing to work.

  • Target Revenue Code is checked so that INCLM will prompt for the CPT code to be entered in the alternate procedure field when the revenue code is entered.

  • Procedure Determinant, accessed with a menu command, contains the CPT codes that are valid for each target revenue code; INCLM edits the CPT code entered to make sure it is valid for the entered revenue code.

  • Price Indicator tells INCLM whether the revenue code or the CPT code is used to price the detail line.
