Xref ID Field

  • There may be some unique business situations where it is necessary to assign more than one provider master record (Provider ID) to one provider. Multiple provider IDs for a single provider can complicate day-to-day processing as you must then decide which ID pertains to a particular transaction. In DIAMOND, each provider’s record, under a single ID number, allows many one-to-one, one-to-many, and many-to-many relationships, using combinations of contract records, address records, and vendor relationships. HSD suggests that you avoid the use of multiple provider IDs.

  • The value entered in this field must be an existing provider ID. If you must maintain more than one provider record for the same provider, you can use this field as a "link" between the provider records.

  • When the cursor is in this field, you can use the F5 lookup function to see if the provider you are entering has another provider record in DIAMOND. If another record for the provider exists in DIAMOND and it has an existing Xref ID, when you select the looked up record as a link, DIAMOND automatically populates the Xref ID field, on the record you are entering, with Xref ID from the looked up record.

  • If another record for the provider exists in DIAMOND and it does not have an existing Xref ID, after you select the looked up record as a link, you must manually add the Xref IDs (opposite provider IDs) to both the existing and the new provider record.

Short Name Field

DIAMOND’s non-institutional provider short name generation scheme:

  • If the provider’s last name is more than 15 characters, the short name will be the first 15 characters of the provider last name. DIAMOND assigns a provider with a last name of "Verylonglastname" a short name of "VERYLONGLASTNAM."

  • If the provider’s last name is less than 15 characters, the short name will be the entire last name followed by the provider’s first name (up to a maximum of 15 characters). DIAMOND assigns provider "Fname Lname" a short name of "LNAMEFNAME." DIAMOND assigns provider "Firstname Lastname" a short name of "LASTNAMEFIRSTNA."

DIAMOND’s institutional provider short name generation scheme:

  • The provider’s short name is the first 15 characters of the provider’s full name. DIAMOND assigns provider "ABCDEFGHIJK Hospital" a short name of "ABCDEFGHIJKHOSP." DIAMOND assigns provider "XYZ Hospital" a short name of "XYZHOSPITAL."

Exclusions to the short name build:

The following characters, or words, are excluded from the short name generation build for Providers and Vendors:

  • A

  • Mr

  • Inc

  • IV

  • The

  • Mrs

  • Incorporated

  • V

  • Jr

  • Co

  • II

 

  • Sr

  • Company

  • III

 

Non-Specific Provider Check Box

In many cases, claims are received from providers who do not have a relationship with the health plan; for example, out-of-area providers who take care of members who are travelling. You can use DIAMOND’s Non-specific provider record as a "proxy" for those one-time or rarely encountered providers. This means you do not have to set up separate provider master file records for these providers. See Provider Contracts, How DIAMOND Thinks for additional information on this functionality.

Using Authorization Class Check Box to Waive Authorizations

In some cases the requirement that a claim must be authorized may be waived if the attending or referring provider is a "participating" provider. (This is in contrast to other authorization waivers that depend on the provider of service.) Because claims logic does not look at the contract information for the attending or referring provider, authorization waivers cannot use any participation status or other contractual indicators. Therefore, in order to mark a provider as "participating" in a way which can be identified when that provider is attending or referring, enter a valid value in the Auth Class field on the Provider Master record. In keyword AWAVE, add a rule using Auth Class as a determinant, and that same valid value as the determinant value.

Specialty Types Field

Because every provider record must have at least one associated specialty record, we recommend you create an "unknown" PTYPE record (e.g., UNK). This UNK value can be used for the provider’s type and specialty in cases where you do not know the provider’s specialty but need to set up provider’s master record.

Designating a default provider record

When many providers are reimbursed using the same pricing rules and schedules you may be able to avoid setting up individual provider contract records for each of them. Instead, you may establish a single provider record with ‘default’ contract characteristics. This provider is called a default provider and is so identified by the parameter record DFTCNTRCTPRV that contains the default provider ID.

In order for DIAMOND to use a default provider’s contract information you must direct use of the default provider in the contract search instructions on the LINBS record. See LINBS reference and How Line of Business Controls Processing user documentation for instructions and examples of contract searches. See PROVC reference user documentation for information about setting up contract information.
