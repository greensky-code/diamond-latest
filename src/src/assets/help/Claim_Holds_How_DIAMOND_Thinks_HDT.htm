Applying User-Defined Claim Header Hold Rules (CLHLD)

Use keyword CLHLD (Claim Hold Rules) to define Hold rules for holding claims at the header level. Claim header Hold rules hold the entire claim, preventing all claim detail line items from being finalized and selected for payment by the Accounts Payable module. (You can also hold entire claims using predefined header Hold rules. For more information, see Applying Predefined Header Hold Rules.)

You can select any claim header field (except date fields) as a determinant for CLHLD header Hold rules. For example, you can hold all claims for a particular Vendor ID.

User-defined claim header Hold rules can be applied either automatically or manually:

  • Online, DIAMOND automatically applies user-defined header Hold rules the first time you move from the Claim header window to the Claim detail window, after performing all other header logic and saving the header.

  • Claims Batch logic automatically applies these rules upon post of the batch.

  • Online, you can also manually apply user-defined header Hold rules using Re-apply Claim Holds command on the Special menu in keywords INCLM, PSCLM, and DNCLM.

DIAMOND applies CLHLD user-defined claim header hold rules as follows.

1. If you are using the INCLM/PSCLM/DNCLM Special menu command Re-apply Claim Holds, DIAMOND first marks all existing CLHLD header hold reason records as historical.

2. DIAMOND then evaluates the claim header data against each rule.

3. If the claim header data does not satisfy the rule criteria, DIAMOND proceeds to the next rule.

4. If the claim header data does satisfy the rule criteria, DIAMOND does the following:

  • Adds a claim header hold record for Line #000, using the Reason Code you specified on the CLHLD rule and other identifying information such as Source ID, the hold rule name specified for that rule prefixed with 'CLHLD'.

  • Selects the Holds check box on the claim header.

5. DIAMOND continues until all rules have been applied.

Note that all CLHLD rules are always evaluated, and each rule that is satisfied creates a separate hold record on the claim.

You can view and modify hold records using the Hold Reasons command on the Topic menu in keywords INCLM, PSCLM, and DNCLM. You can also view hold records using the Hold Reason Display command on the Window menu in keywords INCLM, PSCLM, DNCLM, and CLDSP.

Important considerations:

  • User-defined CLHLD header holds do not modify the Processing Status and Hold Reason codes on the claim detail line items. (Note that predefined header holds, for example, member holds or group holds, do change the Processing Status to H – Held for all detail lines on the claim.)

  • CLHLD header holds do not prevent you from pricing and adjudicating claim detail line items.

  • CLHLD header holds do not prevent claim detail line holds and denials from being applied.

  • If a CLHLD automatic header hold rule is added or changed, DIAMOND automatically applies the new rule to subsequently entered claims. However, existing claims are not subject to new or revised rules unless you manually invoke the INCLM/PSCLM/DNCLM Special menu command Re-apply Claim Holds.

  • Header holds must be manually removed. For more information, see Release a held claim.

  • When you manually apply a user-defined header hold rule using the INCLM/PSCLM/DNCLM Special menu command Re-apply Claim Holds, finalized claim line items are unaffected.

  • You cannot have more than one active header hold reason code record with the same Source ID and Hold Reason Code. You can have additional header hold reason records with the same Source and Hold Reason Code as an active record only when all other records are marked historical.

  • Automatic header holds apply to claims entered online in keywords INCLM, PSCLM, and DNCLM, or entered using Claims Batch or Claims API.

  • A claim may have multiple different hold reason codes applied to a single detail line or to a header. A claim may also have different hold reason codes assigned to a detail line than to the header.

Applying Predefined Header Hold Rules

Several predefined header Hold rules are available for your use in DIAMOND. For example, you can use predefined header Hold rules to hold claims for a member or a group.

Predefined header holds are applied differently than CLHLD user-defined header holds. While CLHLD header holds create a single hold reason record for Line #000, pre-defined header holds instead create a separate hold reason record for each detail line on the claim. However, both are based on claim header field values and both in effect hold an entire claim. For a listing of hold sources, including all predefined header Hold rules, and references to user documentation describing the criteria and logic for each, see Claim Holds: Sources and Effects.

DIAMOND applies predefined claim header hold rules as follows (this does not include CLHLD user-defined hold rules).

1. DIAMOND evaluates the claim header data against the rule.

2. If the claim header data does not satisfy the rule criteria, then DIAMOND continues claim processing.

3. If the claim header data does satisfy the rule criteria, then DIAMOND does the following:

  • Sets the Proc Status list on each claim detail line on the claim to H - Held.

  • Selects the Holds check box on each detail line item.

  • Adds a hold reason record for each claim detail line item on the claim, using the Reason code you specified on the CLHLD rule and including other identifying information such as Source ID. (The Source ID assigned varies by rule.). (DIAMOND does not write a header hold reason record for Line #000 for predefined header holds.)

  • You can view and modify hold records using the Hold Reasons command on the Topic menu in keywords INCLM, PSCLM, and DNCLM. You can also view hold records using the Hold Reason Display command on the Window menu in INCLM, PSCLM, DNCLM, and CLDSP.

4. Selects the Holds check box on the claim header window.

5. DIAMOND continues until all rules have been applied.

Holding Claims and Accounts Payable Processing

Claim holds prevent Accounts Payable from finalizing and selecting claims for payment. This impacts either entire claims or individual claim detail lines depending upon the specific hold. This works as follows:

  • When a CLHLD user-defined header hold is applied, DIAMOND creates a single hold reason record for Line #000. While this hold is active, all claim detail lines on the claim are prevented from being finalized and selected for payment by Accounts Payable.

  • You can also manually enter a header hold reason record for Line #000 on a claim using the INCLM/PSCLM/DNCLM Topic Hold Reasons. While this hold is active, all claim detail line items on the claim are prevented from being finalized and selected for payment by Accounts Payable.

  • When one of the predefined header holds is applied, DIAMOND creates a separate hold reason record for each claim detail line on the claim and changes the Processing Status to H - Held. While these holds are active, the claim detail line items are prevented from being finalized and selected for payment by Accounts Payable.

  • When a detail line hold is applied, DIAMOND creates a hold reason record for the current detail line and sets the Proc Status list to H - Held. While the hold is active, the detail line cannot be finalized and selected for payment by Accounts Payable. In addition, if the system parameter PARTIALPAY is set to "do not allow partial payment", all other detail line items on the same claim are also prevented from being finalized and selected for payment by Accounts Payable.

Holds and Matched Claim/Authorization Processing

You can hold or deny an institutional, professional services, or dental claim when it is matched to an authorization and a third party liability value (WC, TP, AA) is selected in the TPL Code list in keyword AUTHS (Authorization Master). During claims processing (online, batch, and API) DIAMOND uses the authorization’s TPL Code in conjunction with the PARAM - TPLCODEACT parameter (1 and/or 2 and/or 3) and:

IF

Then

TPLCODEACT = Action code entered and reason code entered.

As specified in the parameter, DIAMOND takes the hold or deny action and assigns the reason code to all claim detail lines.

TPLCODEACT = No action code entered and no reason code entered,

or

TPLCODEACT = Action code entered and no reason code entered or reason code fails validation during claims processing,

or

TPLCODEACT = No action code entered and reason code entered.

Processing continues. DIAMOND takes no hold or deny action based on the TPLCODEACT parameter.

Online = DIAMOND issues appropriate processing messages during claims processing.

Batch/API = DIAMOND issues a critical error during claims processing.

More than one authorization matches the claim.

Online = DIAMOND prompts operator to select one authorization.

Batch/API = DIAMOND does not read the TPLCODEACT parameter (takes no hold or deny action) however; the claim is held as a result of the Auth/Claim Match process.

In addition:

  • There is only one reason code displayed for a held claim detail line. If DIAMOND has denied a claim detail line prior the AUTHS - TPL Code processing logic, DIAMOND displays the previously assigned deny reason.

  • DIAMOND invokes the AUTHS - TPL Code processing logic after the assignment of medical definitions (MEDEF) and before adjudication processing.

  • When a claim is denied as a result of the AUTHS - TPL Code processing logic, DIAMOND makes the non-covered amount equal to the allowed amount. As this occurs prior to adjudication, no adjustment to patient liabilities is necessary.

DIAMOND does not include the following claim types/statuses in the AUTHS - TPL Code processing logic:

  • Prepaid claims

  • Claims with a claim status of I (Informational) or N (No Check)

  • Claims with a processing status of F (Final), P (Paid), C (Check Issued), or N (Paid-No Check)
