Restrictions

A claim must match an authorization at the header level in order for authorization/claim detail linking and pricing to be performed. Thus, you must enable authorization/claim match logic to use this functionality. To learn how to enable and use DIAMOND authorization/claim matching logic, see Authorization Claim Links, How DIAMOND Thinks.

Instructions

Step

Comment

1. Use keyword PRCRA (Authorization Procedure Ranges) to define sets of procedure codes to be used during authorization/claim detail linking.

After you define a PRCRA set, you can use it as a procedure code match criterion when you build CDAPL (Claim Detail Authorization Procedure Link) link rules. The PRCRA set is used to determine if submitted claim procedure codes that do not match the authorized procedure code should be considered authorized.

2. Use keyword CDAPR (Claims Detail Authorization Procedure Rules) to define authorization detail match criteria rules based on:

  • Authorized procedure amount

  • Authorized procedure status

  • Whether the submitted procedure code and authorized procedure code are an exact match, in range, or out of range.

After you define CDAPR rules, you can use these rules as authorization detail match criteria when building CDAPL rules.

For examples of CDAPR rules, see Authorization/Claim Detail Linking and Pricing, How DIAMOND Thinks, section Examples of Claims Detail Authorization Procedure Rules.

3.  Use keyword CDAPL (Claim Detail Authorization Procedure Link) to define rules for performing authorization/claim detail linking and pricing.

You define CDAPL rules for a specific line of business, authorization type and effective date combination. When you enter a claim that has a matching header authorization and also matches a CDAPL rule line of business/authorization type/effective date combination, DIAMOND searches for the matching authorization procedure details. DIAMOND then applies the CDAPL rules to the claim and matching authorization to determine what action to take on each claim detail line.

When you enter a CDAPL rule, DIAMOND initiates a default processing rule for the CDAPL rule with a Process Order of 9999. DIAMOND uses this rule if during evaluation none of the prior processing rules can be applied based on the criteria defined in CDAPR. You should complete the fields and save this processing rule in order to provide a "graceful landing" for this evaluation process.

When CADPL rules are evaluated, DIAMOND reviews the CADPR Rule ID to determine whether:

  • An exact match between claim procedure and authorization procedure is required

  • A match within a range of procedure codes

  • A possible match with an out of range procedure

If an exact match is required and this criteria is met, DIAMOND determines whether a price is provided for the authorized procedure and the authorization status. If there is an assigned dollar amount (greater than 0$) on the PRCDR table and the status is a match, the DIAMOND will perform the pay action. In this case, DIAMOND will not evaluate the procedure ranges.

 

4. If you use Whole Claim Pricing, use system parameter CDAPLWCPSYSG to define the system generated procedure code used when the following apply:

  • The claim line is Whole Claim Priced (WCP)

  • An additional adjustment is required to balance the claim total payable

  • A Claim Detail Authorization Procedure Link (CDAPL) rule is used with a Pay Action of Price Using the Auth Price

 

5.  Use keyword PROCD (Procedure Codes) to flag procedure code records for inclusion in Institutional claims authorization/claim detail linking and pricing. Flag each appropriate procedure code record for inclusion by selecting the PROCD check box Inst. Auth Detail Link.
