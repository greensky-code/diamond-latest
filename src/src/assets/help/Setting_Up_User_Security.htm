Setting Up Security

In order to begin using the DIAMOND program, you must set up each user and grant security access to keywords, Special menu commands, and actions.

You can either set up each user from scratch or you can set up access characteristics in templates and assign those templates to users.

Oracle Roles and DIAMOND Security

DIAMOND uses Oracle’s security templates, called roles, to manage user IDs, passwords, and access to the application. This is important because it means DIAMOND users cannot use an external tool, such as SQL*Plus, to access DIAMOND and change records (with a specific exception explained below). DIAMOND manages this relationship to Oracle automatically.

For more information about roles, see Security and Users, How DIAMOND Thinks.

Restrictions

User associations with security templates override any entries for those users as individuals in keywords SWIND and SFUNC. Therefore, keywords SWIND and SFUNC limitations must be defined in the security templates themselves.

1. Enable system security

Enable system security in keyword PARAM (Parameter).

Learn how to enable system security.

2. Add a super user

Add at least one super user in keyword SUSER (Users). A super user is defined as a user with full access to all commands, windows, and capabilities in DIAMOND. You will need more than one super user. Security officers, database administrators, and other management level employees are typically super users.

Learn how to add a super user.

3. Add security templates

Determine if any menu command, specific window, add/update/delete, claims processing, or record access limitations are shared by more than one user in your organization.

To the extent you can define shared security privileges, you can define security templates with those shared characteristics. This can be much easier to maintain than entering specifics for each individual user.

Define the number of templates you need, the access characteristics for each, and give each template a name.

Add security templates in keyword SUSER. A security template is similar to individual user records except:

  • The Template check box is selected.

  • There is no template ID assigned to a template user.

  • You do not use Special menu command Add Database User for template records.

If you are entering claim limit information, note that the entire section must be completed.

Learn how to add a security template.

3. Grant access to menu commands

If applicable, enter any menu command limitations for each template in keyword SFUNC.

Keyword SFUNC allows limitations to certain commands on the menu bar, which reside within keywords, such as the Terminate Member command in keyword MEMBR.

4. Grant access to keyword windows

In keyword SWIND, enter each keyword add (ins), view (sel), change (upd), and delete (del) access for each template.

If no access is granted to a template in keyword SWIND, that template will not be able to do anything in DIAMOND.

To grant access to a keyword you must make sure you enter access to both the keywords themselves and the associated "sheet."

A table of keywords and associated sheets is found in Security and Users, How DIAMOND Thinks.

5. Grant access to fields

In keyword SFLDL, create a new field-level security profile and assign a Field Level Security ID to the profile.

Enter the ID from SFLDL in the Field Level Security ID field in SUSER; this attaches the field-level security profile to the template.

6. Add users assigned to templates

For each individual user whose security privileges are defined by one of the templates, add an SUSER record.

  • Required fields are User ID, User Type, Language, and Row Level Security.

  • Complete the Template ID field with the appropriate SUSER template User ID.

For User Type, currently there is only one valid entry, Operator.

Remember that a user who is assigned a template should not have any entries in SWIND or SFUNC; those access privileges must be set up in the template, and any user-specific entries will be ignored.

Learn how to add a user assigned to a template.

7. Add users not assigned templates

For each individual user whose security privileges fall outside those set up in a template, add a record in keywords SUSER, SWIND, and SFUNC. To set up field-level security restrictions, add an individual record in keyword SFLDL.

To copy an existing record to an individual user’s profile, use one of the following special functions:

  • SFLDL: Copy Field Level Security ID

  • SWIND: Copy Window Access

  • SFUNC: Copy Function Access

Remember, the last step in setting up a user in SUSER is the Special command Add Database User, where you set the user’s password.

Learn how to add a user.

Frequently Asked Questions

Under what circumstances must I establish "database roles?"?

The only circumstance is when you want to grant a power user the ability to change DIAMOND data using an external tool such as SQL*Plus. DIAMOND automatically invokes the DIAMONDVIEW and DIAMONDUPDATE Oracle roles as part of creating a user.

For information about how roles work, see Security and Users, How DIAMOND Thinks.

There are two keywords on the Security menu that aren’t mentioned in this task, SFNDS and SWNDS. Do I have to update records in these?

DIAMOND is delivered with all the necessary entries. The only circumstance in which you might have a reason to update these is if you have created software external to DIAMOND and without Perot Systems assistance, and you want those functions to appear on DIAMOND menus and work with DIAMOND security. At some point you would add those externally created functions to these two keywords.

Can I add other menu or commands to SFNDS so that I can impose security on them?

No. While SFNDS allows entries, they will be ignored by DIAMOND security features. To see a complete list of SFNDS pre-defined functions, see SFNDS reference documentation.

If I give someone access to certain claim records using SUSER Affiliations and I don’t give them access to the claims keywords in SWIND, which instruction prevails?

SWIND will prevail. DIAMOND looks at SWIND first to see if access to a keyword exists at all, and then limits that access according to the affiliations and claim limit parameters in SUSER.
