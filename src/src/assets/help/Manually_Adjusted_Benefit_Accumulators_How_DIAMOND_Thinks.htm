ACCUM and ADJUD Processing Codes

 

Benefits are calculated automatically. However, there are occasions when to-date benefit accumulators must be entered so that going forward, automatic adjudication is accurate.

Use keyword ACCUM (Benefit Accumulator Base Values) to manually enter one-time credits toward a member’s benefit limit in cases where those credits are not carried forward automatically either because of how the benefit packages are defined or because those credits were incurred outside the system.

When Is ACCUM Necessary?

For a current member, expended benefit amounts will carryover from one benefit package to another only if the following two conditions apply:

  • The same benefit rule ID (BRULE) is used in both the old and the new benefit package

  • The benefit rule has been flagged to apply to all groups (BRULE — All Groups?)

For example, a member changes from benefit package OK10P to OK15P, and both of these benefit packages contain the lifetime maximum benefit rule ID — LIFETIME (flagged to apply to all groups). All accumulated claims history (including expended benefit amounts) will be carried over from the old package to the new package. However, if the member’s old benefit package has a $250 deductible (benefit rule ID — DED250) and the member’s new benefit package has a $500 deductible (benefit rule ID — DED500), accumulated deductible amounts paid by the member will not be carried over to the new benefit package.

When you key a member’s base values (expended benefit amounts) directly into the accumulator file there are no associated claim detail lines.

Timing of Using ACCUM and Subsequent Claims Processing

You can use keyword ACCUM to create an independent, one-time carry over toward a member’s new benefit package. The ACCUM records are not affected by claims entries. For example:

  1. A member changes benefit packages effective 4/1/96.

  2. On 4/15/96, the member is credited for a previously expended deductible under their old benefit package using keyword ACCUM.

  3. On 5/1/96, claims with dates of service prior to 4/1/96 are processed.

  4. DIAMOND continues to take a deductible for the claims entered on 5/1/96 (up to the deductible limit) based on the old benefit package (effective prior to 4/1/96).

  5. DIAMOND does not adjust the accumulator file credits entered on 4/15/96 for the amounts deducted as a result of processing the claims based on the old benefit package.

ACCUM and ADJUD Processing Codes

When you add records in keyword ACCUM, DIAMOND assigns a processing code ACCUM to each. You can only change accumulator file records with processing code ACCUM. You cannot change accumulator file records that resulted from the claims adjudication process, which have a DIAMOND assigned processing codes of ADJUD (adjudicated).

"Days" Benefit Limits

For benefit rules accumulating dollars or quantities towards limits, usually only one benefits accumulator file record (with the appropriate Base Amt or Base Qty) is required per member, per benefit package, per benefit rule. An exception occurs when creating a benefits accumulator file record (using keyword ACCUM) for a Limits benefit rule of Limit Type Days.

Here, with the limit expressed in "days," multiple records are necessary when reflecting an expended benefit of two or more days. This is because when adjudicating this type of limit DIAMOND does not use the Base Qty field on the corresponding benefit accumulator records. Instead, DIAMOND counts each unique occurrence of a claim detail date of service on the corresponding benefits accumulator records as one day. Thus you may need to create multiple benefits accumulator records when reflecting an expended benefit of two or more days (that is, two or more unique dates of service).

Compare or Ignore Dates

In ACCUM you cannot enter dates that are prior to the member's eligibility date, therefore, you may run into a situation where the date on an ACCUM record is the same as the date of service for a claim record. When the Rule Type=Limit and the Limit Type=Days, or when the Rule Type=Full Coverage and the Full Coverage Type=Days you will need to select a value for the Compare Dates field on the ACCUM screen. This field determines if the DIAMOND system compares the dates of the ACCUM record against dates on claims or if the DIAMOND system ignores the dates on the ACCUM records and counts each as 1 day regardless of whether or not the date may be the same as a claim date.

Compare Dates

When you select the Compare Dates option on an ACCUM record, the date entered in ACCUM is compared to the date for each claim. If the dates are the same, only 1 day will be accumulated.

Ignore Dates

When you select the Ignore Dates option on an ACCUM record, the record counts as one day even if a claim is submitted for the same date. In this case where you have an ACCUM record and a Claim that have the same date, 2 days will be counted.

DSPBN

DSPBN indicates if an ACCUM record was marked as Compare Dates or Ignore Dates.
