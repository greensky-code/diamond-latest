Running Checks

Checks may be created for claims and/or for capitation payments, run separately. DIAMOND processing creates both check and electronic fund transfer (EFT) transactions.

EFT transactions

Running a job in Keyword CKPRT (Check Print Setup) creates both actual check transactions and EFT transactions. The EFT transaction work file records are available to allow you to develop your own customized EFT interface, which should provide for:

  • Creating standard EFT interface records using DIAMOND’s EFT transaction work file records

  • Flagging valid EFT transaction work file records with a "P" (pulled)

  • Deleting all EFT transaction work file records associated with a batch run after successfully creating EFT transactions.

When you post a check print job in keyword CKPST, check data records are retrieved from the check print work file and flagged "P" records from the EFT transaction work file. During the post process, all processed records from both the check print work file and the EFT transaction work file are deleted. A warning message displays if un-flagged EFT transactions are in the work file when you use CKPST. There is no way to go back and post EFT transactions, so be sure to post the EFTs before running a job in keyword CKPST.

In keyword CKPRT, you must type the check number, even if the batch run is for EFT transactions only.

Restrictions

If you have more than one batch check run to process for the same company and bank account code, you must process the first job in both keywords CKPRT and CKPST before running the second job in keyword CKPRT.

If you make an error and process two such batch check runs in keyword CKPRT before successfully posting the first in keyword CKPST, DIAMOND uses the same starting check number for both CKPRT runs. This is because DIAMOND updates the starting check number for a company only upon successful post of a batch check run in CKPST. When two CKPRT batch runs contain duplicate check numbers, you can post the first in keyword CKPST but not post the second. Instead, you receive an error message that a CKPRT run with one or more duplicate check numbers has been run and posted. Should this happen, in order to have DIAMOND assign non-duplicate check numbers and allow you to post the second batch run, you will need to do the following:

  1. Cancel the second check print job in keyword CKPRT

  2. Re-run the cancelled check print job in keyword CKPRT

  3. Post the re-run check print job in keyword CKPST

To run checks

  1. Create check and remittance advice templates using Intelligent Queryä (IQ). DIAMOND is delivered with one generic template, CHECKS, to print checks only on pre- or unnumbered check stock. This template does not allow for printing remittance advices.

  2. In keyword BANKA, attach your check print templates to each bank account from which you will be paying claims and/or capitation.

  3. Post payable transactions so they are available to accounts payable for payment. For capitation payments, see Post a Capitation Calculation Job. For claim payments, see Run an A/P Update Job.

To maximize performance, use minimal selection criteria for APUPD; additional filtering can be applied in CKPRT. APUPD looks at a variety of conditions to make sure the claim transactions being posted are available for payment. If any of the following conditions apply, claim detail lines will not be posted to accounts payable:

  • Claims on header hold

  • In keyword VENDR, the Hold Payment Flag check box is selected for the vendor

  • Unless partial payment is allowed (parameter PARTIALPAY set to Y), claim detail lines on claims with a mix of available and held lines

  • If a third-party auditing tool is enabled, claims which have not yet completed the third-party process

  • Claim detail lines with status "F" unless the Cap/Special Fund status is "3"

  1. In keyword APSEL, de-select or re-select A/P transactions for payment. When a transaction is de-selected for payment it will not be available to CKPRT; it must be re-selected to re-enter the process. Transactions can be de-selected for payment based on:

  2. Company code

  3. Vendor

  4. Due Date

  5. Entered Dates

  1. In keyword CKPRT, create a (or select a previously defined) job ID. Calculate check and electronic funds transfer (EFT) transactions for that Job ID using CKPRT> Action> Calc. The CKPRT Job ID you create (or select from previously created IDs) has a variety of filters and criteria associated with it that define which transactions are calculated (and later printed and posted).

  2. For printed checks, print alignment checks by selecting CKPRT> Action> Print for the appropriate Job ID. A dialog box will be displayed from which you can click Print Alignment Check. For alignment, use a direct, non-spooled printer. You may print as many alignment checks as necessary. The IQ software icon displays while alignment checks are being printed. The next available check number is displayed in Start Check No (specific to the Bank Account selected). The first valid check printed will be numbered with the sum of the Start Check No. + the number of alignment checks printed in this step.

  3. When you are satisfied with alignment of the check stock, print all checks using CKPRT> Action> Print (click Print All Checks). The IQ software icon displays while checks are being printed.

  4. If you need to reprint the check run, then do the following:

  1. Select Action> Cancel

  2. Select Action> Calc

  3. Select Action> Print

  1. In keyword CKPST, view the check transaction records created by the print function. All the transactions will be displayed with a default status of "Posted." You may change any or all of these records to either cancel or invalidate by selecting the record(s) and changing the status online. EFT transactions are not displayed. Invalidated check numbers cannot be re-used. This status is typically applied to ‘bad’ checks printed on pre-printed check numbered stock. Cancelled check numbers can be reused. This status is typically applied to ‘bad’ checks printed on unnumbered check stock. Cancelled checks can be invalidated; invalidated checks can be cancelled; either can be changed back to the default status of Posted. Keyword CKPST is not used in any way for manual checks. For more information, see Entering a manual check.

  2. Once you are satisfied with the statuses of all check transaction records as displayed in CKPST, on the File menu, click Run Job. Once Run Job has been selected, the status of checks can no longer be changed as in Step 9; they will no longer appear in keyword CKPST. If checks have been invalidated or canceled within a job, they will continue to be displayed on the CKPST window and the check print batch job ID will continue to have a status of Printed. If unflagged (with a "P") EFT transactions remain in the workfile when you post a Job, DIAMOND will display a warning message. This must be corrected using your custom EFT interface program. For non-cancelled, non-invalidated transactions, as a result of CKPST, Run Job:

  3. Accounts payable records ACPAY Processing Status is "Paid."

  4. Associated claim detail line Proc Status is "Paid"

  5. Posted checks are available for lookup and reconciling in keyword CHECK (Check Register).

Frequently Asked Questions

Can I have multiple check runs in various stages of processing at the same time?

Yes, for different bank accounts.

Why would I ever override the next available check number which is automatically displayed?

Here is an example:

  1. You are using check stock with pre-printed check numbers and you are about to print five checks.

  2. The highest check number in the Check Register is 1500.

  3. The DIAMOND displayed Start Check No. is 1501.

  4. You begin printing checks, the printer jams, and the five checks are mutilated or print alignment is off (pre-printed check numbers 1501-1505). You must reprint the five checks.

  5. DIAMOND still considers the Start Check No. to be 1501, as you have not posted the first (five check) check run.

  6. You override the displayed Start Check No. and enter the next available check 1506 and reprint the check run (check numbers 1506-1510).

  7. You post the checks in keyword CKPST.
