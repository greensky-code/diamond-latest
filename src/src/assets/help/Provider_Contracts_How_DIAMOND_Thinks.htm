Conditional Contract Status

How DIAMOND handles partial capitation, cap/special fund situations, and other mixed contract situations

The PROVC record contains several contract status fields. In addition, procedure code pricing records also include a contract status override. Here’s how they all fit together and how they can be used to maximum flexibility. While the contract type on the provider contract record suggests an ‘all or nothing’ status with regard to capitation or fee-for-service ("payable"), DIAMOND easily accommodates partially capitated arrangements.

Partial Capitation

Capitated providers may be reimbursed on a fee-for-service basis for certain claims. For example, a primary care physician is capitated for primary care encounters, but is paid on a fee-for-service basis for outpatient surgeries.

  • You can use the Default Contract Type in the provider’s provider contract record (PROVC>Def Contr Type) to specify the providers who are included in a Capitation Model/Entity. This may be used during capitation calculation to select providers.

  • You can select a provider Contract Type on the capitation entity record (CPENT>Contract Type) to specify the providers who should be included in the capitation calculation only if the associated capitation model’s (CPMOD) Use Prov Contract check box has been selected.

  • DIAMOND uses the provider’s PROVC>Special>Contract Pricing Details>Contract Type and PROCP>Contract Override during claims processing. Typically, Contract Pricing Details specify how a provider’s associated claims are categorized (capitated or fee-for-service) and exceptions are specified in the Procedure Price (PROCP) record.

For example, Dr. Jones is capitated for all but a few special surgeries that are paid fee-for-service. Dr. Jones’ PROVC>Special>Contract Pricing Details>Contract Type = C (Capitated). Those few surgeries are specified in the appropriate PROCP>Contract Override = P (Payable).

In another example, Dr. Smith is capitated for only 40 primary care procedure codes. Dr. Smith’s PROVC>Special>Contract Pricing Details>Contract Type = P (Payable). The 40 primary care procedure codes are specified in the appropriate PROCP>Contract Override = C (Capitated).

  • PROCP is the final word on Contract Type for capitation and payables (fee-for-service); however, you cannot specify a contract type = Cap/Special Fund Distribution on a PROCP record. For more information, see Capitation/Special Fund Distribution, How DIAMOND Thinks.

  • In all cases, the Contract Type used will drive certain results in the claim detail line Claim Status field. Unless the claim is denied, an adjustment, or a no check/adjustment, Contract Type P results in Claim Status P; Contract Type C results in Claim Status C; and Contract Type F (on PROVC only, not used in PROCP) results in Claim Status F. For more information, see Capitation/Special Fund Distribution, How DIAMOND Thinks.

Retaining obsolete contract records

For audit trail purposes, PROVC vendor records should not be deleted after claims have been paid. If the vendor and contract detail information is no longer valid, the current contract should be terminated and a new contract created with newly effective data.

Authorization costing and provider contracts

DIAMOND calculates estimated costs for authorized services using the same contract search and other logic as claims, with one exception: only the provider’s primary fee schedule is used to calculate estimated costs; the alternative fee schedules are not used in authorizations.

How DIAMOND selects a contract when pricing a claim

When DIAMOND prices a claim, the claim’s associated line of business record (LINBS) specifies where to look for pricing data (the Primary Price Search list in keyword LINBS):

  • GROUP - DIAMOND looks for claims pricing data on the associated Group Detail (GRUPD) record first. Then, DIAMOND looks for claims pricing data on the associated Provider Contract (PROVC) record. If DIAMOND finds pricing data on the provider contract record, then it overrides any pricing data found on the associated group detail record.

  • PROVIDER - DIAMOND looks for claims pricing data on the associated Provider Contract (PROVC) record first. Then, DIAMOND looks for claims pricing data on the associated Group Detail (GRUPD) record. If DIAMOND finds pricing data on the group detail record, it overrides any pricing data found on the associated provider contract record.

This is useful if you are administering PPO pre-pricing programs (pricing but not adjudicating claims). For these kinds of programs, it is common for the claims pricing agreements to be defined by employer groups rather than by individual provider contracts. In this case, in keyword LINBS, the Primary Price Search list would be set to Group.

In most managed care programs, pricing is governed by individual provider contracts. In this case, in keyword LINBS, the Primary Price Search list would be set to Provider.

The associated LINBS - Primary Price Search field determines the order in which DIAMOND looks for pricing rules, not the only place DIAMOND looks. Therefore, DIAMOND applies the last pricing information found. If ‘Group’ is the first place DIAMOND looks, PROVC is the second place searched and data found in PROVC will be applied. If ‘Provider’ is the first place DIAMOND looks, GRUPD is the second place searched and data found in GRUPD will be applied.

When DIAMOND searches for a provider pricing contract (PROVC), the claim’s associated line of business record (LINBS) also specifies the provider contract search criteria (LINBS - Contract Search Type) and the order in which this criteria is applied (LINBS - Order).

Combinations of the following can be specified as contract search criteria sets:

  • Provider, Line of Business, Panel, IPA, and Contract Effective Date (always implied)

  • Default Provider, Line of Business, Panel, IPA, and Contract Effective Date (always implied)

Diamond’s provider contract search processing rules include:

  • Search for a match on provider, either search for a contract for the provider of service on the claim or search for a default provider contract.

  • Search for a match on all search criteria between the claim and the pricing contract. That is, if the search criteria includes Prov (#123) + LOB (H01) + Panel (TPA) + IPA (#456) elements, there must be a contract record that contains all of these elements and the claim must contain and match all of these elements. For example, if there is only one contract record that contains Prov (#345) + LOB (X01) + Panel (blank) + IPA (#789) elements, a claim with:

Prov (#345) + LOB (X01) + Panel (blank) + IPA (blank) does not match the contract record.

Prov (#345) + LOB (X01) + Panel (XYZ) + IPA (blank) does not match the contract record.

Prov (#345) + LOB (B99) + Panel (blank) + IPA (#789) does not match the contract record.

Prov (#345) + LOB (blank) + Panel (blank) + IPA (blank) does not match the contract record.

Prov (#987) + LOB (X01) + Panel (XYZ) + IPA (#456) does not match the contract record.

Prov (#345) + LOB (X01) + Panel (blank) + IPA (#789) matches the contract record.

  • The primary date of service on the claim is always verified against the effective date on the pricing contract. The primary date of service on the claim must be on or after the contract effective date and, if applicable, before the contract termination date.

  • When search criteria sets include LOB, Diamond retrieves the LOB from the member's eligibility record.

  • When search criteria sets include Panel, Diamond retrieves the panel from the member's eligibility record.

  • If the member’s eligibility record includes a panel, Diamond searches for a provider contract using the member’s panel. If there is no match using the member’s panel, Diamond continues to search using the next LINBS contract search criteria set.

  • If the member’s eligibility record does not include a panel, Diamond retrieves the panel from the member’s associated Group Panel (GRUPP) record.

  • If more than one panel exists on GRUPP, DIAMOND selects the panel based on sequence number order (the order in which the panel records were originally entered on the GRUPP window, which may differ from the order in which the panel records are displayed on the GRUPP window). If there is no match using the first GRUPP panel record, Diamond retrieves the next sequential panel record and continues the search. If there is no match or if there are no GRUPP panel records, Diamond continues to search using the next LINBS contract search criteria set.

Default Provider Contracts and Default Provider Logic

Background

Use keyword PROVC (Provider Contracts) to add and maintain the data that defines the contractual relationship between the healthcare organization and the healthcare provider. This data can have an impact on the following DIAMOND processes:

  • Membership (relating to PCP Assignment)

  • Capitation

  • Claims

During claims processing, DIAMOND must determine a relationship between the rendering provider and the member (claimant). To do this, DIAMOND uses provider contract search logic and the associated LINBS contract search criteria sets to "match" a provider’s contract to the member. See the sections entitled How DIAMOND Selects a Contract when Pricing a Claim and Provider Contract Search Examples for more information.

It may not be practical to create and manage a unique contract record for each provider, line of business, IPA, and panel. It may be more efficient to create and manage only the very specific relationships of a provider to line(s) of business, IPA(s), and panel(s) and then use a more general contract(s) that can be shared by many providers who have the same relationship with the healthcare organization.

You can create a "default provider" and associated "default provider contract(s)" to handle this many provider, same relationship situation. You also use the DFTCNTRCTPRV parameter record to identify the default provider ID. DIAMOND applies a default provider contract (if the default determinants match) when a provider specific contract is not found during the provider contract search; that is, DIAMOND searched through all of the LINBS contract search criteria sets and did not find a matching specific contract. A default contract(s) is not physically associated with the "many" providers; however, DIAMOND can apply a default contract as a result of the default provider logic.

The number of provider contract records that you must create and maintain can be reduced significantly when you use DIAMOND’s default provider/contract(s) option. In addition, you can use this option to apply default contracts to claims received from providers who are not maintained on DIAMOND; for example, out-of-area providers, non-specific providers.

Default Provider Logic

DIAMOND uses default provider logic during claims processing only. DIAMOND does not use default provider logic during membership (enrollment) or capitation processing.

To use DIAMOND’s default provider logic:

  • Create a default vendor and default vendor address using keywords VENDA (Vendor Master) and VENDA (Vendor Address).

  • Create a default provider using keyword PROVD (Provider Master). The default provider ID you create is not directly associated with a claim during claim processing. This ID acts as a "pointer" to the default provider contract record(s) that DIAMOND will search for when it uses default provider logic.

  • Enter the default contract(s) associated with the default provider using keyword PROVC (Provider Contract). DIAMOND’s provider contract pricing details (PROVC, Special menu > Contract Pricing Details) are referenced based on the Vendor/Vendor Address associated with each provider (including the default provider). You must enter a PROVF > Default Vendor ID and a PROVF > Default Vendor Addr for each provider (including the default provider). During claims processing DIAMOND will not use the default vendor/vendor address on the default provider record as the payee for the claim; however, DIAMOND will use this default vendor/vendor address to locate the appropriate default provider contract pricing details.

  • If required, create a non-specific provider using PROVF (Provider Master). You can use this provider when you pay out-of-area, non-par, or one-time provider claims.

  • Activate the DFTCNTRPRV parameter and enter the default provider ID in the parameter record (see PARAM User Documentation).

  • Set up the appropriate default contract search criteria set(s), in the proper order, on the associated LINBS record.

For more information, see Default Provider/Vendor Set Up Sequence.

Target (alternate) Revenue and CPT4/HCPCS Codes

There are instances in institutional claims where a revenue code is associated with a CPT code and both must be entered on a claim detail line if the line is to be priced accurately. In the detail procedure code record, PROCD, the Price Indicator check box tells claims that an alternate procedure code instead of the revenue code drives the pricing of the claim line. The Procedure Determinant command on the Special menu in PROCD stores CPT4/HCPCS codes that are valid alternates for each revenue code.

If you want INCLM to edit the entry of alternate codes to make sure they are valid for the entered revenue code, you must turn this feature ‘on’ in the institutional provider’s PROVC record by selecting the PROVC>Special menu>Contract Pricing Details - Target Rev Code Edit check box and entering an action and reason code. If the CPT4/HCPCS code is not valid for the revenue code entered on the claim, DIAMOND performs the action and assigns the reason code to the claim detail line.

See Procedures and Pricing - Target Revenue Codes for more information on Target Revenue Codes.
