Assigning G/L Codes to Claims

Restrictions

Instructions

Frequently Asked Questions

Many health plans post claims expenses to different sets of general ledger (G/L) accounts depending on line of business, type of service, provider status, and the like. Use keyword GLASS to assign the G/L accounts and G/L company, depending on determinants you define.

The assignment of G/L accounts applies to online, EDI, and API processed claims.

Several keywords are used to set up the G/L cross reference codes, define the claim data that drives posting to different accounts, and manage the specific data values that determine which company and set of accounts are associated with different categories of claim expense:

  • Use keyword COMPF to add G/L company codes.

  • Use keyword GLREF to add G/L reference codes to cross reference sets of debit and credit accounts for the net amount, for the provider’s withhold amount, for any calculated discount amount, and for any calculated interest amount.

  • Use keyword PARAM, which contains up to six GLASSIGN records, to define the data elements on the claim or member’s group to be determinants of the G/L assignment

  • Use keyword GLASS to add rules that tie values for the GLASSIGN data elements to company and G/L cross-reference codes.

Restrictions

At least one GLASSIGN record must be set up in keyword PARAM.

Each GLASSIGN record must identify a unique determinant.

Instructions

Step

Comment

1. Identify the G/L accounts to which claims expense will be posted.

 

2. In keyword COMPF, add G/L company codes.

 

3. In keyword GLREF, add G/L reference codes.

G/L reference codes are cross references to the actual G/L account numbers—a primary set of accounts to post the claim net amount, and a secondary set of accounts to post provider withholds, a third set of accounts to post claim discount amounts, and a fourth set of accounts to post claim interest amounts.

4. Identify which data elements associated with a claim record will determine the posting of expense to the general ledger.

You may use up to 6 determinants; you must use at least one. Any field on the claim header and claim detail, plus the Group State field on GROUP are available.

5. For each determinant selected, add a GLASSIGNx record in keyword PARAM to define the determinant.

If State from GROUP is used, it must be entered in GLASSIGN1.

The precise field name is entered in Parameter 1, e.g., for line of business the entry is LINE_OF_BUSINESS.

The determinant name (field literal), which is displayed in GLASS, is entered in Parameter 2.

If you use whole claim pricing for institutional claims, you must also enter either HSD_INST_CLAIM_HEADER or HSD_INST_CLAIM_DETAIL in Parameter 3, to specify where the determinant entered in Parameter 1 resides.

For example, if you use LINE_OF_BUSINESS in Parameter 1 and you use whole claim pricing, enter HSD_INST_CLAIM_HEADER (the table where Line of Business is located) in Parameter 3.

If you leave Parameter 3 blank, DIAMOND assumes you are not using whole claim pricing for institutional claims. Institutional and professional claims priced on a line-by-line basis will be assigned the correct G/L code based on the values entered in Parameters 1 and 2. If Parameter 3 is blank, any whole claim priced claim lines will not be assigned to a valid G/L code and the user will receive an error message requiring manual entry of the G/L code on the whole claim price (system-generated) line.

6. In keyword GLASS, enter the values for the determinants defined in Step 5, and the G/L reference and company codes that correspond to those values.

Each unique combination of values for the determinants you have defined must be entered.

Because this may require a large number of entries, it may be more efficient to define the combinations external to DIAMOND, using Microsoft Excel or other matrix, and upload that matrix to DIAMOND.

Frequently Asked Questions

  • Does the G/L Assignment table allow you to use ‘fall through’ logic in assigning G/L codes to professional and institutional claims?

You can define one default G/L code for claims. To create this record, add a GLASS record where all the determinant values on the record are left blank and point it to a specific company and G/L reference code.

Do not add a GLASS record where one determinant field is blank while other determinant fields on the same record contain values. DIAMOND logic assumes an exact match between each of the determinant values on the GLASS table with the corresponding fields on the claim header or detail.

  • Where are G/L codes used in DIAMOND?

DIAMOND G/L codes are used to record premium billing revenue and account receivables; institutional and professional claims expense; and capitation risk pool allocations, depletions, and stoploss.

  • How does this G/L account information get sent to my financial system?

During DIAMOND’s posting process the calculated premiums, cash receipts, claims expense, or capitation calculations are written to a General Ledger Interface table (GL_INTERFACE_TABLE). This table contains information on the transaction’s company code, G/L code, and G/L debit and credit numbers. You can create an extract of this table to feed directly into your financial system.
