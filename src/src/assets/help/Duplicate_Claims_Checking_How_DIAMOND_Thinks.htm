Duplicate Claims Checking, How DIAMOND Thinks

Using different duplicate criteria for different kinds of business

System Performance Implications of DUPRL

How DIAMOND makes a match

Keyword DUPRL enables DIAMOND to automatically detect potential duplicate claims and take specified action when a potential duplicate is identified.

 

Rules for detecting duplicate claims (both online and batch) can be different for each of the claim types: professional services, institutional, and dental.

The degree of certainty that a claim is a duplicate can determine the degree of action that is taken. For example, a potential duplicate claim with more stringent match criteria can be automatically denied, while a potential duplicate claim with less stringent match criteria can be automatically held.

Different duplicate criteria can be applied at the header and detail levels of a claim.

Keyword DUPRL uses fall-through logic much like that used for the calculation of medical definitions. Set up the most specific duplicate checking criteria to be applied first in Rule Level 1, and increasingly less specific Rule level 2 and Rule Level 3 respectively.

Warning

DIAMOND does not include denied claims in the duplicate evaluation. Therefore DIAMOND will not identify as a duplicate a claim that compared to a denied claim, meets the all the criteria of a potential duplicate.

Using different duplicate criteria for different kinds of business

Duplicate checking rules may be set up to work differently depending on the kind of business being administered. This feature is helpful if you operate both as a third party administrator for some groups, where very careful examination of potential duplicate claims is required and also as a fully insured, capitated program for other groups that have a high percentage of claims falling within capitation and therefore not requiring close scrutiny for duplicates.

Each duplicate check rule ID is linked to a line of business code (set up in LINBS) so different rules can apply to different lines of business. DIAMOND applies the rule according to the line of business associated with the member (patient) on the claim.

How DIAMOND makes a match

Keyword DUPRL contains three levels of rules. They are evaluated twice during claims processing and applied in level order. Once a match is found between the claim and the criteria in a rule level, the action specified in that rule level is applied and no further duplicate checking is performed at that point in processing. For all evaluations of claims against duplicate checking rules, a match is only detected when all of the fields indicated within the Dup Rule Level criteria match. And, logically, an exact match on member ID (subscriber ID + person number) is required before any rules are evaluated.

The most specific criteria for a match should be set at the highest level. For example, your organization may feel that a claim with matches on the following fields can be considered a duplicate claim:

  • Rendering provider

  • Total billed amount

  • Detail date of service

  • Procedure code

If duplicate information is found, then the claim can be automatically denied. In this case, you would enter these criteria as Duplicate Rule Level 1, along with the appropriate action (deny) and reason code.

Your organization may consider a claim to be a likely duplicate if there is a match on:

  • Detail date of service and

  • Procedure code

Even if the provider is different, you may want to have the claim manually investigated. In this case, you would enter these criteria in Duplicate Rule Level 2 (or 3) along with the appropriate action (hold) and reason code.

As claims are evaluated against these two rules, a hit on the Level 1 Rule will cause DIAMOND to take the Level 1 action and apply the specified reason code to the claim. Once a match is made, DIAMOND will perform no more duplicate checking (Level 2 or Level 3). If there is no match on Level 1, DIAMOND will proceed to check the Level 2 criteria, and if there is a match, will take the Level 2 action and apply the specified reason code to the claim. If no match is made on Level 2, DIAMOND will move on to check Level 3.

Header vs. Detail Criteria

Depending on your company’s business requirements, you may decide to have claims operators manually check claims that have duplicate information at the header level, regardless of the information at the detail level. In this case you would set duplicate rules to cast a wider net to match on the claim header level. However, if your company desires more automated actions from the duplicate checking system, setting duplicate rules at the claim detail level narrows the scope of possible duplicates.

Warning:

Combining both header level and detail level rules in one duplicate rule sequence may cause more than one action to be applied to a claim.

Special Duplicate Checking for Dental Claims

You can select the DUPRL detail level Same Tooth check box to include tooth attributes in duplicate checking for dental claims. As a DUPRL detail level criterion, Same Tooth is part of DIAMOND’s claim detail level duplicate checking described above. When the Same Tooth check box is selected, DIAMOND includes the following logic in its evaluation of each claim detail line.

DIAMOND first checks the PROCD record for the procedure code on the claim detail line. If the PROCD record has one or more tooth attribute check boxes selected, DIAMOND uses the selected attributes as duplicate criteria. The PROCD tooth attributes check boxes available include Tooth # Req, Surface Req, Arch Req, Quadrant Req, and Oral Cav Req.

When DIAMOND performs duplicate checking for each claim detail line on the current claim, the DUPRL Same Tooth criterion is satisfied if all selected PROCD tooth attributes values match those of a previous claim detail line. If all tooth attributes and values match, and the claim line also satisfies all other duplicate rule attributes, then DIAMOND applies the duplicate rule action and reason code as detailed in How DIAMOND makes a match.

System Performance Implications of DUPRL

The criteria available for evaluating whether or not a claim is a duplicate can reference data on the claim header or data on the claim detail line. Rules that evaluate header data only execute faster than those that look at data on both header and detail.
