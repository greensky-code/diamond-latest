COB - Other Information

COB Warning Message Window

When you enter the Subscriber ID and Person Number on the claim header, DIAMOND displays the COB warning message window showing the first matching COBHS primary coverage record for the member and primary date of service. Also displayed is the action specified in the parameter COBCLAIMACTN. To match, a COBHS record must match the Subscriber ID and Person Number and the claim from date of service must fall within the COBHS record Coverage Effective/Term Date range.

If a COBHS record is found, DIAMOND populates the new column PRIMARY_OC_EXISTS with Y (Yes). Otherwise, this column is populated with N (No).

Note that the COBCLAIMACTN action that is listed on the Other Primary Coverage window is at this point only a potential action. The COBCLAIMACTN action may be overridden by an Other Carrier selection in the Other Carrier pop-up window or by a COB Hold/Deny rule. For more information, see the section below entitled System Parameter COBCLAIMACTN.

System Parameter COBCLAIMACTN

You may use the system parameter COBCLAIMACTN to take the following actions for potential COB claims:

  • Generate a warning message

  • Generate a warning message, hold all detail lines and assign a hold reason code

  • Generate a warning message, deny all detail lines and assign a deny reason code in the Not Covered Reason field

COBCLAIMACTN applies to both online processing and Claims EDI.

COBCLAIMACTN is only invoked if the following conditions are met:

  • The member has other primary coverage (a COBHS record with COB Order = P Primary) where the claim primary date of service falls within the COBHS record’s Coverage Effective/Term Date range.

  • You have not selected an OC Carrier Code in the Other Carrier Information pop-up window. (The Other Carrier Information window pops up when you select EOB or EOMB in the EOB Indicator field -- see EOB Indicator and the Other Carrier Information Pop-Up Window section.)

  • A COB Hold/Deny rule does not apply. (If a COBHD rule applies, the action on the COBHD rule supercedes the COBCLAIMACTN action.)

EOB Indicator and the Other Carrier Information Pop-Up Window

You may receive a Medicare EOMB (Explanation of Medicare Benefits) or non-Medicare carrier EOB (Explanation of Benefits) for a DIAMOND claim. If so, or if an Other Carrier Allowed and/or Other Carrier Paid Amount is otherwise available (for example, verbal notification), in the claim header field EOB Ind (EOB Indicator) select "EOB" for a commercial carrier or "EOMB" for Medicare.

When you select EOB or EOMB, DIAMOND displays the Other Carrier Information window. This window displays all matching COBHS coverage records for the member plus the selection "Other". To match, a COBHS record must match the Subscriber ID and Person Number, and the claim from date of service must fall within the COBHS record Coverage Effective/Term Date range. (DIAMOND does not display the Other Carrier Information Window if you select None in the EOB Ind field.)

If you select a displayed COBHS record, DIAMOND populates the claim header field OC Carrier Code (Other Carrier - Carrier Code) with the COBHS Carrier and makes the OC Allowed field available for input on all claim detail lines. If none of the COBHS records displayed corresponds to the EOB or EOMB received, you can select Other. This also makes the OC Allowed field available for input on all claim detail lines. Selecting Other does not hold or deny claim detail lines. However, the selection of Other can be used in COB Hold/Deny Determinants (COBHD) with an action of hold or deny.

If you received an EOB from an Other Carrier but no matching COBHS coverage records are found, the Other Carrier Information window displays the options "Other" and "Undefined". If you select Other, DIAMOND proceeds as noted above. If you select Undefined, (undefined carrier code), DIAMOND makes the OC Allowed field available for input on all claim detail lines but also places all lines on Hold with a Hold Reason Code of COBOC. If desired, you can then create the needed COBHS record, re-enter the OC Carrier Code and re-adjudicate the claim to link the claim to the COBHS record and relieve the holds. Or, you can mark the COBOC hold reason as historical and manually adjudicate the claim.

If you choose not to select a Carrier record on the Other Carrier Information window and you select "Cancel", DIAMOND changes the EOB Indicator to None.

Overriding the Commercial COB Allowed Amount Rule

You can define the Commercial COB Allowed Amount rule on each PROVC record. The value you define is defaulted to each corresponding claim when a COB calculation method of Non-Duplication of Benefits or Integration of Benefits applies. If you don’t select a rule on PROVC Non-Duplication of Benefits or Integration of Benefits applies, DIAMOND uses DIAMOND Allowed Amount as the rule.

You can override a claim’s Commercial COB Allowed Amount rule for COB Calculation Methods Integration of Benefits and Non-duplication of Benefits only. To change the Commercial COB Allowed Amount Rule, from the PSCLM/INCLM claim header choose Special menu command COB Info, select the correct rule in the Com COB Allowed field and save the header.

If you attempt to select the Special menu command COB Info for a claim where neither Integration of Benefits nor Non-duplication of Benefits applies, DIAMOND generates and error message.

Manually Applying a COB Payment

You can manually apply COB to a claim line by entering an Other Carrier Paid Amount (PSCLM/INCLM/DNCLM detail – OC Pd Amt). DIAMOND subtracts the OC Pd Amt from the DIAMOND calculated Net Amount and displays the result as the adjusted Net Amount.

Claims Batch (EDI) and Claims API – COB Restrictions and Exclusions

  • DIAMOND does not apply COB calculation methods for Claims Batch and Claims API processing. DIAMOND applies COB calculation methods for online processing only. If DIAMOND determines that a claim submitted via Claims Batch or Claims API is potentially a COB claim, DIAMOND applies the rules specified in the parameter COBCLAIMACTN.

  • DIAMOND does apply COB Hold/Deny rules unless the rule includes a field not included in Batch/API processing (for example, EOB Indicator) in Claims Batch/API.

  • If a matching primary COBHS record is found for the claim, Claims batch and Claims API populate the column PRIMARY_OC_EXISTS with Yes.

Other COB Restrictions and Exclusions

  • DIAMOND does not perform automatic COB processing and adjudication for Whole Claim priced claims. Whole claim priced claim lines must be manually adjudicated to apply a COB payment.

  • DIAMOND does not perform automatic COB processing and adjudication for Heads Down Claims processing.

  • It is recommended that you carefully evaluate use of external claim review tools (GMIS/claim check, HPR Code Review, Auto-Audit) with COB claims. The way an external claim review tool views, splits, and returns claim lines may not allow for accurate processing of other carrier paid amounts. As a result, you may want to exclude COB claims from being sent to an external claims review tool. You may do so as follows. On the PSCLM/INCLM/DNCLM headers, the field "EOB Ind" (EOB Attached Indicator) is a drop down list with values EOB, EOMB or None. The selection in this field populates the column EXPLANATION_ATTACHED. Thus in the Claims Audit Filter (CAUDT) function you can set up a rule to exclude claims where an EOB is attached, since this indicates the need for COB. Set up the Claims Audit Filter rule using the exclusion Determinant Name EXPLANATION_ATTACHED with From Value = B - EOB and then the same exclusion determinant EXPLANATION_ATTACHED again with From Value = M - EOMB.

  • DIAMOND does not perform COB Hold/Deny functionality for prepaid claims.

  • Other COB Hold/Deny restrictions may apply. See COB Hold/Deny Processing Restrictions section below for more information.

COB Hold/Deny Processing

You define COB Hold and Deny rules using the COB Hold/Deny Determinants (COBHD) function. You also define COB Hold/Deny conditions using the system parameter COBCLAIMACTN. During claims processing, DIAMOND applies these rules and conditions as follows.

  • When you enter an institutional, professional services, or dental claim header and save, DIAMOND applies COB Hold/Deny rules. (Note that COB Hold/Deny processing occurs regardless of whether any COBHS records exist for the member.) Also, DIAMOND re-applies COB Hold/Deny rules when you modify the member number on a claim header using INCLM/PSCLM/DNCLM Special menu command Change Header.

Warning!   When you modify the claim header member number and save, DIAMOND deletes the entries in the following fields: Other Carrier Code, Other Carrier Allowed and OC Paid Amt. DIAMOND also resets the EOB Indicator to None. You must make new entries as needed.

  • If you have not defined any COB Hold/Deny rules, DIAMOND generates a processing message to that effect.

  • DIAMOND applies COB Hold/Deny rules for the matching claim type (institutional, professional services, or dental) in sequential order by Order Number. To be a match, the claim must satisfy all defined determinants for the Order Number. Selected determinants are available from the Institutional, Professional, or Dental claim header (depending on the claim type), COB History, Group Detail, and Member Master tables. (You may not use the columns INSERT_DATETIME, UPDATE_DATETIME, INSERT_USER, UPDATE_USER or INSERT_PROCESS as determinants.) When evaluating determinants, DIAMOND uses the group or member from the claim header being processed. When evaluating COB History determinant values, DIAMOND retrieves the corresponding values based on the Carrier Code selected on the claim header. When evaluating Group Detail determinant values, DIAMOND uses the claim primary date of service to retrieve the corresponding values.

  • If a match is found, DIAMOND applies the Action (Hold or Deny) and the Reason Code from the matching COB Hold/Deny rule to all claim detail lines as they are adjudicated. Note that once the first match is found, no further COB Hold/Deny rules are evaluated, nor is the COBCLAIMACTN parameter evaluated.

  • If a match is not found for any COB Hold/Deny rule or there are no COB Hold/Deny rules defined, DIAMOND proceeds with claims processing.

COB Hold/Deny Processing Restrictions:

  • DIAMOND does not automatically apply the COB Hold/Deny rules when a claim detail line is added or changed.

  • You cannot manually re-apply COB Hold/Deny rules.

  • If you add, delete or modify a COB Hold/Deny rule, DIAMOND applies the new set of COB Hold/Deny rules only when you log out of DIAMOND, log back in, and subsequently enter new claims or modify the claim header member number (as noted above) for existing claims.

  • COB Hold/Deny rules are not applied to prepaid claims.

COB and Provider Withhold Amounts

If the provider on the claim has a contract that includes a provider withhold amount but the claim is a COB claim that is automatically adjudicated, DIAMOND does not include the withhold amount in COB Net Amount calculations. Note: DIAMOND does still display the withhold amount on the claim line.

If the provider on the claim has a contract that includes a provider withhold amount but the claim is a COB claim that is manually adjudicated (meaning that after adjudication a not covered, copay, coinsurance, deductible, or withhold amount is manually entered or changed on the claim detail line), DIAMOND calculates the Net Amount including any withhold amount. (The COB calculation method on the group detail record is no longer used in determining the Net Amount.)

COB Calculation Methods and Examples

DIAMOND provides 12 COB calculation methods, each with specific fixed calculation logic. Using one of these calculation methods, you may coordinate additional coverage with the coverage directly on the claim. In DIAMOND coordination of benefits functionality, DIAMOND is the secondary payer.

The specific calculation logic for each of these methods is provided below, along with examples.

Certain methods are typically applied when the other carrier type is Medicare and others are typically applied when the other carrier type is Commercial, Other or any other non-Medicare type. The selection of a method, however, is up to you. DIAMOND applies whatever method you specify in your set-up, regardless of the carrier type of the other carrier.

Assumptions - Examples 1 - 2:

  • A Percent of Billed pricing method applies with percent of billed = 90%.

  • A 20% coinsurance benefit rule applies.

  • A $250 deductible benefit rule applies.

  • The benefit rule order in the benefit package specifies that the deductible benefit rule be applied before the coinsurance benefit rule.

Assumptions - Examples 3 - 12:

  • A Percent of Billed pricing method applies with percent of billed = 80%.

  • A 20% coinsurance benefit rule applies.

  • A limit benefit rule applies in some examples as indicated.

 

1. Integration of Benefits

Logic: Integration of Benefits allows benefits from multiple carriers to be added on top of each other so that the member receives the full benefit from their primary carrier and the secondary carrier pays their entire benefit up to 100% of allowed charges. DIAMOND calculations proceed as follows.

1. Calculate what DIAMOND would pay if Primary as follows.

Net Amount if DIAMOND was Primary = COB Allowed Amount - Total Not Cov - Total Copay - Total Co-ins - Total Deductible - Withhold Amt

Where:

COB Allowed Amount is determined per the Commercial COB Allowed Rule on the provider contract corresponding to the claim. COB Allowed Amount will be one of the following valid values: Allowed Amount, Lesser of Allowed or OC Allowed.

2. If DIAMOND Net Amount as if Primary + OC Pd Amt < COB Allowed Amt,

DIAMOND Net Amount = DIAMOND Net Amount as if Primary.

If DIAMOND Net Amount as if Primary + OC Pd Amt > COB Allowed Amt,

DIAMOND Net Amount = COB Allowed Amt - OC Pd Amt

3. Adjust benefit accumulators and patient liabilities if appropriate (see examples)

Integration of Benefits calculation logic uses several fields from various sources in DIAMOND, as detailed in the following examples. All values are user entered unless otherwise indicated.

Benefit Accumulators and Patient Liabilities Adjustment:

BENEF header "Apply Pt Liability to" = Lesser of Billed or Allowed or OC Allowed = 4000 (OC Allowed). DIAMOND calculates Patient Liabilities by applying the applicable benefit rules to this 4000 amt. In this example, DIAMOND applies a Deductible BRULE for 250 and a Co-ins BRULE for 20% of (COB Allowed - Deductible Amt) = 20 % x (4000-250) = 750.

DIAMOND similarly adjusts Patient Liabilities, setting Total Deduct = 0 and Total Co-ins = 0.

Since DIAMOND adjusted the Patient Liabilities, there is no patient liability covered by COB and the COB Pt Liab Cov (COB Pt Liability Coverage) = 0.

2. Non-Duplication of Benefits

Logic: Non-Duplication of Benefits ensures that benefits from two carriers are not duplicated or added on top of each other. DIAMOND calculations proceed as follows.

1. Calculate what DIAMOND would pay if Primary as follows.

Net Amount if DIAMOND was Primary = COB Allowed Amount - Total Not Cov - Total Copay - Total Co-ins - Total Deductible - Withhold Amt

Where:

COB Allowed Amount is determined per the Commercial COB Allowed Rule on the provider contract corresponding to the claim. COB Allowed Amount will be one of the following valid values: Allowed Amount, Lesser of Allowed or OC Allowed.

2. Subtract the Primary Other Carrier’s payment

3. The result is the DIAMOND Net Amount, that is:

DIAMOND Net Amount = Net Amount as if DIAMOND is Primary - Other Carrier Paid Amount

Exception: If the DIAMOND Net Amount is negative, set the DIAMOND Net Amount = 0.

COB Order of Liability Processing Logic

COB Order is a required field on Member COB History (COBHS) records. COB Order indicates whether the medical coverage provided by the Other Carrier selected on the COBHS record is primary, secondary, tertiary or other for the DIAMOND member and period specified on the record. (The Other Carrier is a carrier other than the carrier on the DIAMOND claim.)

When entering a new COBHS record, you can select the appropriate COB Order value manually or accept the COB Order field default value of O (Other) and let DIAMOND automatically assign the COB order based on rules you define in COB Order of Liability (COBOR). If you choose to let DIAMOND assign COB Order, DIAMOND applies the COBOR rules according to the following:

  • When you enter and save a new COBHS record with the COB Order field = O - Other, DIAMOND first checks to make sure the COBHS coverage effective date falls within one of the COBHS member’s eligibility effective periods. If not, a warning message is generated indicating that COB Order cannot be automatically assigned. If so, DIAMOND applies the COBOR rules, if any exist.

  • If no COBOR rules exist, the COB Order remains O - Other.

  • DIAMOND applies the COBOR rules in sequential order by Rule Order number. To be a match, the claim must satisfy all defined determinants for the Rule Order.

  • If a matching rule is found, DIAMOND populates the COB Order value from the matching COBOR rule to the COB Order field on the COBHS record. Once a matching COBOR rule is found, no other COBOR rules are applied.

  • If no matching COBOR rule is found, DIAMOND sets the COB Order = O - Other with a message that the COB Order cannot be automatically assigned. You can accept or override this value.

  • You can go back and override the COB Order assigned from COBOR rules.

  • If a COBOR rule requires that a date on the COBHS record be evaluated to determine if it is before, during or after a member’s DIAMOND coverage and there is more than one period of eligibility, the search begins in the most current period of eligibility. The search continues through all periods of eligibility until a match is found. If no match is found, DIAMOND sets COB Order = O - Other.

  • You can use the COBHS Special menu command Determine COB Order to reapply the COB Order assignment logic for an existing record.

  • Multiple COBHS records can exist with the same COB Order.

  • Multiple COBHS records cannot be evaluated when determining the COB Order of the current COBHS record.

  • When you enter or modify a COB Order rule and save, you must log out of DIAMOND and then log back in for the rule to take effect.

 

Warning:

Automatic COB Order of Liability assignment may give unreliable results if you use Relationship as a determinant in COBOR and select All, Include or Exclude in the COBHS - Other Family Coverage field. If you want to use Relationship as a determinant in COBOR rules, instead select None in the COBHS - Other Family Coverage field and create separate individual COBHS records for dependents.

COB Order - Sample Rules and Scenarios

You use COB Order rules when you want DIAMOND to assign a specific COB Order based on a particular set of circumstances. The following describes sample scenarios demonstrating how you define COB Order rules to generate a desired COB Order.

These scenarios simply demonstrate the manner in which you can set up rule determinants to achieve a desired result. The determinants and desired COB Order assignment at your site will depend on your own internal processes and requirements.

Some of the examples include dependent determination rules, which determine COB Order for dependent children. Two common dependent determination rules, the "Birthday Rule" and the "Gender Rule", are included in the examples below.

These rules are typically defined as follows:

  • Birthday Rule: When a dependent child has two coverages, the coverage whose Subscriber’s birthday occurs earlier in the calendar year (regardless of birth year) is Primary. For example, say a husband and wife both work and have insurance coverage through their respective employers. The wife is the Subscriber on her employer’s coverage and the husband is the Subscriber on his employer’s coverage. The husband’s birthday is 6/26/61 and the wife’s birthday is 1/11/63. So in the calendar year the wife’s birthday in January comes before the husband’s in June. So for any dependent children, with all else being equal and both plans using the Birthday Rule, the wife’s coverage is Primary and the husband’s Secondary.

  • Gender Rule: When a dependent child has two coverages and one coverage has a male Subscriber and the other a female Subscriber, the coverage with the male Subscriber is Primary. For example, say a husband and wife both work and have insurance coverage through their respective employers. The wife is the Subscriber on her employer’s coverage and the husband is the Subscriber on his employer’s coverage. For any dependent children, all else being equal and both plans using the Gender Rule, the coverage with the husband as Subscriber is Primary.

In DIAMOND, you select the dependent determination rule pertaining to a plan on the group detail record (GRUPD). If appropriate, you can also select a Dependent Determination Rule on the current COBHS record. If the two are the same, the rule is applied as defined above. If they are different, your internal processes must determine which rule takes precedence.

COB Warning Message in Authorizations

You can activate an automatic COB warning message in Authorizations (online and API) using the parameter COBAUTHACTN. Since the corresponding logic uses the authorization Request Date, activating COBAUTHACTN also makes authorization Request Date a required field.

If COBAUTHACTN Parameter 1 = Y - Yes, when you enter the Member ID on an authorization DIAMOND displays (Auth API returns) a COB warning message when the following are true:

  • The member has a COB History (COBHS) record where the authorization Request Date is within the COBHS record’s Effective/Term Date range, and

  • The COB History record has a COB Order = P - Primary

COB - Other Information

  • The COB calculation method considered on a claim detail line is reflected in the claim’s processing messages.
