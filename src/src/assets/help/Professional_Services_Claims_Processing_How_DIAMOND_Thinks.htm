Setting Up a Claims Batch

When you first display keyword PSCLM, the Claims Batch Initiation dialog box displays. It allows you to type a default claim batch number and received date to use as long as you remain in keyword PSCLM. Both fields are optional. If you do not enter default values, then during claims entry, the Batch Number box will be blank and the Date Received box will display the current (system) date. If necessary, you can type over the displayed batch number and received date. You also can change the defaults at any time from the Professional Services Claims header window using the Change Default Batch command on the Special menu.

Saving a detail line item

When you save a claim detail line item, DIAMOND compares the Adj Method on the claim detail line to the member’s associated group price/adjudication requirement. If the member’s group pricing is set to adjudicate (in keyword GRUPD, the Group Req list is set to A – Adjud), then you must adjudicate the line item before you save it.

For more information, see GRUPD, Group Detail.

Determining Claim Eligibility

How DIAMOND determines claim eligibility

First, DIAMOND uses the service date (Srv Date box) on the Professional Services Claims Header window to determine eligibility. However, if a through date (Thru Date box) is entered, then DIAMOND verifies the member’s eligibility based on both dates. A warning displays if the member is ineligible or if the member’s eligibility expired before the through date.

If DIAMOND determines the claim is eligible, then it next performs checks for Coordination of Benefits (COB), student dependent status, and pre-existing conditions. DIAMOND displays a warning message if exceptions to eligibility are found.

Also, during adjudication DIAMOND validates that the date of service (Date box) falls within the date range of one of the member’s eligibility periods. If it does not, then the line item is denied. DIAMOND sets the Claim Status list to D - Denied, sets the Allowed Amt and Total Not Cov boxes to $0.00, and assigns the Reason code associated with reason code type ME as the Not Cov Rsn code. Reason code type ME (only one is allowed) must exist in keyword REASN.

What happens when a claim is ineligible

If the member is ineligible, the claim date of service is outside of their eligibility period, then the claim is denied. If the eligibility status on the member record is invalid because it does not match the valid codes stored in the ELIGSTAT parameter, then the claim is held

There are additional methods that can automatically hold a claim. For more information, see Claim Holds, How DIAMOND Thinks.

You can manually price, adjudicate, or expense a professional services claim when a member is outside an eligibility period. However, because there is no group/plan information on the claim header DIAMOND cannot validate the Group Req. indicator to set the appropriate flag on the Pre_Price_Only column, the claim detail line items will not be finalized.

Eligibility changes after a claim is processed

DIAMOND takes a "snapshot" of the member’s benefit package/plan/rule(s) to be used during the claim adjudication process as soon as the member identification is displayed. If the associated benefit package/plan/rules (BENEF, BRULE) are changed after the benefits snapshot, the benefit changes are not immediately updated on the still open claim header window and DIAMOND uses the benefit snapshot (old benefit data) if the claim is adjudicated during the session. However, the updated benefit data will display when you exit and then re-open the claim. If the claim was adjudicated using old benefits (snapshot) data, then on the Special menu, point to Pricing and Adjudication, and then click Adjudicate Claim to re-adjudicate the claim to include the updated (changed) benefit data.

About Pricing

Each claim detail line may be priced (allowed amount calculation only) or fully adjudicated. The Group Req list in keyword GRUPD (Group Detail) controls whether claims for a group are priced or fully adjudicated.

Pricing of claim detail lines may be based on fee schedules, percentages of billed charges, conversion factors, per-diems, diagnostic-related groups, or a combination of methods. Claim pricing may be driven by the group contract, the provider contract, specific procedure code, or a combination of these factors.

You can also manually price a claim detail line by overriding the calculated Allowed Amount for either a new claim detail line or any existing unfinalized detail line. For more information, see Manually Pricing a Claim Detail Line.

Covering providers

A covering provider is, usually, a physician who provides services in the absence of a PCP to that PCP’s assigned members. When a covering provider renders services in this role, he or she may be paid according to the PCP’s contract rather than his or her own, or may not, depending on the rules of the health plan, the nature of the contracts, and health plan practices.

DIAMOND provides a mechanism for recognizing potential covering provider situations and allowing the covering provider to be paid according to a PCP’s provider contract PROVC record. You can set up covering providers with a default action, use the PCP’s contract or use their own contract, and this default action can also be overridden manually during claims processing.

When you change the default action using the Covering Provider Information window:

  • DIAMOND performs a provider contract search based on the data entered in the window.

  • DIAMOND compares the provider contract data currently existing on the claim against the provider contract data derived as a result of the new provider contract search.

  • DIAMOND takes no action if the contract data in #2 above is the same and DIAMOND displays a "Change successful" message.

  • If the contract data in #2 above is different, DIAMOND updates the claim contract data and displays a "Change successful. Go to detail window and readjudicate the entire claim" message.

In this last case, in order to employ the change in pricing logic you must use the Special menu command Adjudicate Claim.’

Pricing anesthesia claims

When a Procedure code with a code class of "A" (on the PROCD record) or if the Anesthesia check box is selected on the Modifier record in keyword MODIF, then the Anesthesia Time window will display automatically. The Anesthesia Time window may also be selected manually.

To enter time for anesthesia, you may enter either the start and end time or the total elapsed time. If you enter start and end times DIAMOND calculates the elapsed time as long as the time period is less than 24 hours. If greater than 24 hours, you must calculate and enter the total elapsed time manually.

For more information, see Pricing Anesthesia Time, How DIAMOND Thinks.

Allowed greater than billed

When the Line of Business record is set up, one of the parameters is a check box that permits the calculated allowed amount to be greater than the entered billed amount. In the absence of this selection, when professional services claims associated with this line of business are priced, DIAMOND:

  • Prices the claim detail line item (determines the allowed amount).

  • Compares the allowed amount to the claim detail line billed amount.

  • Lowers the allowed amount to the billed amount if the allowed amount is greater than the billed amount.

If this option is selected, when professional services claims associated with this line of business are priced, DIAMOND:

  • Prices the claim detail line item (determines the allowed amount).

  • Permits the allowed amount to be greater than the billed amount.

  • Does not lower the allowed amount to the billed amount if the allowed amount is greater than the billed amount.

If the allowed amount is greater than the billed amount and the BENEF>Apply Pat Liability To = "Lsr B/A" or "Lsr B/A/OA" for the associated (applied) benefit rule, DIAMOND displays the difference in the PSCLM>Claim Detail Window > Add Back Amt field. See BENEF, Apply Pat Liability To.

Only professional services claims are affected by this check box. Institutional claims pricing logic permits the allowed amounts to be greater than billed amounts. When this occurs, DIAMOND displays the difference in the INCLM>Claim Detail Window > Add Back Amt field.

See keyword BENEF (Benefit Packages) for information regarding the selection of a benefit package’s claim detail line amount type. DIAMOND uses the selected amount type selected when it calculates patient liabilities.

About Adjudication

Adjudication is the process in which benefit rules, determined by a member’s benefit package, are applied to the claim detail line. This results in the calculation of deductibles, not-covered amounts, co-payments, coinsurance, and exclusions that may reduce the amount paid to the provider. When a claim is adjudicated, each claim detail line is both priced (an allowed amount is calculated) and adjudicated (benefit rules are applied). See Benefit Package Maintenance (BENEF), Benefit Rule Maintenance (BRULE), and Group Detail (GRUPD) User Documentation for additional information.

As part of the adjudication process, DIAMOND includes Coordination of Benefits (COB) processing when applicable. See Coordination of Benefits Overview and related topics for more information on COB.

Calculating the Net Amount

DIAMOND calculates the claim detail line Net Amount differently depending on whether the line is automatically or manually adjudicated, and whether a COB calculation method applies.

  1. When DIAMOND automatically adjudicates a claim line and no COB applies:

Net Amount = Allowed Amt - Total Not Cov - Total Copay - Total Co-ins - Total Deduct - Pt Liability amounts for Benefit Rule type 99 (Other) - Withhold Amt

In some cases, the sum total of the Withhold Amt and the patient liabilities (including not covered amounts) may be greater than the Allowed Amt. In these situations, rather than placing a negative dollar amount in the Net Amount field, DIAMOND sets the Net Amount field to 0 and stores the overage figure in the Withhold Surplus field.

The Total Co-ins amount is the sum of all calculated coinsurance amounts. DIAMOND calculates each coinsurance amount as the benefit rule coinsurance percentage (BRULE Coinsure %) times the claim detail line allowed amount net of any other benefit rules applied before the current coinsurance rule. (You set the processing order for benefit rules in each benefit package.) That is, DIAMOND calculates coinsurance on a decreasing total based on the application of any prior ordered benefit rules.

Example:

Allowed Amount = 200.00

Benefit Rules in processing order:

1. Copay = 10.00

2. Deductible = 50.00

3. Coinsurance1 = 20%

4. Coinsurance2 = 10%

Coinsurance1 and Coinsurance2 amounts are calculated as follows:

200.00 (Allowed Amount) - 10.00 (Copay) = 190.00

190.00 - 50.00 (Deductible) = 140.00

140.00 x 20% (Coinsurance1) = 28.00

140.00 - 28.00 = 112.00

112.00 x 10% (Coinsurance2) = 11.20

112.00 - 11.20 = 100.80

  1. When DIAMOND automatically adjudicates a claim line and a COB calculation method applies:

DIAMOND performs the specific calculation logic of that particular COB calculation method. See Coordination of Benefits – How DIAMOND Thinks for details and examples of COB calculation methods.

Note  For a COB claim that is automatically adjudicated, if the provider on the claim has a contract that includes a provider withhold amount, DIAMOND does not include this amount in COB and Net Amount calculations. DIAMOND does still display the Provider withhold amount on the claim line.

  1. When you manually adjudicate a claim line (that is, after automatic adjudication you manually enter or change a total Not Cov, Total Copay, total Co-ins Total Deduct amount or Withhold amount), whether or not a COB calculation method applies:

Net Amount = Allowed Amt or OC Allowed (per COB rules) - Total Not Cov - Total Copay - - Total Co-ins - Total Deduct - OC Paid Amt - Pt Liability amounts for Benefit Rule type 99 (Other) - Withhold Amt

In some cases, the sum total of the Withhold Amt and the patient liabilities (including not covered amounts) may be greater than the Allowed Amt. In these situations, rather than placing a negative dollar amount in the Net Amount field, DIAMOND sets the Net Amount field to 0 and stores the overage figure in the Withhold Surplus field.

Changing or adjusting automated benefit calculations

Changing the entries in the not covered, copayment, coinsurance, deductible, and reason code fields can change the results of automated adjudication. However, when more than one exclusion (not covered), copay, coinsurance, or deductible benefit rule has been applied to a claim detail line, the corresponding Total Not Cov, Total Copay, Total Co-ins, and/or Total Deduct boxes are grayed out and unavailable for entry, as are the corresponding reason code fields (which are filled with asterisks).

You must use the Patient Liabilities window to view/change multiple underlying Copay, Coinsurance, Limits (Not Covered), Deductible, and/or Exclusions benefit rules/amounts that constitute the total amounts displayed in the Total Not Cov, Total Copay, Total Co-ins, and/or Total Deduct fields on the INCLM, Claim Detail Window. You must also use the Patient Liabilities window to add an "Other Amount."

The Patient Liabilities window has four focus boxes. The first focus box displays claim header member/provider information as displayed on the claim header window. The second focus box is a picklist of claim detail line items. The third focus box is a picklist of benefit rule data for each benefit rule that has been applied to the claim detail line selected in the second focus box. The fourth focus box is a data entry/display area of Rule Type, Amount, Benefit Rule (ID) and Reason Code data for the benefit rule selected in the third focus box.

Only detail records that have not been finalized, but have been adjudicated by DIAMOND, are available to change. To make changes enter a new Amount and if necessary a new Reason code.

If you are changing an automatically determined reason code, be aware that your change will not be validated. In addition, accumulations under the member’s benefit plan might not be affected by the change, since they are added together based on reason codes.

DIAMOND changes the BRULE ID here to a default value, SYSTEM, and the Adj Method second character to "M" (adjudicated manually) when you change or add a member liabilities record using this window.

Adding Patient Liabilities Records

Patient Liabilities added using this function (rather than automatically by DIAMOND) are not added to the member’s accumulated benefits history.

Press CTRL+N to add a member liabilities record in addition to the DIAMOND adjudicated member liabilities records, and the following fields are available for entry.

  • Amount

  • Rule Type: You may use Rule Type 99 (Other) for patient liabilities that do not fit within the other rule type categories. Rule Type 99 is a valid rule type only for the Patient Liabilities window. This rule type cannot be entered in keyword BENEF (Benefit Package Maintenance). Rule Type 99 amounts from these records are not included in the Total Not Cov, Total Copay, Total Co-ins, or Total Deduct fields displayed on the claims detail window; however, these amounts are included in the Net Amt displayed on the claim detail window.

  • Reason Code

Discount and Interest Calculations

Discount or interest calculations can be automatically applied against payable claims and reversed payable claim lines. Interest calculations can take into account any unclean days tracked on a claim. A claim may be considered unclean if it requires additional substantiating documentation from a source outside of your organization and if the missing information prevents you from paying the claim timely.

You can manually apply a discount or interest amount on an adjusted claim line.

For a complete explanation of discount and interest calculations, see Claims Discount and Interest Calculations, How DIAMOND Thinks.

About Processing Messages

System or user-defined processing messages may display when you enter, price, or adjudicate claims. They may display on both the Professional Services Claims header and detail windows. Some messages display automatically and some only display when you select the Processing Messages command on the Window menu in claims.

On a claim detail line item, an "X" displays when one or more header level or detail level messages are present for the current claim detail line item.

System messages are primarily defined in keywords MSGMT (Message Maintenance) and PMSCT (Processing Message Control).

User-defined messages may be defined in keywords PRULD (Price Rule Detail), PRULW (Whole Claim Pricing Rules), and BRULE (Benefit Rule).

Providers

Out of plan providers and one-time providers

When a provider is considered "out of plan," you can use a non-specific provider. A non-specific provider has a "Y" in the Non Specific Prov field on the Provider Master file (PROVF) record. If you use a non-specific provider, you must specify a vendor during claims processing. Use of the non-specific provider and default provider features to handle out of plan or one-time providers requires careful setup. See Provider Contracts, How DIAMOND Thinks for a complete explanation of this feature and its use.

Multiple vendor addresses

In addition to selecting the correct vendor record, DIAMOND prompts the user to select the address to which payment should be sent in cases where the vendor has multiple address records. A list of valid addresses is displayed, and the user picks from this list.

Provider contract data

DIAMOND uses provider contract search logic to verify provider contract data. See Line of Business (LINBS) documentation for more information on the provider contract search.

Linking with Accounts Payable

Keyword APUPD (Accounts Payable Update) creates A/P claims records and selects either medical (claims) or capitation records for payment. At the same time the originating records in claims and capitation are finalized. As part of the finalization of the original claims records a transaction number is generated and appropriate general ledger accounts are updated. Once claim detail lines are finalized using APUPD, they cannot be changed; however, finalized claim detail lines can be adjusted or reversed (as can unfinalized lines).

Use keyword PROVC (Provider Contracts) to set up the printing of Remittance Advices (RAs) for a specific provider. Use keyword LINBS (Line of Business) to set up the printing of Explanation of Benefits (EOBs) for a specific line of business.

You can access/override the RA and EOB printing defaults for a specific claim using the Payment Detail command on the Special menu. RA and EOB printing status codes are also displayed on accounts payable records (ACPAY, RA/EOB Print Flag).

Deleting Claim Headers and Detail Line Items

You can delete both claim detail line items and claim header records. However, you cannot delete a claim header if it includes any detail line items. To delete a claim header that includes detail line items, first delete the line items.

You cannot allow a user to delete claim detail line items but not claim header records. You can allow a user to delete both claim header and detail line items, or allow the user to delete neither header nor detail records.

Learn how to delete a claim header or detail line item.
