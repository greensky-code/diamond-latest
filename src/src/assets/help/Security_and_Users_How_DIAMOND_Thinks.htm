SWIND Keywords and Sheets

 

Add a user ID in keyword SUSER (Users). The final step in this set up process is a menu command, Add Database User. This step has two functions, it sets the user’s password and it connects that user ID to two DIAMOND system-wide Oracle roles, DIAMONDVIEW and DIAMONDUPDATE.

How DIAMONDVIEW Works

DIAMONDVIEW lets users see DIAMOND data. The DIAMONDVIEW role exists for every DIAMOND user even when not in a DIAMOND session. This role is used when someone writes a report using DIAMOND data with an external report writer. In order to connect a report writer to the DIAMOND tables, a user ID and password are required, and when a user enters his or her DIAMOND ID and password this DIAMONDVIEW role is invoked and data can be read ('selected') from DIAMOND tables. DIAMONDVIEW does not allow users to update data, within or outside of the DIAMOND session.

How DIAMONDUPDATE Works

The DIAMONDUPDATE role exists system-wide in DIAMOND. This role allows users to not only read data, but also to add, change, and delete data (and as limited for each user ID defined in keywords SUSER, SWIND and SFUNC). Unlike DIAMONDVIEW, a role that persists within and outside of DIAMOND sessions, DIAMONDUPDATE is invoked only when a user signs on to DIAMOND and is terminated when that user logs off. This role cannot be invoked external to a DIAMOND session. Therefore, a user cannot use DIAMONDUPDATE privileges to change data using an external tool.

The DIAMONDUPDATE role is password protected; however, it is system-wide, invisible to users, and created automatically by DIAMOND. The only time your security or database administration staff need be aware of this password is in the rare instance in which you must change it. For example, when an employee who is privy to the password leaves your employ, or if you have organizational security guidelines which require passwords be updated. To change this system-wide password, use the special menu command in SUSER, Update Role Password. This new password is automatically invoked by DIAMOND users when the DIAMONDUPDATE role is activated.

Power Users and the DIAMONDADMIN Role

In some cases you may find it necessary to empower a user to access DIAMOND with an external database tool. There is a separate Oracle role called DIAMONDADMIN created for this purpose. DIAMONDADMIN must be explicitly granted by a database administrator to a user. It is not password protected and is invisible to the DIAMOND application.

To use DIAMONDADMIN, see your database administrator or security officer. Obviously, this role should be used sparingly, if at all.

How SUSER and SWIND Work Together

Super Users and "Regular" Users

When you add a super user in keyword SUSER, you do not grant that user window access in keyword SWIND. For all other users and templates you must create an SWIND record which grants access to keywords (and associated add, delete, change, and view capabilities).

Claim Limits and Affiliations

At first glance it seems as if you can set up contradictory user privileges in SUSER and SWIND. For example, you can establish access to certain claim records using affiliations in SUSER even if the user does not have specific access to claims-related keywords in SWIND.

While it is possible to set a user up this way, i.e. DIAMOND will not edit on these screens for illogical entries, the result is governed by SWIND. So, regardless of claim security attributes in SUSER, if the user does not have access in SWIND, there is not access at all.

It is helpful to think of this in a negative access sense when setting up SUSER claim security attributes. SWIND grants access to claims, and SUSER limits that access to certain records, even if you use a positive affiliation set up, i.e. Include, not Exclude. In both cases you are limiting the claims access, not creating claims access.

How Claim Limit Security is Calculated

Claim limit attributes must be thought of as a set. If any one field in the Claim Limit section is entered, all fields must be completed. The workings of these fields are completely interdependent.

The arithmetic is fairly simple. DIAMOND will add up the detail lines to a maximum, and if it exceeds the user’s Claim Limit, the detail lines are put on hold.

Warning!   If a user with a claim limit accesses a claim processed via DIAMOND’s batch process and the claim amount exceeds the user’s claim limit, the claim will be held, regardless of whether the operator made any changes to the claim because the claim logic is reinvoked.

Which amount from the detail line is added up?

This is specified in the Determinant field, you may choose from Allowed, Billed or Net.

What hold reason code is used?

You enter the code in Reason Code. It must be a code that exists in keyword REASN with a type SC.

Which detail lines are added up?

All the detail lines on the claim, unless they have a Claim Status of D (denied) or I (informational), or a Processing Status of F (finalized), P (paid), or N (paid, no check).

Once a claim has an SC reason code applied, how can the claim be released from hold?

Only a user whose claim limit exceeds the amount of the claim can remove an SC hold and finish the claim’s processing.

Affiliation Security

Affiliation security is available for member, authorization, and claims records only.  

Each affiliation is evaluated separately and only records which meet the conditions for all affiliations are accessible to the user.  

For each affiliation you can use either an Exclude logic or an Include logic; you cannot both include and exclude values for a single affiliation.   However, of course, you can use Include for providers, for example, and exclude for groups.

When imposed, affiliation security applies to the following keywords:

Members

ALIAS, COBHS, GROUP, GRPCT, GRUPB, GRUPC, GRUPD, MELIG, MEMBA, MEMBB, MEMBR, MEMMC, and MEMWA.

Authorizations

APPEL, AUTHS, DVEXT, PHYAD, PRCDR, and SECOP.

Claims

INCLM, PSCLM, DNCLM, DSPBN, and CLDSP.

Field Level Security

You can apply security restrictions at the field level as well as the keyword level. Using keyword SFLDL (Field Level Security), you can allow users to view, but not change data in fields; you can also prohibit users from viewing data in fields.

There are two levels of field level security restriction:

  • V – View: Users cannot view any data in a field with a V restriction.

  • C – Change. Users may view data in a field with a C restriction. They cannot add, delete, or change any data in the field.

When you restrict access to a keyword field using SFLDL, you restrict access to the underlying DIAMOND database column that stores data from the restricted field. Any DIAMOND field or table view that displays data from the restricted database column inherits the same security restriction. Fields in special functions cannot be protected by field level security. To protect special function fields, you must restrict access to the special function using SFUNC.

Note   Field level security only applies to DIAMOND front-end access. It does not restrict access to the DIAMOND database using 3rd party products or ad hoc database query tools.

Field Level Security and SWIND

You do not need to restrict a user’s access to specific fields in a keyword if the user does not have access to the keyword. If users have not been granted access to a specific keyword through keyword SWIND, it is not necessary to restrict access further using SFLDL.

Note   If a user has been granted delete privileges for a keyword in SWIND, that user may delete an entire record even if field level security was applied to some or all fields in the keyword.

Password Profile Rules

Password profile rules define the minimum acceptable standards for a user’s password and how it may be used. These rules are only enforced for user accounts which have been assigned the password profile in keyword SUSER.

The password profile requires that:

  • Passwords must be between 6 and 30 alphanumeric characters in length.

  • Passwords must start with an alpha character; they may not start with a number.

  • Passwords may use only alpha characters, or a combination of alpha characters and numbers.

  • Passwords may contain the special characters _ (underscore), $ (dollar sign), and # (pound). No other special characters may be used in a password.

The password profile also limits how long a password may be used, when it may be reused, and the number of login attempts allowed before the account is locked. These restrictions are determined by your database administrator.

User passwords can be managed as follows:

  • A password may be changed by checking the Change password check box on the DIAMOND login screen, then following the prompts.

  • The User Password Profile is applied to a DIAMOND user account in keyword SUSER using the Use Password Profile check box.

  • If DIAMOND locks a user account, a user with security level access can unlock the account using the Unlock User Account menu command in SUSER.

  • A user account password can be reset using the Reset Database Password menu command in SUSER. This menu command is restricted to users with security level access to DIAMOND.
