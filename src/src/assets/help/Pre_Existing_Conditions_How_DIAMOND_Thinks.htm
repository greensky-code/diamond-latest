What is a Pre-Existing Condition?

A pre-existing condition is a condition for which a member has received treatment during a specified period of time before becoming covered under a health plan. You can set up DIAMOND to use Pre-Existing Conditions logic, which manages members with pre-existing conditions.

There are two types of pre-existing conditions:

  • Member pre-existing conditions

  • Default line of business pre-existing conditions

You can enter member pre-existing conditions using keyword MCOND (Member Conditions). For more information, see MCOND, Member Conditions.

You can enter default line of business pre-existing conditions using keywords PXRUL (Pre-Existing Condition Rule) and LBPEC (Line of Business Pre-Existing Conditions).

Pre-Existing Conditions processing logic includes both types of pre-existing conditions. Pre-Existing Conditions processing logic is managed in the following general order:

  1. In keyword MELIG (Member Eligibility History), DIAMOND examines the PEC Waived check box.

  2. In keyword GRUPD (Group Detail), DIAMOND examines the Exclude Capitated Claim Lines field.

  3. In keyword MCOND (Member Conditions), DIAMOND evaluates the pre-existing conditions criteria defined at the member level when the field Condition Type equals Pre-existing, Rated-up, or Ridered-out.

  4. In keyword LBPEC (Line of Business Pre-Existing Conditions), DIAMOND evaluates the default pre-existing conditions criteria defined at the line of business and group level.

  5. In keyword GRUPD (Group Detail), DIAMOND examines the Apply PEC to PCP check box.

  6. In keyword MEMBR (Member Master), DIAMOND evaluates the PEC Waived check box and PEC End Date box.

  7. In keyword GRUPD (Group Detail), DIAMOND evaluates the Exclusion Type and Exclusion Period boxes.

How Pre-Existing Conditions Logic Works

After completing pricing, medical definitions, and adjudication of a claim (Institutional, Professional Services, or Dental), DIAMOND logic examines keywords MELIG, GRUPD, MCOND, AND LBPEC for pre-existing condition data. DIAMOND logic does not examine Pre-Existing Conditions logic for denied claim detail lines.

Institutional Claim Pre-Existing Conditions logic

DIAMOND pre-existing conditions logic for Institutional claims proceeds as follows:

1. Evaluate the keyword MELIG PEC Waived check box on the member eligibility period corresponding to the claim.

If selected, do not perform Pre-existing Conditions logic  

If cleared, perform Pre-Existing Conditions logic

2. Evaluate keyword GRUPD Exclude Capitated Claim Lines field for the member's group.

Exclude capitated claims from further Institutional Claim Pre-Existing Conditions logic if Exclude Capitated Claim Lines equals Institutional or Both. (Possible values for Exclude Capitated Claim Lines include claim types Institutional, Professional, Both (Institutional and Professional), or None.)

3. Use the Member ID on the claim header to locate the member's keyword MCOND member condition records and determine which are pre-existing conditions records.

If the MCOND member condition record field Condition Type is Pre-existing, Rated-up, or Ridered-out, the record is a pre-existing condition record. Continue with Pre-Existing Conditions logic for each such record.

If the MCOND member condition record field Condition Type is not Pre-existing, Rated-up, or Ridered-out, the record is not a pre-existing condition record. Do not include each such record in further Pre-Existing Conditions logic.

4. Evaluate the claim against each of the member's MCOND pre-existing condition records. For Institutional claims, use the claim header primary diagnosis code Dx1 and Admit date as the diagnosis code and date of service for evaluation purposes.

Once one matching MCOND pre-existing condition record is found and an MCOND Claim Action applied, do not evaluate the claim any further.

a. Compare the claim header primary diagnosis code Dx1 to the MCOND record diagnosis From Value through Thru Value diagnosis range.

If the claim header primary diagnosis code Dx1 is within the MCOND record diagnosis From Value through Thru Value diagnosis range, continue to step 4b and evaluate the date of service.

If not, go to the next MCOND record for the member and evaluate the diagnosis code. If no more MCOND records exist for the member, go to step 5.

b. Compare the claim header Admit date to the MCOND effective period. The MCOND effective period is the MCOND Effective Date through Term Date, unless no Term Date is specified. If no MCOND Term Date is specified, use the MELIG PEC End Date from the member eligibility record corresponding to the claim.

If the claim header Admit date is within the MCOND record effective period, apply the MCOND record Claim Action (Pay, Hold, or Deny) and Reason, and stop.  

If the claim header Admit date is not within the MCOND effective period, go to STEP 4A and evaluate the next MCOND record for the member. If no more MCOND records exist for the member, go to step 5.

5. Identify each LBPEC record that matches the line of business and group of the member on the claim.

6. For each matching LBPEC record, use the effective PEC Rule IDs to identify the corresponding PXRUL default pre-existing condition rules. Then evaluate the claim against the PXRUL rules identified.

a. Compare the claim header Admit date to the effective period of the PEC Rule ID on the LBPEC record. The PEC Rule ID effective period is the LBPEC Effective Date through Term Date for that PEC Rule ID, unless no Term Date is specified. If no LBPEC Term Date is specified, use the MELIG PEC End Date from the member eligibility record corresponding to the claim as the term date.

If the claim header Admit date falls within the PEC Rule ID effective period, continue the evaluation for that PEC Rule ID.

If not, go to the next PEC Rule ID on the LBPEC record and evaluate it in the same manner. If no more PEC Rule IDs exist on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record. If no more matching LBPEC records exist, go to step 7.

b. Use the PEC Rule ID to locate the corresponding PXRUL rule.

c. Compare the claim header primary diagnosis code Dx1 to the PXRUL rule diagnosis code ranges. The PXRUL rule diagnosis code ranges are defined in each row in the PXRUL section box Diagnosis Data as the From Value through Thru Value.

If the claim header primary diagnosis code Dx1 is within one or more PXRUL rule diagnosis code ranges, go to step 6d and evaluate the date of service.

If not, go to step 6a and evaluate the next PEC Rule ID on the matching LBPEC record. If no more PEC Rule IDs remain to be evaluated on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record. If no more matching LBPEC records exist, go to step 7.

d. Compare the claim header Admit date to the PXRUL effective period. The PXRUL effective period is the PXRUL Effective Date through Term Date. If no Term Date is specified, the effective period is open ended.

If the claim header Admit date is within the PXRUL effective period, go to step 6e.

If not, go to step 6a and evaluate the next PEC Rule ID on the matching LBPEC record. If no more PEC Rule IDs remain to be evaluated on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record. If no more matching LBPEC records exist, go to step 7.

e. For each matching diagnosis code, compare the claim header Admit date to the diagnosis code's keyword DIAGN (Diagnosis) record effective period. The DIAGN record effective period is the DIAGN Effective Date through Term Date, unless no Term Date is specified. If no Term Date is specified, use the keyword MELIG PEC End Date instead.

If the claim header Admit date falls within the DIAGN record effective period, place the claim on hold and stop.

If the claim header Admit date does not fall within the DIAGN effective period, evaluate the next matching diagnosis code.

If no further matching diagnosis codes exist, go to go to step 6a and evaluate the next PEC Rule ID on the matching LBPEC record.

If no more PEC Rule IDs remain to be evaluated on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record.

If no more matching LBPEC records exist, go to step 7.

7. Evaluate the date in the keyword MEMBR PEC End Date box.

If the PEC End Date box is blank, skip to step 10 and evaluate the keyword GRUPD Exclusion Type and Exclusion Period.

If the PEC End Date box contains a date, continue to step 8 and evaluate the keyword GRUPD Apply PEC to PCP check box.

8. Evaluate the GRUPD Apply PEC to PCP check box.

If the check box is selected or the service is not provided by the member's PCP, continue to step 9.

If the check box is cleared and the member's PCP or a covering provider is performing the service, then stop Pre-Existing Conditions logic.

9. Compare the claim header Admit date to the date in the PEC End Date box in keyword MELIG.

If the claim header Admit date is before or equal to the PEC End Date, then hold the claim line (set Proc Status to H-Held) with the reason code from the Reason box in keyword MELIG.

If the claim header Admit date is past the PEC End Date, continue to step 10.

10. Evaluate keyword GRUPD Exclusion Type and Exclusion Period boxes.

If they are blank, stop pre-existing conditions logic for this claim.  

If they have values entered, calculate the exclusion period end date as the member's earliest keyword MELIG effective date for that group or plan plus the number of days or months specified by keyword GRUPD fields Exclusion Type (days or months) and Exclusion Period (number of days or months).

11. Compare the claim header Admit date to the calculated exclusion period end date.

If the claim header Admit date is before or equal to the end date, hold the claim line (change Proc Status to H - Held) with the reason code from the keyword GRUPD Reason box.

If the claim header Admit date is past the end date, then stop.

In all cases where a claim line is held, the message "Possible pre-existing condition, line pended" displays.

If pricing only is set up, Pre-Existing Conditions logic is still invoked after pricing.

For institutional claims, if an informational line is assigned a hold code prior to the Pre-Existing Conditions logic check, pricing and pre-existing checks do not occur.

Professional Services and Dental Claim Pre-Existing Conditions logic

DIAMOND pre-existing conditions logic for Professional Services and Dental claims proceeds as follows:

1. Evaluate the keyword MELIG PEC Waived check box on the member eligibility period corresponding to the claim.

If selected, do not perform Pre-Existing Conditions logic

If cleared, perform Pre-existing Conditions logic

2. Evaluate keyword GRUPD Exclude Capitated Claim Lines field for the member's group.

Exclude capitated claims from further Professional Services claims Pre-Existing Conditions logic if Exclude Capitated Claim Lines equals Professional or Both. (Possible values for Exclude Capitated Claim Lines include claim types Institutional, Professional, Both (Institutional and Professional), or None.) Dental claims are not addressed since Dental claims processing does not include capitation.

3. Use the Member ID on the claim header to locate the member's keyword MCOND member condition records and determine which are pre-existing conditions records.

If the MCOND member condition record Condition Type box is Pre-existing, Rated-up, or Ridered-out, the record is a pre-existing condition record. Continue with Pre-Existing Conditions logic for each such record.

If the MCOND member condition record Condition Type box is not Pre-existing, Rated-up, or Ridered-out, the record is not a pre-existing condition record. Do not include each such record in further Pre-Existing Conditions logic.

4. Evaluate each claim detail line on the claim against each of the member's MCOND pre-existing condition records. Use the claim line from Date as the date of service for evaluation purposes. Also, use the claim line Dx Ptr boxes to locate the corresponding diagnosis codes on the claim header or Other Claim Info window and use them for match purposes.

Once one matching MCOND pre-existing condition record is found and an MCOND Claim Action applied, do not evaluate the claim any further.

a. Compare the claim line from Date to the MCOND effective period. The MCOND effective period is the MCOND Effective Date through Term Date, unless no Term Date is specified. If no MCOND Term Date is specified, use the MELIG PEC End Date from the member eligibility record corresponding to the claim.

If the claim line from Date is within the MCOND record effective period, continue to step 4b and evaluate the diagnoses.

If the claim line from Date is not within the MCOND effective period,   evaluate the next MCOND record for the member. If no more MCOND records exist for the member, evaluate the next claim line in the same manner. If no more claim lines exist, go to step 5.

b. Compare each claim line diagnosis code to the MCOND record diagnosis From Value through Thru Value diagnosis range.

If the claim line diagnosis code is within the MCOND record diagnosis From Value through Thru Value diagnosis range, apply the MCOND record Claim Action (Pay, Hold, or Deny) and Reason to the line. Do not include this claim line in further evaluations. Go to step 4a and evaluate the next claim line. If no more claim lines exist, go to step 5.

If not, evaluate the next diagnosis code on the claim line. If no more diagnosis codes exist on the claim line, go to step 4a and evaluate the next MCOND record for the member. If no more MCOND records exist for the member, go to step 4a and evaluate the next claim line. If no more claim lines exist, go to step 5.

5. Identify each LBPEC record that matches the line of business and group of the member on the claim.

6. For each matching LBPEC record, use the effective PEC Rule IDs to identify the corresponding PXRUL default pre-existing condition rules. Then evaluate each claim line against the PXRUL rules identified. Use the claim line from Date as the date of service for evaluation purposes. Also use the claim line Dx Ptr boxes to locate the corresponding diagnosis codes on the claim header or Other Claim Info window and use them for evaluation purposes.

a. Compare the claim line from Date to the effective period of the PEC Rule ID on the LBPEC record. The PEC Rule ID effective period is the LBPEC Effective Date through Term Date for that PEC Rule ID, unless no Term Date is specified. If no LBPEC Term Date is specified, use the MELIG PEC End Date from the member eligibility record corresponding to the claim as the term date.

If the claim line from Date falls within the PEC Rule ID effective period, go to step 6b and continue the evaluation for that PEC Rule ID.

If not, go to the next PEC Rule ID on the LBPEC record and evaluate it in the same manner. If no more PEC Rule IDs exist on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record. If no more matching LBPEC records exist, go to step 6a and evaluate the next claim line. If no more claim lines exist, go to step 7.

b. Use the PEC Rule ID to locate the corresponding PXRUL rule.

c. Compare the claim line from Date to the PXRUL effective period. The PXRUL effective period is the PXRUL Effective Date through Term Date.

If the claim line from Date is within the PXRUL effective period, go to step 6d.

If not, go to step 6a and evaluate the next PEC Rule ID on the matching LBPEC record. If no more PEC Rule IDs remain to be evaluated on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record. If no more matching LBPEC records exist, go to step 6a and evaluate the next claim line. If no more claim lines exist, go to step 7.

d. Compare the claim line diagnosis codes to the PXRUL rule diagnosis code ranges. The PXRUL rule diagnosis code ranges are defined in each row in the PXRUL section box Diagnosis Data as the From Value through Thru Value.

If a claim line diagnosis code is within one or more PXRUL rule diagnosis code ranges, go to step 6e.

If not, go to step 6a and evaluate the next PEC Rule ID on the matching LBPEC record. If no more PEC Rule IDs remain to be evaluated on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record. If no more matching LBPEC records exist, go to step 6a and evaluate the next claim line. If no more claim lines exist, go to step 7.

e. For each matching diagnosis code, compare the claim line from Date to the diagnosis code's keyword DIAGN (Diagnosis) record effective period. The DIAGN record effective period is the DIAGN Effective Date through Term Date, unless no Term Date is specified. If no Term Date is specified, use the keyword MELIG PEC End Date instead.

If the claim line from Date falls within the DIAGN record effective period, place the claim line on hold (set the Proc Status to H - Held). Do not include this claim line in further evaluations. Go to step 6 and evaluate the next claim detail line. If no further claim lines exist, go to step 7.

If the claim line from Date does not fall within the DIAGN effective period, evaluate the next matching diagnosis code.

If no further matching diagnosis codes exist, go to go to step 6a and evaluate the next PEC Rule ID on the matching LBPEC record.

If no more PEC Rule IDs remain to be evaluated on the current LBPEC record, go to step 6 and evaluate the next matching LBPEC record.

If no more matching LBPEC records exist, go to step 6a and evaluate the next claim line. If no more claim lines exist, go to step 7.

7. Evaluate the date in the keyword MEMBR PEC End Date box.

If the PEC End Date box is blank, skip to step 10 and evaluate the keyword GRUPD Exclusion Type and Exclusion Period.

If the PEC End Date box contains a date, continue to step 8 and evaluate the keyword GRUPD Apply PEC to PCP check box.

8. Evaluate the GRUPD Apply PEC to PCP check box.

If the check box is selected or the service is not provided by the member's PCP, continue to step 9 and evaluate the first claim line.

If the check box is cleared and the member's PCP or a covering provider is performing the service, then go to step 10 and evaluate the keyword GRUPD Exclusion Type and Exclusion Period.

9. Compare the claim line from Date to the date in the PEC End Date box in keyword MELIG.

If the claim line from Date is before or equal to the PEC End Date, then hold the claim line (set the Proc Status to H - Held) with the reason code from the Reason box in keyword MELIG. Do not include this claim line in further evaluations. Then evaluate the next claim line in the same fashion. If no more claim lines exist, go to step 10.

If the claim line from Date is past the PEC End Date, continue to step 10.

10. Evaluate keyword GRUPD Exclusion Type and Exclusion Period boxes.

If they are blank, stop pre-existing conditions logic for this claim.

If they have values entered, calculate the exclusion period end date. The exclusion period end date is calculated as the member's earliest keyword MELIG effective date for that group or plan plus the number of days or months specified by keyword GRUPD fields Exclusion Type (days or months) and Exclusion Period (number of days or months). Then continue to step 11 and evaluate the first claim line.

11. Compare the claim line from Date to the calculated exclusion period end date.

If the claim line from Date is before or equal to the end date, hold the claim line (set the Proc Status to H - Held) with the reason code from the keyword GRUPD Reason box. Evaluate the next claim line in the same manner. If no more claim lines exist, stop.

If the claim line from Date is past the end date, evaluate the next claim line in the same manner. If no more claim lines exist, stop.

In all cases where a claim line is held, the message "Possible pre-existing condition, line pended" displays.

If pricing only is set up, Pre-Existing Conditions logic is still invoked after pricing.

About Eligibility Records and Pre-Existing Conditions

When a new Member Eligibility History record is entered in keyword MELIG, you can enter member level pre-existing condition overrides. If keyword MELIG data is entered or filled by creating a new record and no pre-existing data in keyword GRUPD exists, then a message "No PEC exists at group level" will display to remind the operator that GRUPD rules must be set up or the MELIG pre-existing data must be removed.

In keyword MEMBR, when you select certain commands on the Special menu, DIAMOND automatically creates a new Member Eligibility record in keyword MELIG. If there is no break in service, then the PEC Waived check box, PEC End Date box, and Reason box will be filled with what was on the previous record. If there is a lapse of coverage, the fields will not be filled on the new record. In either situation, if the new record is generated by a change in group, plan or rider, a message "Verify accuracy of PEC data" will display for the operator.

During claim processing, the exclusion period will be calculated from the effective date of the earliest Member Eligibility record in keyword MELIG for that group or plan combination.
