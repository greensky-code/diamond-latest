Setting Up a Claims Batch

When you first display keyword DNCLM, the Claims Batch Initiation dialog box displays. It allows you to type a default claim batch number and received date to use as long as you remain in keyword DNCLM. Both fields are optional. If you do not enter default values, then during claims entry, the Batch Number box will be blank and the Date Received box will display the current (system) date. If necessary, you can type over the displayed batch number and received date. You also can change the defaults at any time from the Dental services Claims header window using the Change Default Batch command on the Special menu.

Saving a Claim Detail Line

When you save a claim detail line, DIAMOND compares the Adj Method field on the claim detail line to the member’s associated group price/adjudication requirement. If the member’s group pricing is set to adjudicate (in keyword GRUPD, the Group Req list is set to A – Adjud), then you must adjudicate the line before you save it.

For more information, see GRUPD, Group Detail.

How DIAMOND Determines Member Eligibility

DIAMOND uses the service date from the Dental Services Claims Header window to determine if the member is eligible. If a through date is entered, then DIAMOND verifies the member’s eligibility based on both dates. DIAMOND displays a warning if the member is ineligible or if the member’s eligibility expired before the through date.

If DIAMOND determines the claim is eligible, then it performs checks for coordination of benefits (COB), student dependent status, and pre-existing conditions. DIAMOND displays a warning message if it finds exceptions to eligibility.

During adjudication, DIAMOND validates that the date of service falls within the date range of one of the member’s eligibility periods. If it does not, then the line is denied. DIAMOND sets the claim status to Denied, sets the allowed amt and total not covered amounts to $0.00, and assigns the reason code associated with reason code type ME as the Not Cov Rsn code. Reason code type ME (only one is allowed) must exist in keyword REASN.

If the member is ineligible (the claim date of service is outside of the member’s eligibility period), then the claim is denied. If the eligibility status on the member record is invalid because it does not match the valid codes stored in the ELIGSTAT parameter, then the claim is held.

There are additional methods that can automatically hold a claim. For more information, see Claim Holds, How DIAMOND Thinks.

You can manually price, adjudicate, or expense a dental services claim when a member is outside an eligibility period. However, these claims cannot be finalized; because there is no group or plan information on the claim header, DIAMOND cannot validate the Group Req Indicator and, therefore, cannot set the appropriate flag on the Pre_Price_Only column.

Member Eligibility Changes After a Claim Has Been Processed

DIAMOND takes a "snapshot" of the member’s benefit package/plan/rule(s) to be used during the claim adjudication process as soon as the member identification is displayed. If the associated benefit package/plan/rules (BENEF, BRULE) are changed after the benefits snapshot, the benefit changes are not immediately updated on the still open claim header window and DIAMOND uses the benefit snapshot (old benefit data) if the claim is adjudicated during the claim entry session. However, the updated benefit data will display when you exit and then re-open the claim. If the claim was adjudicated using old benefits (snapshot) data, then on the Special menu, point to Pricing and Adjudication, and then click Adjudicate Claim to re-adjudicate the claim to include the updated (changed) benefit data.

Duplicate Claims Check

Each time a claim is entered, DIAMOND automatically checks for duplicate claims based on the duplicate checking criteria set up for the line of business associated with the claim. See LINBS, Claim Duplicate Rule and Claim Duplicate Check Rules (DUPRL) for additional information.

Making Tooth Fields Required

Use the PROCD (Procedure Code) window to make the tooth fields, tooth number, surface, arch, quadrant, and oral cavity, required fields for particular procedure codes in the following windows:

  • DNCLM (Dental Services Claims)

  • PRCDR (Authorization Procedures)

  • THIST (Tooth History)

  • TRULE (Tooth Rules)

Top

Tooth Rules

You use tooth rules to restrict data entry for tooth fields; for example, you can create a tooth rule to restrict the member’s age for particular procedures or to specify that only certain surfaces are valid for a procedure performed on a certain tooth. Each tooth rule applies either to a specific tooth number, to a specific procedure code, or to a combination of tooth number and procedure code.

Tooth rules restrict data entry in the following keywords:

  • DNCLM (Dental Services Claims)

  • PRCDR (Authorization Procedures)

  • THIST (Tooth History) for manual entry of tooth history records

Each tooth rule includes an effective date and a termination date. You must supply an effective date. The effective date determines the date when this tooth rule becomes active. You do not need to supply a termination date; however, you can do so if you no longer want DIAMOND to evaluate this tooth rule.

Within each tooth rule, you can set criteria such as an acceptable age range or acceptable values for each of the tooth fields. A tooth rule will only be applied to a claim if the claim matches all of the criteria defined in the tooth rule.

Rule Criteria

For each rule, you can set the following criteria:

  • From Age

  • To Age

  • Surfaces

  • Arch

  • Quadrant

  • Oral Cavity

From Age and To Age

Use the From Age and To Age fields to set the acceptable age range for this tooth rule. If the member’s age does not fall within the acceptable range for the procedure code specified, DIAMOND will not allow the claim entry to continue. The claims entry operator must then supply an acceptable age for the member, or change the procedure code in order to complete claim entry.

Surfaces, Arch, Quadrant, and Oral Cavity

Each of these fields on TRULE displays information (All, Some, or None) from its associated command window located on the Special menu. In each of these command windows you can specify that you want a rule to match on all of the codes, or you can specify particular codes.

Tooth History

Use the THIST (Tooth History) window to store and maintain a history of all dental procedures performed on members.

Each time you enter a new dental services claim, whether you enter it online or through the DDS API, DIAMOND automatically adds a corresponding tooth history record. You can also manually enter tooth history records into THIST. The Source Type field on the THIST window indicates if a tooth history record was automatically created or if it was manually entered. You cannot edit or delete automatically created tooth history records; however, you can edit or delete manually entered tooth history records.

Whether or not the tooth fields on the THIST screen are required is determined by the procedure code entered for the claim line and how that procedure code is set up in PROCD. See Making Tooth Fields Required.

You can use information stored in tooth history to determine if a newly submitted claim is clinically appropriate. To do this, you set up claim evaluation rules in keyword CERUL. For more information, see Claim Evaluation Rules.

Top

Claim Evaluation Rules

Use CERUL (Claim Evaluation Rules) to build rules that test the clinical appropriateness of claim detail lines. In the case of dental claims, you can build rules that will check against tooth history; for example, if a claim is submitted for a sealant procedure, DIAMOND can check in THIST (Tooth History) to ensure that the tooth has not been extracted in a previous claim. If DIAMOND determines, based on a claim evaluation rule, that the claim line is clinically inappropriate, it can provide a warning or it can apply a hold or deny status to the line depending on the action defined in the evaluation rule. If a hold or deny status is applied to the claim, the associated reason code (if one is specified in CERUL) is also applied to the claim.

The Message to Operator field on CERUL allows you to define a message of up to 60 alphanumeric characters that will be displayed to the claim entry operator if that rule is selected as a match for the claim.

For each claim evaluation rule, you define determinants that are used when DIAMOND is evaluating the rules against new claims, you can also define history matching criteria. History matching criteria determines whether or not DIAMOND searches through tooth history and if so, the timeframe within which it will search.

You can make a claim evaluation rule inactive by clearing the Active checkbox on CERUL.

Rule Order ID

Rule Order ID is how DIAMOND determines the order in which it evaluates the rules. Claim evaluation rules are evaluated in order from lowest number to highest number. DIAMOND stops searching as soon as it finds a rule that matches the information on the claim line; therefore, the more specific rules should have lower rule order numbers, while the more general rules should have higher rule order numbers.

Search Sequences

Each rule must have at least one search sequence and each search sequence must have a determinant table and a column selected. Search sequences may have multiple determinants defined. Search sequences are evaluated in numeric order, from the lowest number to the highest.

If you define multiple search sequences for a claim evaluation rule, only claims that meet the criteria of all search sequences will match the rule.

History Matching Criteria

You can specify primary, secondary, and additional filters to use as matching criteria. DIAMOND compares the matching criteria you specify against any records it finds in tooth history to see if it can find a match.

Primary matching criteria:

  • Tooth Number

  • Quadrant, regardless of tooth

  • Arch, regardless of tooth

  • Blank, no primary match selection applies

Secondary matching criteria:

  • Provider (rendering provider on the claim header)

  • Procedure Code

  • Benefit Package

Additional matching criteria:

  • Medical Definitions

Secondary Matching Criteria

If you want the secondary matching criteria to be an exact match, then place a check in the appropriate secondary match box on CERUL. If you want to specify that all values are acceptable or if you want to define individual and or ranges of values for any of the secondary matching criteria, you must use the appropriate command window available from the Special menu on CERUL. Each of these command windows allows you to select a check box that indicates that all values are acceptable. They each also include options for you to include or exclude single values or ranges of values.

History Search Timeframe

You can specify any one of the following choices as a timeframe:

  • Calendar year

  • Contract period

  • Elapsed year

  • Elapsed days

  • Lifetime

No matter which timeframe option you select, DIAMOND uses the claim detail line’s From Date Of Service as the start date and searches backward through the history records.

X-Ray Points

PROCD (Procedure Code) includes the field Weighted Val (weighted value) that you can use to assign points to x-ray procedures. The weighted value option makes it possible for you to place different values on different types of x-ray procedures and limit the x-ray benefit based on a maximum of points allowed in a specified time frame. The Weighted Val field from PROCD works in conjunction with Benefit Rule Type 20 (Limits). Use BRULE (Benefit Rules) to set up limits benefit rules. Here you can specify the limit type to be weighted values, set the total allowed value, and set the timeframe within which the value is tracked. You can view a member’s x-ray point information including the total number of x-ray points allowed, the number of x-ray points used to date, and the number of x-ray points remaining, in keyword DSPBN (Display Benefit Rules).

Choices for Assessing Dental Limits Benefit Rules

You have the option of assessing dental benefit limits by member or assessing them by the combination of member and provider. This option is set using the Per Provider check box located on keyword BRULE (Benefit Rules). If you select to assess limits by the combination of member and provider, DIAMOND searches claim history each time you enter a new claim to see if the limit for this procedure has already been reached for this provider.

Pricing Claims

Each claim detail line may be priced (allowed amount calculation only) or fully adjudicated. The Group Req list in keyword GRUPD (Group Detail) controls whether claims for a group are priced or fully adjudicated.

Pricing of claim detail lines may be based on fee schedules, percentages of billed charges, conversion factors, per-diems, diagnostic-related groups, or a combination of methods. Claim pricing may be driven by the group contract, the provider contract, specific procedure code, or a combination of these factors.

You can also manually price a claim detail line by overriding the calculated Allowed Amount for either a new claim detail line or any existing unfinalized detail line. For more information, see Manually Pricing a Claim Detail Line.

Covering Providers

A covering provider is, usually, a physician who provides services in the absence of a PCP to that PCP’s assigned members. When a covering provider renders services in this role, he or she may be paid according to the PCP’s contract rather than his or her own, or may not, depending on the rules of the health plan, the nature of the contracts, and health plan practices.

DIAMOND provides a mechanism for recognizing potential covering provider situations and allowing the covering provider to be paid according to a PCP’s provider contract PROVC record. You can set up covering providers with a default action, use the PCP’s contract or use their own contract, and this default action can also be overridden manually during claims processing.

When you change the default action using the Covering Provider Information window:

  • DIAMOND performs a provider contract search based on the data entered in the window.

  • DIAMOND compares the provider contract data currently existing on the claim against the provider contract data derived as a result of the new provider contract search.

  • DIAMOND takes no action if the contract data in #2 above is the same and DIAMOND displays a "Change successful" message.

  • If the contract data in #2 above is different, DIAMOND updates the claim contract data and displays a "Change successful. Go to detail window and readjudicate the entire claim" message.

In this last case, in order to employ the change in pricing logic you must use the Special menu command Adjudicate Claim.

Pricing Anesthesia Claims

When a Procedure code with a code class of "A" (on the PROCD record) or if the Anesthesia check box is selected on the Modifier record in keyword MODIF, then the Anesthesia Time window will display automatically. The Anesthesia Time window may also be selected manually.

To enter time for anesthesia, you may enter either the start and end time or the total elapsed time. If you enter start and end times DIAMOND calculates the elapsed time as long as the time period is less than 24 hours. If greater than 24 hours, you must calculate and enter the total elapsed time manually.

For more information, see Pricing Anesthesia Time, How DIAMOND Thinks.

Allowed Greater Than Billed

When the Line of Business record is set up, one of the parameters is a check box that permits the calculated allowed amount to be greater than the entered billed amount. In the absence of this selection, when dental services claims associated with this line of business are priced, DIAMOND:

  • Prices the claim detail line (determines the allowed amount).

  • Compares the allowed amount to the claim detail line billed amount.

  • Lowers the allowed amount to the billed amount if the allowed amount is greater than the billed amount.

If this option is selected, when dental services claims associated with this line of business are priced, DIAMOND:

  • Prices the claim detail line (determines the allowed amount).

  • Permits the allowed amount to be greater than the billed amount.

  • Does not lower the allowed amount to the billed amount if the allowed amount is greater than the billed amount.

If the allowed amount is greater than the billed amount and the Apply Pat Liability To field on BENEF is equal to "Lsr B/A" or "Lsr B/A/OA" for the applied benefit rule, DIAMOND displays the difference in the Add Back Amt field on the dental claim detail window. For more information, see Apply Pat Liability To in BENEF field definitions.

Only professional services and dental services claims are affected by this check box. Institutional claims pricing logic permits the allowed amounts to be greater than billed amounts. When this occurs, DIAMOND displays the difference in the Add Back Amt field of the dental claim detail window.

See keyword BENEF (Benefit Packages) for information regarding the selection of a benefit package’s claim detail line amount type. DIAMOND uses the selected amount type selected when it calculates patient liabilities.

Adjudication of Claims

Adjudication is the process in which benefit rules, determined by a member’s benefit package, are applied to the claim detail line. This results in the calculation of deductibles, not-covered amounts, co-payments, coinsurance, and exclusions that may reduce the amount paid to the provider. When a claim is adjudicated, each claim detail line is both priced (an allowed amount is calculated) and adjudicated (benefit rules are applied). For additional information, see BENEF (Benefit Package Maintenance), BRULE (Benefit Rule Maintenance), and GRUPD (Group Detail).

As part of the adjudication process, DIAMOND includes Coordination of Benefits (COB) processing when applicable. For more information on COB, see Coordination of Benefits.

Calculating the Net Amount

DIAMOND calculates the claim detail line Net Amount differently depending on whether the line is automatically or manually adjudicated, and whether a COB calculation method applies.

1. When DIAMOND automatically adjudicates a claim line and no COB applies:

Net Amount = Allowed Amt - Total Not Cov - Total Copay - Total Co-ins - Total Deduct - Pt Liability amounts for Benefit Rule type 99 (Other) - Withhold Amt

In some cases, the sum total of the Withhold Amt and the patient liabilities (including not covered amounts) may be greater than the Allowed Amt. In these situations, rather than placing a negative dollar amount in the Net Amount field, DIAMOND sets the Net Amount field to 0 and stores the overage figure in the Withhold Surplus field.

The Total Co-ins amount is the sum of all calculated coinsurance amounts. DIAMOND calculates each coinsurance amount as the benefit rule coinsurance percentage (BRULE Coinsure %) times the claim detail line allowed amount net of any other benefit rules applied before the current coinsurance rule. (You set the processing order for benefit rules in each benefit package.) That is, DIAMOND calculates coinsurance on a decreasing total based on the application of any prior ordered benefit rules.

Example:

Allowed Amount = 200.00

Benefit Rules in processing order:

1. Copay = 10.00

2. Deductible = 50.00

3. Coinsurance1 = 20%

4. Coinsurance2 = 10%

Coinsurance1 and Coinsurance2 amounts are calculated as follows:

200.00 (Allowed Amount) - 10.00 (Copay) = 190.00

190.00 - 50.00 (Deductible) = 140.00

140.00 x 20% (Coinsurance1) = 28.00

140.00 - 28.00 = 112.00

112.00 x 10% (Coinsurance2) = 11.20

112.00 - 11.20 = 100.80

2. When DIAMOND automatically adjudicates a claim line and a COB calculation method applies:

DIAMOND performs the specific calculation logic of that particular COB calculation method. See Coordination of Benefits – How DIAMOND Thinks for details and examples of COB calculation methods.

Note:

For a COB claim that is automatically adjudicated, if the provider on the claim has a contract that includes a provider withhold amount, DIAMOND does not include this amount in COB and Net Amount calculations. DIAMOND does still display the Provider withhold amount on the claim line.

3. When you manually adjudicate a claim line (that is, after automatic adjudication you manually enter or change a total Not Cov, Total Copay, total Co-ins Total Deduct amount or Withhold amount), whether or not a COB calculation method applies:

Net Amount = Allowed Amt or OC Allowed (per COB rules) - Total Not Cov - Total Copay - - Total Co-ins - Total Deduct - OC Paid Amt - Pt Liability amounts for Benefit Rule type 99 (Other) - Withhold Amt

In some cases, the sum total of the Withhold Amt and the patient liabilities (including not covered amounts) may be greater than the Allowed Amt. In these situations, rather than placing a negative dollar amount in the Net Amount field, DIAMOND sets the Net Amount field to 0 and stores the overage figure in the Withhold Surplus field.

Changing or Adjusting Automated Benefit Calculations

Changing the entries in the not covered, copayment, coinsurance, deductible, and reason code fields can change the results of automated adjudication. However, when more than one exclusion (not covered), copay, coinsurance, or deductible benefit rule has been applied to a claim detail line, the corresponding Total Not Cov, Total Copay, Total Co-ins, and/or Total Deduct boxes are grayed out and unavailable for entry, as are the corresponding reason code fields (which are filled with asterisks).

You must use the Patient Liabilities window to view/change multiple underlying Copay, Coinsurance, Limits (Not Covered), Deductible, and/or Exclusions benefit rules/amounts that constitute the total amounts displayed in the Total Not Cov, Total Copay, Total Co-ins, and/or Total Deduct fields on the INCLM, Claim Detail Window. You must also use the Patient Liabilities window to add an "Other Amount."

The Patient Liabilities window has four focus boxes. The first focus box displays claim header member/provider information as displayed on the claim header window. The second focus box is a picklist of claim detail lines. The third focus box is a picklist of benefit rule data for each benefit rule that has been applied to the claim detail line selected in the second focus box. The fourth focus box is a data entry/display area of Rule Type, Amount, Benefit Rule (ID) and Reason Code data for the benefit rule selected in the third focus box.

Only detail records that have not been finalized, but have been adjudicated by DIAMOND, are available to change. To make changes enter a new Amount and if necessary a new Reason code.

If you are changing an automatically determined reason code, be aware that your change will not be validated. In addition, accumulations under the member’s benefit plan might not be affected by the change, since they are added together based on reason codes.

DIAMOND changes the BRULE ID here to a default value, SYSTEM, and the Adj Method second character to "M" (adjudicated manually) when you change or add a member liabilities record using this window.

Adding Patient Liabilities Records

Patient Liabilities added using this function (rather than automatically by DIAMOND) are not added to the member’s accumulated benefits history.

Press CTRL+N to add a member liabilities record in addition to the DIAMOND adjudicated member liabilities records, and the following fields are available for entry.

  • Amount

  • Rule Type: You may use Rule Type 99 (Other) for patient liabilities that do not fit within the other rule type categories. Rule Type 99 is a valid rule type only for the Patient Liabilities window. This rule type cannot be entered in keyword BENEF (Benefit Package Maintenance). Rule Type 99 amounts from these records are not included in the Total Not Cov, Total Copay, Total Co-ins, or Total Deduct fields displayed on the claims detail window; however, these amounts are included in the Net Amt displayed on the claim detail window.

  • Reason Code

Discount and Interest Calculations

Discount or interest calculations can be automatically applied against payable claims and reversed payable claim lines. Interest calculations can take into account any unclean days tracked on a claim. A claim may be considered unclean if it requires additional substantiating documentation from a source outside of your organization and if the missing information prevents you from paying the claim timely.

You can manually apply a discount or interest amount on an adjusted claim line.

For a complete explanation of discount and interest calculations, see Claims Discount and Interest Calculations, How DIAMOND Thinks.

Processing Messages

System or user-defined processing messages may display when you enter, price, or adjudicate claims. They may display on both the Dental services Claims header and detail windows. Some messages display automatically and some only display when you select the Processing Messages command on the Window menu in claims.

On a claim detail line, an "X" displays when one or more header level or detail level messages are present for the current claim detail line.

System messages are primarily defined in keywords MSGMT (Message Maintenance) and PMSCT (Processing Message Control).

User-defined messages may be defined in keywords PRULD (Price Rule Detail), PRULW (Whole Claim Pricing Rules), and BRULE (Benefit Rule).

Providers

Out of Plan Providers and One-Time Providers

When a provider is considered "out of plan," you can use a non-specific provider. A non-specific provider has a "Y" in the Non Specific Prov field on the Provider Master file (PROVF) record. If you use a non-specific provider, you must specify a vendor during claims processing. Use of the non-specific provider and default provider features to handle out of plan or one-time providers requires careful setup. For a complete explanation of this feature and its use, see Provider Contracts, How DIAMOND Thinks.

Multiple Vendor Addresses

In addition to selecting the correct vendor record, DIAMOND prompts the user to select the address to which payment should be sent in cases where the vendor has multiple address records. A list of valid addresses is displayed, and the user picks from this list.

Provider Contract Data

DIAMOND uses provider contract search logic to verify provider contract data. See Line of Business (LINBS) documentation for more information on the provider contract search.

Linking with Accounts Payable

Keyword APUPD (Accounts Payable Update) creates A/P claims records and selects either medical (claims) or capitation records for payment. At the same time the originating records in claims and capitation are finalized. As part of the finalization of the original claims records a transaction number is generated and appropriate general ledger accounts are updated. Once claim detail lines are finalized using APUPD, they cannot be changed; however, finalized claim detail lines can be adjusted or reversed (as can unfinalized lines). See Adjust Line/Reverse Line on DNCLM Commands on the Menu Bar.

Use keyword PROVC (Provider Contracts) to set up the printing of Remittance Advices (RAs) for a specific provider. Use keyword LINBS (Line of Business) to set up the printing of Explanation of Benefits (EOBs) for a specific line of business.

You can access/override the RA and EOB printing defaults for a specific claim using the Payment Detail command on the Special menu. RA and EOB printing status codes are also displayed on accounts payable records (ACPAY, RA/EOB Print Flag).

Integrating GMIS ClaimCheck® Software

A real-time integration between GMIS’s ClaimCheck® software is available within keyword DNCLM. Use ClaimCheck to verify the clinical accuracy of professional claims and automate the process of detecting claim submission inaccuracies by analyzing claims to ensure that the correct procedures are reimbursed.

For more information, see keyword CAUDT (Claim Audit Filter) and CARUL (ClaimCheck Rules & Results).

Deleting Claim Headers and Detail Line Items

DIAMOND allows you to delete both claim detail line items and claim header records. Note that you cannot delete a claim header if it includes any detail line items. To delete a claim header that includes detail line items, first delete the line items.

You cannot allow a user to delete claim detail line items but not claim header records. You can allow a user to delete both claim header and detail line items, or allow the user to delete neither header nor detail records.

For instructions on deleting a claim header or detail line item, see Delete a claim header or detail line item.
