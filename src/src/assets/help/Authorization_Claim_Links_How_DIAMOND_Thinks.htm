Introduction

There are several reasons why claims for medical services must be authorized. A variety of actions result whether or not the authorization was obtained.

  • Some services require prior approval as to medical necessity

  • Some member benefits vary depending on whether authorization was obtained

  • Pricing of services may depend on the prior authorization of the service

  • Some services are not reimbursable at all unless authorized

Authorization Claim Links features

  • Automatic search of the authorization file from claims for potential matching authorizations, and automatic linking when there is a clear match

  • Automatic carryover of field entries from the authorization to the claim record

  • Automatic application of benefits depending on whether a claim is authorized

  • By line of business, different rules for determining what constitutes a matching authorization

  • Depending on whether the provider of service is covering for the authorized provider, different criteria and logic for determining a matching authorization

  • Automatic holding or denying of claims when a matching authorization is not found

  • Automatic rules for waiving the requirement for an authorization depending on user defined characteristics of the claim, such as the provider’s specialty, service, diagnosis, line of business, and so forth

  • Easy reapplication of authorization matching and or waiving of authorization requirements when information on the claim is changed

  • Commands on the Special menu in claims and authorizations to display a list of all institutional, professional services, or dental services claims with the same authorization number

  • Authorization/Claim Detail Linking and Pricing logic, which requires authorization/claim matching

Affected keywords and parameters

Keyword BENEF (Benefit Package) contains member benefits that vary, depending on whether a claim was authorized.

Keyword LINBS (Line of Business) contains the rule that tells DIAMOND whether to automatically match authorizations to claims in keywords DNCLM (Dental Services Claims), INCLM (Institutional Claims), PSCLM (Professional Services Claims), and in EDI claims.

Keyword MCHRL (Auth/Claim Match Rules) contains the rules DIAMOND uses to decide whether an authorization matches a claim.

Keyword PARAM (Parameter) contains the following parameters that affect the authorization claim link.

  • AUTHCLSLINK: Used by Batch Authorization Close, but is related to the authorization claims link. If the Parameter 1 box is set to Y, then authorizations will be selected for batch close if they are linked to a claim. If the Parameter 1 box is set to N, then the criteria of whether or not an authorization is linked to a claim cannot be used to select authorizations for batch close.

  • DUPAUTHRULE2: Parameter 1 identifies whether the authorization Request Date or the authorization Actual Admit Date will be used as the primary date of service for authorization/claim match purposes (and duplicate authorization checking). Parameter 2 identifies whether the authorization Expiration Date or the authorization Discharge Date will be used as the through date of service for authorization/claim match purposes.

  • USEAUTHDATA: If the Parameter 1 box is set to Y and the Auth/Claim Match check box is cleared in keyword LINBS for the line of business, then the following occurs when you type an authorization number in the Auth No. box on the claim header:

  • If you are submitting a professional or dental services claim, then the data in the Member ID, Prov Code (referral provider and referring provider), Place Of Service, DX 1, Dx 2, and DX 3 fields on the authorization are moved to the claim header.

  • If you are submitting an institutional claim, then the data in the Member ID, Prov Code (admitting provider, attending provider, and facility/institution), Place Of Service, Admit Reason (mapped to the Serv Rsn box), DX 1, DX 2 and DX 3 fields on the authorization are moved to the claim header.

If the Parameter 1 box is set to N or is blank, then DIAMOND will not fill the claim header fields with authorization data.

  • USEAUTHLINK: If the Parameter 1 box is set to Y, then Diamond will use ARULE to define actions on a claim that has an authorization number on the header. If the Parameter 1 box is set to N or blank, then Diamond will not use ARULE.

Keyword AWAVE (Auth Waive Rules) contains rules that negate, according to characteristics of the claim, the need for an authorization that would otherwise be required.

Keyword ARULE (Authorization Claim Link Rule) assesses the administrative validity of the authorization that has been matched to a claim, and takes action on the claim depending on that validity. Validity characteristics of an authorization include whether it has expired, is held, closed, and other status data. Only used if USEAUTHLINK (in the Parameter file PARAM) is set to Y (yes).

Keywords DNCLM, INCLM, and PSCLM each contain commands on the menu bar that allow re-application of authorization matching or authorization waivers when claim data is changed.

Setting up

A match between a claim being processed and a previously entered authorization can be determined manually or automatically. The LINBS record determines whether DIAMOND will attempt to automatically match authorizations to claims (during the initial save of the claim header only) by enabling this feature when the Auth/Claim Match check box is selected.

Regardless of whether the match is automatic or manual (see "Mechanics of manually matching an authorization to a claim" and "Mechanics of automatic authorization/claim matching" sections below), DIAMOND enforces matching on certain fields and conditions according to the rules you enter in keyword MCHRL. MCHRL is used to identify unique or multiple potential matching authorizations during claims processing automatically (according to the LINBS entry) and also to make sure that an authorization number manually entered on a claim is a matching authorization. This prevents a claims processing user from linking an unrelated authorization to a claim.

DIAMOND will take automatic actions on the claim depending on different aspects of the authorization according to the rules you enter in two different keywords.

  • MCHRL uses medically-related criteria on the authorization and claim to identify whether the authorization record should be used to ‘approve’ the claim, i.e. whether the services being claimed were authorized. These criteria are:

  • Line of Business

  • Claim Type

  • Authorization Status

  • Authorization Level

  • Provider ID (excluding covering providers)

  • Authorization Begin Date and Days Before

  • Authorization End Date and Days After

  • Claim Primary DOS

  • Claim Through DOS

  • Facility ID number (professional services or dental services claims only)

  • Surgical Procedure code (institutional claims only)

  • Place of Service (exact match or within specified range)

  • DIAMOND ID

  • Provider’s Primary Specialty

  • Provider’s Vendor ID

  • Diagnosis Code (full match or specified number of first digits)

In some cases, certain actions are taken on the claim based on these medically-related criteria, and those claim actions are defined as part of the MCHRL rules.

For example, a heart transplant requires authorization like any other major surgical procedure. In addition, though, the heart transplant surgery claim is held for review by the Medical department. The hold code and hold reason are included in the MCHRL rule defining a heart transplant. When a heart transplant claim is entered and matched to the heart transplant authorization, the hold code and hold reason from the MCHRL are automatically entered on the claim.

  • ARULE contains administrative validity criteria that are applied to the already-matched authorization. These criteria include whether or not the authorization has expired prior to the dates of service on the claim, whether the authorization has already been closed or ‘used up’ by being linked to previous claims, whether the dates of service are outside the authorized range of dates, and the like. DIAMOND will take automatic action on the claim, as defined in ARULE, depending on these authorization characteristics. This may be in addition to actions taken according to the MCHRL.

Using the ‘date range’ match criteria

MCHRL vs. ARULE and defining the date range

While there are differences in attributes and functions between MCHRL and ARULE, both MCHRL and ARULE can be used to define date ranges on the authorization within which the claim dates of service must fall. Technically either or both can be used for this purpose. However, in order to minimize claims processing and customer and provider service issues, Perot Systems recommends that one or the other be selected during implementation and that administrative procedures be defined to enforce this use.

Date ranges for covering providers

MCHRL does not apply to situations where a covering provider is rendering the service. In that case, instead of MCHRL, DIAMOND uses hard-coded match logic, see "How DIAMOND decides if an authorization matches a claim, PSCLM/DNCLM covering providers". When a covering provider renders the service, the only place you can define date range match criteria is in ARULE.

Which dates are used to calculate the date range?

For all claim types, the Days After and Days Before entries in LINBS are added to and subtracted from the primary date of service on the claim header (PSCLM Svc Date; INCLM Admit Date; DNCLM Svc Date). If the request date on the authorization is within the calculated range, it is a potentially valid authorization for the subject claim and is eligible for matching.

Determining whether to look for a match

There is a single trigger in DIAMOND that launches the process (or processes) to attempt to find an authorization record which matches the claim being processed. That single trigger is the record in LINBS associated with the member’s eligibility. If the Auth/Claim Match check box is selected, DIAMOND will attempt to find a matching authorization. If it is not selected, no such attempt will be made, although a match can be made manually by the claims processor.

This automatic trigger is invoked regardless of whether any actions would result if the claim were or were not authorized.

DIAMOND automatically executes table-driven authorization/claim matching only during the initial save of the claim header. After that, if a change is made to the claim header, you must use the Special menu command Authorization Match/Waive, options Reapply Match or Reapply Match_Waive to again execute table-driven authorization/claim matching logic using the revised data.

Matching authorizations to claims

There are two functions in DIAMOND that contain the rules used to determine if any authorizations are matched, or potentially matched, to a claim being processed. In both cases, the LINBS record ‘turns on’ the automatic matching function.

PSCLM/DNCLM covering providers

For PSCLM and DNCLM claims submitted by providers who are covering for the originally authorized provider, DIAMOND has a set of hard-coded rules that are used to evaluate for a match. This set of rules is invoked so long as LINBS has checks in both Auth/Claim Match and Covering Prov. These hard-coded rules replace the rules set up in MCHRL for other claim situations.

Warning!   VENDORFIRST parameter logic cannot be used if you select the Covering Prov check box (the VENDORFIRST Parameter 1 must be "blank.")

In order to be considered a potential match all of the following conditions must apply.

  • The member number and person number on the claim header are the same as the member number and person number on the Authorization Master (AUTHS) record.

  • The rendering Provider ID on the claim header must be part of the same covering providers group as the Referral Provider ID (a Provider with a Provider Code of "Referral Provider) on the Auth Provider record. If other conditions apply but the Provider ID field on the Authorization is blank (it’s not a required field), there will not be a unique match.

  • The primary service date on the claim header must be within a date range bounded by the requested date on the AUTHS record minus/plus the number of days specified on the member’s associated LINBS > Days Before/Days After record.

A unique match occurs when only one authorization is found that matches all three above criteria. When a unique match occurs, Diamond populates the PSCLM/DNCLM Auth No. field with the matched authorization number.

All other claims

For all other claims, DIAMOND uses the implementation-specific rules set up in MCHRL. MCHRL offers a variety of claim and authorization criteria to determine whether a unique or potential match is available. The claim and authorization criteria that can be used include line of business, claim type, provider ID, various dates of service, place of service, and surgical procedure codes.

Mechanics of manually matching an authorization to a claim (LINBS Auth/Claim Match box not selected)

If you enter or select a valid authorization number on the PSCLM, INCLM, or DNCLM claim header window, Diamond uses the entered authorization number.

If you do not enter or select a valid authorization number (Auth No.) on the PSCLM, INCLM, or DNCLM claim header window, when you enter the Member ID Diamond displays a dialogue box asking if you want to look up all authorizations for the member. If your response is Yes, Diamond displays all authorizations for the member. You may select a valid authorization or return to claims entry. If you respond No, Diamond allows you to continue with claim entry.

Mechanics of automatic authorization/claim matching (LINBS Auth/Claim Match box is selected)

No authorization/secondary authorization number (Auth No.) is entered on the PSCLM, INCLM, or DNCLM header:

When entry of claim header data is complete and you save the header for the first time, Diamond retrieves the authorization/claim matching rules corresponding to the claim’s line of business, file type, authorization status and authorization level and applies the rules in sequential order, comparing selected claim header data to data on the authorizations. (This is only done the first time you save the header.) At the conclusion of this process, Diamond does one of the following:

  • If there is one matching authorization and the Apply Auth to Claim check box is selected on the match rule, Diamond enters the matching authorization number in the Auth No. field on the PSCLM, INCLM, or DNCLM header.

  • If the Action entry field on the authorization/claim match rule which identified the match specifies ‘Hold’, Diamond changes the Process Status for all existing non-finalized claim detail lines to Held and assigns the Reason Code from the match rule to each line. Diamond will also assign a Process Status of Held (plus Reason Code) to any detail lines added subsequently for as long as this auth/claim match applies.

  • If the Action entry field on the match rule which identified the match specifies ‘Deny’, Diamond changes the Claim Status for all existing non-finalized claim detail lines to Deny and assigns the associated Reason Code to each line. Diamond will also assign a Claim Status of Deny (plus Reason Code) to any detail lines added subsequently for as long as this auth/claim match applies.

  • If there is one matching authorization but the Apply Auth to Claim check box is not selected on the match rule, Diamond does not populate the Auth No. field on the PSCLM, INCLM, or DNCLM claim header window with the matching authorization number. Instead, DIAMOND displays the message "The Authorization is not applied to the claim". DIAMOND does, however, still apply the MCHRL Action to the claim as described below.

  • If there are multiple matching authorizations, Diamond displays a picklist of matching authorizations on the Select Matched Authorizations window (described in a separate section below) and allows you to select one. Diamond then populates the Auth No. field with the authorization number of the selected authorization.

  • If the Action entry field on the authorization/claim match rule which identified the selected match indicates ‘Hold’ (or ‘Deny’), Diamond changes the Process Status for all existing non-finalized claim detail lines to Held (or Denied) and assigns the associated Reason Code to each line. Diamond will also assign the same Process Status and Reason Code to any detail lines added subsequently for as long as this auth/claim match applies.

  • If there were no matching authorizations, Diamond displays a message to that effect and continues the PSCLM, INCLM, or DNCLM claims entry process leaving the Auth No. field blank.

A valid authorization/secondary authorization (Auth No.) is entered or selected on the PSCLM, INCLM, or DNCLM header:

When entry of claim header data is complete and you save the header for the first time, Diamond verifies that the authorization/secondary authorization number entered satisfies the authorization/claim matching rules. (This is only done the first time you save the header.) First, Diamond retrieves the authorization/claim matching rules corresponding to the claim’s line of business, file type, authorization status and authorization level. Diamond then applies the rules in sequential order, comparing selected claim header data to data on the authorizations. At the conclusion of this process, Diamond does one of the following:

  • If the entered authorization satisfies the match criteria for a retrieved rule and no other matching authorizations are found for the same rule (confirmed unique match), Diamond retains the authorization number entered on the claims window and claims entry continues. (Once the authorization satisfies the criteria from one retrieved rule, Diamond does not continue to evaluate other rules.)

  • If the entered authorization satisfies the match criteria for a retrieved rule but other matching authorizations are found for the same rule (identified multiple matches), Diamond displays a picklist of matching authorizations on the Select Matched Authorizations window (described below) and allows you to select one. (Once one retrieved rule results in multiple matches, Diamond does not continue to evaluate other rules.) If you select an authorization, Diamond populates the Auth No. field with your selection. If you do not make a selection, Diamond allows you to continue with claim entry.

  • If the entered authorization does not satisfy the match criteria for any retrieved rule (failed match), Diamond does the following:

  • Stores the authorization number entered in the PSCLM/INCLM/DNCLM Submitted Auth No. field

  • Clears the Auth No. entered from the PSCLM/INCLM/DNCLM header window

  • Performs authorization/claim matching as though no authorization was originally entered.

You can view the authorization number of the authorization you originally entered and the Diamond matched authorization using the PSCLM/INCLM/DNCLM Special menu > Submitted Authorization.

Waiving authorization requirements

Keyword AWAVE (Auth Waive Rules) allows you to automatically waive the need for claim authorizations under particular circumstances. For example, a health care plan may waive the need for an authorization for outpatient procedures so long as they are performed at a participating hospital by a participating attending physician. When the claim is saved DIAMOND applies authorization waiver rules to claims for members whose line of business so requires according to the LINBS record Perform Auth Waive check box.

You can define AWAVE rules using as many criteria on the claim header as you need, with effective dates for each rule and an order in which the rules are applied. DIAMOND uses fall through logic, similar to its use in MEDEF.

When the claim header data satisfies all of the determinant values on an AWAVE rule, the need for an authorization is waived and the Method on the claim is set to A.

When the claim header does not satisfy all of the determinant values on an AWAVE rule, any logic based on the existence of an authorization is applied depending on whether or not an authorization number is attached to the claim, and the Method remains unchanged, either M or blank.

AWAVE is integrated with Electronic Data Interchange (EDI).

DIAMOND allows you to maintain an audit trail of authorization waive rule adds, updates and deletes. You may view this audit trail information using the AWAVE Window menu command Audit Display. For more information, see the process overview Managing the Authorizations Audit Trail.

Effect of waiving authorization requirements

Certain benefits vary depending on whether a claim is authorized. In the absence of authorization, a lesser amount may be reimbursed or the service not covered at all. However, when an AWAVE rule is applied to that claim, the benefit reduction is not calculated and the claim is reimbursed as if it were authorized.

Effective dates of rules and retroactive application

Keyword AWAVE allows you to enter new waiver conditions and rules with retroactive effective dates. Theoretically, such rules should apply to claims that have already been processed. However, DIAMOND does not automatically retroactively apply AWAVE rules. In order to apply such newly-entered rules to older claims, retrieve each claim online and apply the Special menu command Authorization Match/Waive > options Reapply Waive or Reapply Match_Waive.

Using the Authorization Class to Waive Authorizations

In some cases, the requirement that a claim be authorized may be waived when the attending or referring provider is participating. (This is in contrast to other authorization waivers that depend on the provider of service.) Because claims logic does not look at the contract information for the attending or referring provider, authorization waivers cannot use participation status or other contractual indicators. Therefore, in order to be able to waive the authorization requirement for attending or referring providers, you must mark a provider as "participating" in another way. You do this by first entering a valid value in the Auth Class field on the corresponding Provider Master record. Then, in AWAVE, you enter a rule using Auth Class as a determinant, with that same valid value as the determinant value.

Determining the order in which rules are evaluated

By default, claim processing first evaluates the auth/claim match rules in keyword MCHRL. Then, claim processing evaluates the auth waive rules in keyword AWAVE. However, you can reverse the order in which claim processing evaluates the rules by setting the Waive Match Order field in keyword LINBS.

Provider Data Carryover

As each provider is entered on an AUTHS record, he or she is associated with a code that indicates the role the provider is authorized for, such as admitting physician, referring provider, institution, and the like. When the authorization is linked to a claim, this code tells DIAMOND which provider ID to put in each provider field on that claim. Admitting, attending, and institutional providers are linked to the same fields on an Institutional claim (INCLM). Referring and facility providers are linked to the same fields on the professional services (PSCLM) or dental (DNCLM) claim; Referral provider on the AUTHS record is linked to provider on the claim.
