Examples of PDTER Evaluation

Some institutional contracts have different reimbursement arrangements for different types of services. For example, a health plan may reimburse a hospital on a per diem basis for medical/surgical inpatient stays and on a per case basis for inpatient detoxification stays. This kind of pricing is called Whole Claim Pricing in DIAMOND.

To whole claim price, you create a Whole Claim Price Rule (PRULW) associated with the institution’s/facility’s price rule (PRULE). Within the PRULW, you create additional PRULW records, one record for each unique reimbursement arrangement, and assign a price determinant code (PDTER) to each so that DIAMOND can distinguish which rule applies to which type of claim or claim line.

Price Determinants (PDTER) defines each price determinant code according to the claim’s data elements and their values that are included or excluded. This tells DIAMOND when a whole claim price rule record should be applied to a particular institutional claim.

Example

Within the same whole claim price rule (PRULW), you want to create one whole claim price rule with a price determinant code MEDS to define a medical/surgical inpatient admission and whole claim price rule with a price determinant code DETOX to define a detoxification stay. You use PDTER to define which kinds of claims are DETOX and which are MEDS by selecting the claim data elements and the included or excluded values that define each. A medical/surgical admission could be defined as a combination of inpatient Service Place (01) and a Service Reason of INMED and a detoxification stay could be defined as a combination of inpatient Service Place (01) and Service Reason INTOX.

To construct price determinants and whole claim price rules for you would:

1. Use PRULE to enter a price rule for the institution/facility.

2. Assign a unique price determinant code for each reimbursement arrangement to be defined in the whole claim price rule (PRULW) for the institution/facility (MEDS and DETOX in our example).

3. Use PDTER to further define the MEDS and DETOX price determinant codes.

4. Use PRULW to enter one whole claim price rule record with a PRULW > Price Determinant of MEDS and a PRULW > Calc Method of Per Diem and a PRULW > Sys Proc Code of SMEDS (the procedure code that will appear on the system generated whole claim price line).

5. Use PRULW to enter another whole claim price rule record with a PRULW > Price Determinant of DETOX and a PRULW > Calc Method of Case Rate and a PRULW > Sys Proc Code of SITOX (the procedure code that will appear on the system generated whole claim price line).

6. Use PROCP to create a procedure code pricing record for each of the PRULW > Sys Proc Code (one pricing record for SMEDS and one pricing record for SITOX).

Processing/Calculation Logic

Price Rule Determinants and Medical Definitions

Price rule determinants are similar to medical definitions (MEDEF); however, there are some key differences:

Price Rule Determinants

Medical Definitions

Multiple price rule determinants can be assigned to a whole claim.

Only one medical definition is assigned to each claim detail line detail.

When determining whether or not a price rule determinant applies to a claim, information from the claim header and all the non-system generated claim detail lines are validated.

When determining a medical definition, information from the claim header and only the current claim detail line are validated.

The price rule determinant is stored in the pricing decision for the claim.

The medical definition is stored on the claim detail line.

Price rule determinants have no inherent priority.

You must define a validation priority to medical definitions.

Price Determinant Evaluation

During whole claim pricing, Diamond sorts the PDTER include/exclude values, compares them against the corresponding claim header and claim detail line values, and evaluates the comparisons as outlined below.

1. Sort the PDTER claim header includes/excludes alphabetically, by column name.

2. Sort the PDTER claim detail includes/excludes alphabetically, by column name.

3. Using and/or operators, compare the sorted PDTER claim header include/exclude value(s) to the corresponding claim header value(s).

4. PDTER include/exclude entries that have the same column name are connected by an "or" operator. PDTER include/exclude entries that have different column names are connected by an "and" operator. For example, within a PDTER record:

2. Assign an evaluation status to each header comparison.

3. If any header exclude comparison results in an evaluation status of "Fail," exit the header comparison process and do not apply the associated whole claim price rule (PRULW) to the claim.

4. If all header exclude comparisons complete without a Fail, and there are no PDTER header include comparisons, and there are no PDTER claim detail includes/excludes to be compared, apply the associated whole claim price rule (PRULW) to the claim.

5. If all header exclude comparisons complete without a Fail, and at least one header include comparison results in an evaluation status of "Pass" (across one continuous or/and include comparison), and there are PDTER claim detail includes/excludes to be compared, begin the detail comparison process (step #9).

6. Using and/or operators, compare the sorted PDTER claim detail include/exclude value(s) to the corresponding claim detail line value(s). PDTER include/exclude entries within the same column are connected by an "or" operator. PDTER include/exclude entries that have different column names are connected by an "and" operator. See the example in step #3.

7. Assign an evaluation status to each detail comparison.

8. If any detail exclude comparison results in a Fail, exit the detail comparison process and do not apply the associated whole claim price rule (PRULW) to the claim.

9. If all detail exclude comparisons complete without a Fail, and there are no PDTER claim detail include comparisons, apply the associated whole claim price rule (PRULW) to the claim.

10. If all detail exclude comparisons complete without a Fail, and at least one detail include comparison results in a Pass (across one continuous or/and include comparison), apply the associated whole claim price rule (PRULW) to the claim.

11. If all detail include/exclude comparisons complete without a Fail, and none of the detail include comparisons result in a Pass (across one continuous or/and include comparison), exit the detail comparison process and do not apply the associated whole claim price rule (PRULW) to the claim.
