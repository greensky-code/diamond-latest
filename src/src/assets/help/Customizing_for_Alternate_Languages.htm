Customizing for Alternate Languages

Restrictions

Instructions

Frequently Asked Questions

You can change the appearance of DIAMOND’s menus, screens, and messages. For example, you may have your own local or health plan specific terms which you would like to use in place of DIAMOND’s standard descriptions (such as, "employees" instead of "subscribers"). Or, you want to use languages in addition to or instead of American English.

You can customize the appearance of the DIAMOND user interface with one or more alternate languages, and associate those alternate language versions to individual users.

Restrictions

DIAMOND is delivered with keyword descriptions in American English, with a language code of AMERI. This set of descriptions cannot be changed, although it need not be used in your production system. Follow the instructions below to set up an alternate language in which you can make changes to American English text delivered with DIAMOND.

One of the steps to setting up an alternate language, keyword SCNMT (Screen Maintenance), is fairly time consuming when first set up in the new language because it defines every hard-coded field on every screen. If you plan to use multiple alternate languages, then talk with your DIAMOND Account Representative about employing a back end tool to update keyword SCNMT.

Instructions

Step

Comment

1. In keyword LANGM (Language), add a Language code.

If you will be using multiple languages, then you may want to devise a structure for the language codes themselves.

  • The Session Language Code list identifies the Oracle session language associated with the Language code.

  • The Description box is used in drop-down windows where Language codes are selected, such as keyword KWDMT (Keyword Maintenance).

2. Define screen literals in the new language for each of the following keywords.

  1. KWDMT

  2. LOVMT (List of Values Maintenance)

  3. MNUMT (Menu Maintenance)

Each of these works in the same way.

  • Press F5 to lookup and select the base language entries under that keyword in the left hand box. Note: Base language/default language delivered with DIAMOND is 0-American English. When the default language is changed, that new default language becomes the new base language.

  • Use the Language command on the Special menu to select the language in which you are establishing the new entries.

  • A right hand box will appear with all of the entries from the default

  • Override the default entries with new text in the selected (alternate) language.

  • Save the record (on the File menu, click Save).

3. Establish new screens and microhelp in the new language using keyword SCNMT. Select SCNMT, choose a keyword (from Search), and then choose the new language code. DIAMOND will then create a table entry for every screen for that new language.

Although keyword SCNMT works the same way as the functions explained in Step 2 above, it will take several minutes or more for DIAMOND to create all the table entries for all the screens. (It creates all of them when it does not find an entry for the selected keyword and selected language.) Once these are all created automatically, you may bring up each keyword in which you require new language entries and make changes as appropriate. This allows you to change or translate only those DIAMOND screens that you use. For example, if you do not use Premium Billing, then you do not need to translate any of the premium billing related screens or fields. However, keyword SCNMT will have created versions that are attached to the new language code so if a user attempts to look at those screens they are available, albeit not translated.

4. Define entries for user messages in MSMGT (Message Maintenance) and user-defined field literals in UDTXT (User Defined Fields), selecting the new language as the alternate language in each.

 

5. If applicable, use keyword UDATT (User Defined Attributes) to enter a new phone number mask and other address attributes, and activate the Country field.

The Country field is a hidden field in the base language when DIAMOND is first delivered. You may choose to require Country as part of the addresses in DIAMOND if you are working in more than one Country.

7. In keyword CODES (System Codes Maintenance), enter descriptions of user-defined codes.

 

8. In keyword LANGM, identify which language is the default.

DIAMOND is delivered with a base language AMERI, representing American English. AMERI is initially designated as the default.

The language designated as the default is automatically assigned to existing users in SUSER.

Warning: DIAMOND will allow you to change the default language to an incompletely defined language. When automatically assigned to existing users in SUSER, this will prevent those users from accessing the system. Do not change to a default until it is completely defined. See FAQs below for more information.

9. Select the SUSER record for each user who requires a language other than the default (newly assigned in Step 6, or the base AMERI) and enter the appropriate Language Code.

 

Frequently Asked Questions

  • Can different users employ different languages?

Yes, you may set up as many languages as you like and associate them to users as you require.

  • Do I have to update everything, or can I just change menus or add the Country field?

You may change as little or as much as you require. Because you cannot change any of the text delivered with DIAMOND under language code AMERI, you must set up a new language code under which any changes are entered and used, such as a change from ‘subscriber’ to ‘insured.’ Because DIAMOND copies all of the AMERI entries into the new language code, where you make the changes, you can change a s little as a single module name, field, or microhelp phrase if you like.

See the question below for another consideration.

  • What is an incompletely defined language?

An incompletely defined language is one where the alternate language values have not been established for each of the following keywords:

  • KWDMT

  • LOVMT

  • MNUMT

  • SCNMT

  • MSGMT

  • UDTXT

Recommended, but not required for a language code to be completely defined, are

  • NAVMT (Navigation Maintenance)

  • CODES

  • UDATT

The Language field will not accept a language code that is incompletely defined, even though the code is shown in the drop-down list.

This must be taken into consideration if you are planning on changing just a few fields or menu items. In order for the newly defined language code to be available to assign to users, it must have entries in all of the required keywords listed above. Now, since DIAMOND copies the base entries over to the newly defined language, allowing you to accept them without changes, this is not that big of a task. However, keep in mind that to change one menu item name, you will have to create new table entries in that new language for keywords, valid values, menu commands, module names, and hard-coded field names. Consider carefully how important small changes are in terms of efficiency when weighed against the system maintenance they require.

  • I can see an alternate language available in the drop-down list for the language field in SUSER, but when I select this language the field reverts to the base language that was already shown in that field. What’s wrong?

The language you are attempting to assign to a user is incompletely defined. When you have completed all the required keyword entries (see previous question for list) DIAMOND will recognize the assignment of the language to the user. In the meantime, DIAMOND will revert the field to the base language.
