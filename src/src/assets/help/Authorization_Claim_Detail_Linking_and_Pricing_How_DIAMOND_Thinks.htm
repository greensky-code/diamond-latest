Introduction

You can use authorization/claim detail linking and pricing to perform the following actions, based on rules you define by line of business (LOB) /authorization type/effective date combination:

  • Match submitted claim detail procedure/revenue codes to ranges of authorized codes on a matching authorization’s keyword PRCDR (Authorization Procedures) authorization procedure detail records

  • Price matching claim detail lines using:

  • The Auth Price from the matching authorization procedure detail record

  • The Auth Procedure from the matching authorization procedure detail record

  • Standard DIAMOND provider pricing logic

  • Hold or deny matching claim detail lines based on the application of an authorization/claim detail linking and pricing rule

  • Assign specific allowed, hold, and deny reason codes to matching claim detail lines when applying an authorization/claim detail linking and pricing action

DIAMOND performs authorization/claim detail linking and pricing for online and DIAMOND Data Service (DDS) API transactions.

Processing Logic

Authorization/claim detail linking and pricing processing logic includes:

  • Finding and Using a Matching Authorization

  • Handling Multiple Matching Authorizations

  • Including Institutional Claims Procedure Code Records

  • Performing Authorization/Claim Detail Linking and Pricing

  • Authorization Overall Status Override

  • Special Menu Command Pricing and Adjudication

Finding and Using a Matching Authorization

In order to perform authorization/claim detail linking and pricing for a claim, you need an authorization that matches the claim. DIAMOND can find a matching authorization during automatic authorization/claim matching logic or you can specify one.

DIAMOND performs automatic authorization/claim matching logic during claim header processing. This logic includes automatic matching using either keyword MCHRL (Authorization/Claim Match Rules) match rules or, for covering providers, hard coded match logic. You can also manually select a matching authorization when entering or updating a claim header. See Authorization Claim Links, How DIAMOND Thinks and MCHRL, Authorization/Claims Matching Rules for more information on automatic and manual authorization/claim matching.

If a matching authorization is found or selected, DIAMOND searches for a keyword CDAPL (Claim Detail Auth Proc Link) rule that matches the claim’s LOB/Auth Type/Effective date range combination. If a matching CDAPL rule is found, DIAMOND performs authorization/claim detail linking and pricing when processing the claim detail lines. DIAMOND only evaluates the authorization procedure detail records of the matching authorization on the claim header for linking at the claim detail level.

When CADPL rules are evaluated, DIAMOND reviews the CADPR Rule ID to determine whether:

  • An exact match between claim procedure and authorization procedure is required

  • A match within a range of procedure codes

  • A possible match with an out of range procedure

If an exact match is required and this criteria is met, DIAMOND determines whether a price is provided for the authorized procedure and the authorization status. If there is an assigned dollar amount (greater than 0$) on the PRCDR table and the status is a match, the DIAMOND will perform the pay action. In this case, DIAMOND will not evaluate that the procedure ranges.

Handling Multiple Matching Authorizations

If the claim header matches two or more authorizations and you select one as the matching authorization, DIAMOND performs authorization/claim detail linking and pricing using that selected authorization. However, it is possible that none of the procedure codes on the selected authorization’s procedure detail records match the procedure codes on any of the claim detail lines.

When this happens, DIAMOND issues a warning message indicating no procedure code matches were found using the authorization selected on the claim header. You then have the option to go back to the header and use Special menu command Authorization Match/Waive, option Reapply Match to choose another authorization. Then you can re-price the claim using Special menu command Pricing and Adjudication to perform authorization/claim detail linking and pricing using the newly selected authorization.

Tip   You define procedure matching criteria using keyword CDAPR (Claim Detail Auth Proc Rules). You then include the Rule ID of the desired CDAPR rule on the CDAPL processing rule.

Including Institutional Claims Procedure Code Records

For institutional claims, you must flag the procedure code records to include in authorization/claim detail linking and pricing. To flag a procedure code record for inclusion, use keyword PROCD (Procedure Codes) and select the Inst. Auth Detail Link check box.

Performing Authorization/Claim Detail Linking and Pricing

Authorization/claim detail linking and pricing logic proceeds as follows:

  1. When a matching authorization has been found and you enter or update a claim detail line, DIAMOND searches for a matching CDAPL rule. To match, the CDAPL rule LOB/Auth Type/effective date range combination must match the claim header, claim detail line, and the matching authorization as follows:

  2. The CDAPL LOB must match the member’s LOB on the claim header

  3. The CDAPL Auth Type must match the Auth Type of the matching authorization on the claim header

  4. The CDAPL effective date range must include the from Date on the claim detail line

  5. The CDAPL effective date range must also include the Request or Admit date and the Expire or Discharge date (both determined by parameter DUPAUTHRULE2) on the matching authorization

  1. If a matching CDAPL rule is not found, DIAMOND uses standard pricing logic to price the claim. If a matching CDAPL rule is found, authorization/claim detail linking and pricing proceeds for that claim detail line using the CDAPL rule’s processing rules as follows.

  2. DIAMOND evaluates the first and next processing rule on the CDAPL rule in order. Each CDAPL rule includes one or more processing rules. The Process Order on each processing rule determines the order of processing.

DIAMOND searches through the processing rules until a rule is found whose effective date range matches the claim detail line and the matching authorization, as follows:

  • The processing rule’s effective date range must include the from Date on the claim detail line

  • The processing rule’s effective date range must also include the Request or Admit date and the Expire or Discharge date (both determined by parameter DUPAUTHRULE2) on the matching authorization

Tip   If the processing rule does not include an Effective Date or Term Date, the rule is effective for the CDAPL rule effective date range.

If an applicable rule is found, it is applied as follows and no further processing rules are evaluated. If no prior applicable rule is found, DIAMOND applies the default processing rule you defined to provide a graceful landing.

  1. If an applicable CDAPL processing rule is found, DIAMOND searches for PRCDR authorization procedure details that match the submitted procedures on the claim detail lines. The procedures must match in accord with the CDAPL processing rule procedure match criteria.

CDAPL processing rule procedure match criteria include:

  • CDAPR rule criteria

  • Include all (PRCRA) range rows processing

  • Match on details for Inpatient Stay

CDAPL processing rule procedure match criteria are applied as follows.

CDAPR rule criteria:

  • CDAPL processing rule field Rule ID identifies the applicable CDAPR rule. CDAPR rule criteria include: Submt vs Auth Proc Code, Price on Auth Proc, and Auth Proc Status. These match criteria are applied as follows.

  • Submt vs Auth Proc Code: The submitted procedure code on the claim detail line must match the procedure code on the authorization procedure record. The type of match is defined by the Submt vs Auth Proc Code value.

Submt vs Auth Proc Code valid values are: E - Exact match, I - In-range or O - Out of range. If the value is I - In-range or O - Out of range, DIAMOND uses the PRCRA range record identified by the CDAPL processing rule field Range ID in its evaluation. The Range ID corresponds to the PRCRA Auth Proc Range ID. If no Range ID is entered, DIAMOND assumes the procedure code match must be an exact match, regardless of the Submt vs Auth Proc Code field value.

  • Price on Auth Proc: If this check box is selected, the authorization procedure detail record must have an Auth Amount greater than $0.00 to match. If the check box is clear, the authorization procedure detail record can have any Auth Amount - equal to zero, greater than zero, or less than zero - and still match.

  • Auth Proc Status: To match, the authorization procedure detail record must have an authorization Status that matches the Auth Proc Status.

Include all (PRCRA) range rows processing:

  • If the processing rule’s Incld all Rng rows check box is selected, the submitted procedure code and authorized procedure code must also fall within the same range row of the PCRA procedure ranges to be considered a match.

Match on details for Inpatient Stay:

  • The CDAPL Match on details for Inpatient Stay check box specifies whether to search for matching authorization procedure detail records for both institutional and professional services claims matching an institutional stay authorization.

If this check box is selected, DIAMOND searches for matching authorization procedure details for both institutional and professional services claims matching an inpatient stay authorization. For this to occur for professional services claims, the Facility ID on the professional services claim must match the Provider ID on the matching institutional stay authorization.

If the check box is clear, DIAMOND only searches for matching authorization procedure details for institutional claims matching an inpatient stay authorization. No search is done for professional services claims matching the inpatient stay authorization.

  1. If the procedure on a claim line matches an authorization procedure record based on the processing rule’s procedure match criteria, DIAMOND checks if the authorized quantity and authorized amount available for the procedure are sufficient to cover the quantity and amount of the claim detail line, as follows.

Note   The PRCDR fields Actual Qty and Actual Amt reflect the quantity or amount already used by linked claim detail lines. These fields are updated whenever you process an additional linked claim detail line.

Further, if a claim line is deleted or reversed, the PRCDR Actual Qty and Actual Amt fields are updated as well, decreasing by the quantity or allowed amount of the claim line. This additional quantity or amount is then available for use on other claims that link to these authorization procedure detail records. A claim line can be paid and reversed using, for example, PSCLM command Reverse Line or keyword CVSUP (Clear, Void, Stop, Update).

Warning!   DIAMOND updates PRCDR fields Actual Qty and Actual Amt as described above only as long when the status on both the authorization header and the PRCDR record is open.

  • DIAMOND calculates the available authorized quantity for the matching PRCDR authorization procedure record as the PRCDR Auth Qty (the total authorized quantity) minus the PRCDR Actual Qty (the quantity used by other claim detail lines to-date).

  • DIAMOND compares the available authorized quantity to the claim detail line quantity and takes the following action:

  • If the claim detail line quantity is less than or equal to the available authorized quantity, DIAMOND increases the PRCDR Actual Qty by the amount of the claim detail line quantity.

  • If the claim detail line quantity is greater than the available authorized quantity, DIAMOND applies the hold or deny Action and Reason code from the CDAPL rule’s Exceed Auth Detail Days/Visits fields to the claim line.

  • DIAMOND calculates the available authorized amount for the matching PRCDR authorization procedure record as the PRCDR Auth Amt (the total authorized amount) minus the PRCDR Actual Amt (the amount used by other claim detail lines to-date).

  • DIAMOND compares the available authorized amount to the claim detail line amount and takes the following action:

  • If the claim detail line amount is less than or equal to the available authorized amount, DIAMOND increases the PRCDR Actual Amt by the amount of the claim detail line amount.

  • If the claim detail line amount is greater than the available authorized amount, DIAMOND applies the hold or deny Action and Reason code from the CDAPL rule’s Exceed Auth Detail Amount fields to the claim line.

  1. If multiple matching authorization procedure detail records are found, DIAMOND uses the CDAPL fields Tie-Breaker on Auth Price and Tie-Breaker on Auth Proc to select a single authorization detail for further processing.

  2. Tie-Breaker on Auth Price specifies which of two or more matching in-range priced procedure codes to use to price the claim detail line, either the highest or lowest based on price.

  3. Tie-Breaker on Auth Proc specifies which of two or more matching in-range priced procedure codes to use to price the claim detail line, either the highest order or lowest order procedure code.

  1. Once a single authorization detail record that matches the processing rule is found or determined by tie-breaker, DIAMOND applies the Pay Action on the rule to the claim detail line. Depending on the Pay Action, DIAMOND also assigns reason codes using the values in the processing rule fields Allowed Reason and Hld/Dny Reason. For more information on Pay Actions and subsequent processing, see the sections CDAPL Pay Actions and Institutional Claims - Whole Claim Pricing and Special Processing.

In applying the Pay Actions that use the authorization price, DIAMOND processes a claim detail line submitted with a modifier 1 code and/or modifier 2 code in a special manner.

  • When a claim detail line is submitted with a modifier 1 and/or modifier 2 code, DIAMOND cannot price the detail line with submitted modifiers when the Pay Action uses the authorization price. So instead DIAMOND uses the matching Authorization detail price as the allowed amount on the claim detail line. DIAMOND also applies the action and reason code from the CDAPL rule fields Auth Price used with Submt Modifier - Action and Auth Price used with Submt Modifier - Reason to the claim detail line.

  • If you hold or deny a claim using these fields, you can then review the claim line to ensure that the authorization priced line is correct without applying the submitted modifier code(s).

Authorization Overall Status Override

The Overall Status on the authorization master overrides the processing of the authorization claim detail linking. If the matching authorization’s status is held or deny, the whole claim is held or denied based on the action defined in the keyword ARULE.

If the matching authorization’s Overall Status is Deny, all of the claim lines deny also. The status on the PRCDR record is not evaluated when the authorization master status is denied.

If the matching authorization’s Overall Status is Held, depending on the action defined in the keyword ARULE, the claim is held at the header level. The PRCDR records are still processed and evaluated, but the claim is put on hold until this hold is released.

Special Menu Command Pricing and Adjudication

The authorization claim detail linking and pricing process is included in the existing Special menu command Pricing and Adjudication. When you select an option that includes pricing, DIAMOND automatically performs authorization/claim detail linking and pricing.

Warning!   If a claim line was previously manually priced and you re-price using Special menu command Pricing and Adjudication, DIAMOND performs authorization/claim detail linking and pricing and overrides the manual price.

CDAPL Pay Actions

The authorization/claim detail linking and pricing process includes the following CDAPL Pay Actions:

  • 1 - Price with Standard Provider Pricing

Price the claim detail line using standard provider pricing and apply the allowed reason code from the applicable price rule.

  • 2 - Price with Authorized Price

If there is a price on the matching authorization PRCDR detail record, use that price to override the allowed amount on the claim detail line and use the CDAPL rule allowed reason code.

  • 3 - Price with Authorized Procedure

If there is not a price on the matching authorization PRCDR detail record, price the PRCDR procedure code and use that price to override the allowed amount on the detail claim line. Apply the allowed reason code from the allowed reason table.

Note   For this option, DIAMOND prices the claim using the authorized PRCDR procedure code. However, DIAMOND still uses the submitted procedure code on the claim detail line to assign a medical definition. The adjudication process takes place starting with the allowed amount on the claim detail line.

  • 4 - Price and Deny with Standard Provider Pricing

If the procedure Status on the PRCDR record is Deny, deny the detail claim line with the standard provider pricing allowed amount and allowed reason code and use the deny reason code in the CDAPL rule Hld/Dny Reason field.

  • 5 - Price and Deny with Authorized Price

If the procedure Status on the PRCDR record is Deny, and there is a price on the authorization, use that price to override the allowed amount on the claim detail line and use the allowed reason code from the CDAPL rule Allowed Reason field. Deny the claim detail line and use the deny reason code in the CDAPL Hld/Dny Reason field.

  • 6 - Price and Deny with Authorized Procedure

If the procedure Status on the PRCDR record is deny, and there is not a price on the authorization, price the procedure code on the auth and use that price to override the allowed amount on the detail claim line. Apply the allowed reason code from the allowed reason table. Deny the claim detail line with the reason code in the CDAPL rule Hld/Dny Reason field.

Note   DIAMOND prices the claim using the authorized procedure code and uses this amount as the allowed amount on the claim line detail. However, DIAMOND still uses the submitted procedure code on the claim detail line to assign a medical definition. The adjudication process starts from the Allowed Amount on the claim detail line.

  • 7 - Price and Hold with Standard Provider Pricing

If the procedure Status on the PRCDR record is hold, hold the claim detail line with the standard provider pricing allowed amount and allowed reason code, and apply the hold reason code from the CDAPL rule Hld/Dny field.

  • 8 - Price and Hold with Authorized Price

If the procedure Status on the PRCDR record is hold, and there is a price on the authorization, use that price to override the allowed amount on the claim detail line and use the allowed reason code from the CDAPL rule Allowed Reason field. Hold the claim detail line, and use the hold reason code from the CDAPL rule Hld/Dny Reason field.

  • 9 - Price and Hold with Authorized Procedure

If the procedure Status on the PRCDR record is hold, and there is not a price on the authorization, price the procedure code on the authorization and use that price to override the allowed amount on the claim detail line. Apply the allowed reason code from the CDAPL rule Allowed Reason field. Hold the claim detail line with the reason code from the CDAPL rule Hld/Dny Reason field.

Note that DIAMOND prices the claim using the authorized procedure code and uses this amount as the allowed amount on the claim detail line. However, DIAMOND will still uses the submitted procedure code on the claim detail line to assign a medical definition. The adjudication process starts from the allowed amount on the claim detail line.

Institutional Claims - Whole Claim Pricing and Special Processing

Authorization/Claim Detail Linking and Pricing for institutional claims proceeds as described in the Processing Logic section, except when the claim is being whole claim priced. Special processing for whole claim pricing is detailed in Performing Authorization/Claim Detail Linking and Pricing for Whole Claim Priced Claims.

For inpatient institutional claims, the only services that typically need an authorization are level of care (UB92 bed day codes) and carve outs. Other ancillary codes on institutional claims do not need to be specifically authorized, so you can choose to not include them in the authorization procedure code records.

For outpatient institutional claims, one or more of the procedures can require an authorization at the detail level.

Only appropriately flagged procedure code records are included in Institutional claims authorization/claim detail linking and pricing. Flag the appropriate procedure code records for inclusion using PROCD check box Inst. Auth Detail Link.

Performing Authorization/Claim Detail Linking and Pricing for Whole Claim Priced Claims

When the claim is being whole claim priced, DIAMOND performs different authorization/claim detail linking and pricing logic. In addition the specific differences depend on the CPAPL rule Pay Action.

  • When the CDAPL rule Pay action is Price with the Standard DIAMOND pricing, DIAMOND performs standard pricing and adjudication.

  • When the CDAPL rule Pay Action is Price with Authorized Price, DIAMOND performs the following logic:

  1. If the PROCD record for the procedure code on the claim detail line has the flag Include In Detail Auth Link as Y, and the procedure code is an exact match or within the procedure code range of the PRCDR record based upon the criteria specified on the CDAPR rule, DIAMOND inserts the Auth Amt from the PRCDR record into the Auth Price column on the claim detail table.

  2. DIAMOND continues to price and adjudicate the claim detail line using standard DIAMOND Whole Claim Pricing rules, applying the claim status of I - Informational.

  3. Once all the claim detail lines have been entered, you perform Whole Claim pricing (Ctrl F8, or through the DDS API transaction) and DIAMOND continues to generate the applicable Whole Claim Pricing system generated line(s) of base line(s), cap and outlier lines.

  4. As part of Whole Claim pricing, DIAMOND evaluates any Authorization Price that resulted from a claim detail authorization detail record match as follows.

  5. If there are one or more amounts in the Auth Price column for each of the informational detail lines associated with the Whole Claim Priced claim, DIAMOND sums the Auth Price columns for all the Informational lines for that specific claim to calculate a Temporary Auth Price sum.

  6. When there is an amount in the Temporary Auth Price sum DIAMOND performs a second calculation. DIAMOND sums the system generated lines allowed amounts to calculate a Temporary Sys Gen totals amount. This Temporary Sys Gen total will include the applicable system generated lines including the base lines, cap and outlier lines.

  7. DIAMOND then compares the Temporary Auth Price Sum to the Temporary Sys Gen sum to determine if additional processing is required.

If the Temporary Auth Price Sum is equal to the Temporary Sys Gen Sum, then DIAMOND performs no additional processing and the standard DIAMOND WCP pricing remains.

If the Temporary Auth Price Sum is less than the Temporary Sys Gen Sum DIAMOND generates an adjustment claim detail line. This adjustment claim detail amount is a negative adjustment line that DIAMOND generates to ensure that the Grand Total Sum of the Allowed Amounts equals the Temporary Auth Price Sum. You establish the system generated code used for this adjustment line using system parameter of CDAPLWCPSYSG (Claim Detail authorization procedure link whole claim price system generated code). You must also configure the CDAPLWCPSYSG code within the DIAMOND support files.

If the Temporary Auth Price Sum is greater than the Temporary Sys Gen Sum, DIAMOND also generates an adjustment claim detail. The adjustment claim line detail amount in this case is a positive amount that DIAMOND generates to ensure that the Grand Total Sum of the Allowed Amounts from the equal the Temporary Auth Price Sum. You again establish the system generated code used for this adjustment line using system parameter of CDAPLWCPSYSG. You must also configure the CDAPLWCPSYSG code within the DIAMOND support files.

  1. When a CDAPL system generated adjustment line is created, DIAMOND also assigns an internal system revenue code. DIAMOND uses this internal system revenue code any outgoing HIPAA transaction set instead of the internally defined DIAMOND system generated code.

  2. System-generated codes (sysgen) are user-defined codes used only for whole claim pricing in DIAMOND. You define these codes using keyword PROCD (Procedure Code master) and enter them into a PRULW record to be the system-generated code for the payable line when Whole Claim Pricing (WCP) has completed.

  3. In order to be HIPAA compliant, The CDAPL adjustment line inherits the highest sys rev code that has already been assigned to the base line(s) for each specific claim. This is also the same sys rev code used on any cap or outlier line during the Whole Claim pricing process.

Tie Breakers

DIAMOND uses a Tie Breaker rule when a CDAPL rule successfully matches more than one PRCDR authorization detail record of a matching authorization. This can occur when the CDAPL rule includes a CDAPR rule that uses a Submt vs Auth Proc Code value of I - In Range and the procedure code on more than one PRCDR authorization detail record falls within the PRCRA range included on the CDAPL rule.

The two CDAPL tiebreaker options are:

  • Tie-Breaker on Auth Price

  • Tie-Breaker on Auth Proc

Tie-Breaker on Auth Price

DIAMOND uses the CDAPLTie-Breaker on Auth Price field when:

  • The Pay Action specified on CDAPL for the applied CDAPR is set to Price with Authorized Price, Price and Deny with Authorized Price, or Price and Hold with Authorized Price

  • DIAMOND finds two or more priced authorization procedure records that are in the range for the submitted procedure code.

DIAMOND applies the CDAPLTie-Breaker on Auth Price field value to determine a single matching PRCDR authorization procedure detail for further processing. Valid values are:

  • H - Use Highest Auth Price

  • L - Use Lowest Auth Price

DIAMOND uses the CDAPLTie-Breaker on Auth Proc field when:

  • The Pay Action specified on CDAPL for the applied CDAPR is set to Price with Authorized Procedure, Price and Deny with Authorized Procedure, or Price and Hold with Authorized Procedure

  • DIAMOND finds two or more authorization procedure records are located that are in the range for the submitted procedure code.

DIAMOND applies the CDAPLTie-Breaker on Auth Procedure field value to determine a single matching PRCDR authorization procedure detail for further processing. Valid values are:

  • H - Use Highest Order Auth Procedure

  • L - Use Lowest Order Auth Procedure

APC Pricing and Authorization/Claim Detail Linking and Pricing

Ambulatory Payment Classification (APC) pricing can override other pricing on claim detail lines. Do not use both APC pricing and authorization/claim detail linking and pricing.

Third Party Claim Audit Tools and Authorization/Claim Detail Linking and Pricing

Third party claim audit tools override pricing resulting from authorization/claim detail linking and pricing logic. Do not use both third party claim audit tools and authorization/claim detail linking and pricing.
