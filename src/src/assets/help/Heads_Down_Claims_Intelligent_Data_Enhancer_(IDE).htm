Overview

DIAMOND’s Intelligent Data Enhancer (IDE) module enhances the text files you:

  • Create during Heads Down Claims entry. See Heads Down Claims Entry User Documentation for additional information.

  • Receive from external sources, such as clearing houses after they have been formatted using Sterling® or other third party software.

The IDE module performs the following searches, verifications and determinations using the "minimal" data that you entered during Heads Down Claims entry:

  • Member Search and Verification (first)

  • Vendor Search and Verification (second)

  • Provider Search and Verification (third)

  • Claim Total Billed vs. Total Paid Verification

  • Primary Date of Service Determination

As a result of the above searches, verifications and determinations, the IDE module attempts to determine the additional data that must be included in a Claims Batch Process "control" file. See Submitter Maintenance (SUBMT), Process EDI (PREDI) and EDI Claims Batch Process User Documentation for additional information.

Heads Down Claims entry and the IDE module allow you to enter a high volume of claims and reduce significantly the number of Claims Batch Process "critical errors" that can occur because of missing data.

Source Files

A claims text file is created on your local workstation as you enter claims using the Heads Down Claims entry window. Your Systems Administration group gathers all of the completed text files from various workstations and merges them into one IDE source (input) file.

The merged IDE source file is loaded (SQL*LOADER) into the following Oracle "temporary" work tables:

  • IDE_PROFSVC_CLAIM_HEADER_W

  • IDE_PROFSVC_CLAIM_DETAIL_W

The IDE module reads each claim in the worktables and enhances (adds to) the data captured during Heads Down Claims entry.

When all of the claim records in the work tables are processed, the IDE module:

  • Creates an enhanced claims file that becomes the input file to DIAMOND’s Claims Batch Process. See Heads Down Claims Entry and Heads Down Claims - Process User Documentation for additional information.

  • Deletes all claim records in the worktables.

You process the enhanced claims file using Diamond’s Process EDI (PREDI) function. See Heads Down Claims - Process and PREDI User Documentation for additional information.

During the IDE process, processing messages are written to the following message table:

  • IDE_ERROR_MESSAGES

You can use SQL to access this table for reporting purposes.

Member Search

As you enter the Insured’s ID during Heads Down Claim Entry:

IF:

THEN:

The Insured’s ID is 9 digits, a possible Social Security Number (SSN)…

The Insured’s ID (9 digits) is written to the Subscriber ID and SSN fields on the claims text file.

The Insured’s ID is not 9 digits or is alphanumeric, not a possible SSN…

The Insured’s ID is written only to the Subscriber ID field on the claims text file.

The Insured’s ID is not entered…

Nothing is written to the Subscriber ID field on the claims text file.

As the IDE module performs its unique Member Search:

IF:

THEN:

The Subscriber ID on the claims text file is 9 digits, a possible Social Security Number (SSN)…

Compare the text file Subscriber ID to Subscriber IDs in DIAMOND’s Member Master table.

Single match found when text file Subscriber ID compared to Subscriber IDs in DIAMOND…

1. Store the respective DIAMOND Person Number in the worktable.

2. Store the respective DIAMOND Person Number in the worktable.

3. Proceed to Member_Eligibility Search.

No match found when text file Subscriber ID compared to Subscriber IDs in DIAMOND…

Try LastName_FirstName_SSN Search.

Multiple matches found when text file Subscriber ID compared to Subscriber IDs in DIAMOND…

Try PersonNumber Search.

The Subscriber ID on the claims text file is not 9 digits or is alphanumeric — not a possible Social Security Number (SSN)…

Compare the text file Subscriber ID to Subscriber IDs in DIAMOND’s Member Master table.

Single match found when text file Subscriber ID compared to Subscriber IDs in DIAMOND…

1. Store Subscriber ID in worktable.

2. Store the respective DIAMOND Person Number in the worktable.

3. Proceed to Member_Eligibility Search.

No match found when text file Subscriber ID compared to Subscriber IDs in DIAMOND…

Try LastName_FirstName_SSN Search.

Multiple matches found when text file Subscriber ID compared to Subscriber IDs in DIAMOND…

Try PersonNumber Search.

The Subscriber ID on the claims text file is blank…

Try LastName_FirstName_SSN Search.

Member_Eligibility Search

The IDE module reads the primary date of service on the claim, checks the member’s eligibility on DIAMOND and:

IF:

THEN:

The member is eligible…

1. Store the member’s line of business, IPA and panel in the temporary worktable. This information may be used during the IDE Provider Search.

2. Proceed to IDE Vendor Search.

If the member is not eligible…

Try IDE Vendor Search.

Vendor Search

The IDE module performs a unique Vendor Search if a DIAMOND member number and person number have been identified for the incoming claim.

IF:

THEN:

DIAMOND subscriber ID/person number found (IDE Member Search successful)…

Check if Tax ID was entered during Heads Down Claims entry.

Tax ID not entered during Heads Down Claim entry…

Try Short_Name Search.

Tax ID entered during Heads Down Claim entry…

1. Strip dashes from entered Tax ID.

2. Try to match Tax ID to the Vendor ID on DIAMOND’s Vendor Master file.

No match found on Vendor ID…

Try IRS_TaxID Search.

Single match found Vendor ID…

1. Store Vendor ID.

2. Search for vendor (street) address on incoming claim.

Proceed to IDE Provider Search.

Provider Search

The IDE module performs a unique Provider Search if a DIAMOND member number, person number and Vendor ID have been identified for the incoming claim. The IDE module:

  • retrieves the stored Vendor ID and the DIAMOND index to the Vendor Address (if applicable).

  • reads the DIAMOND Provider Contract Vendor file and identifies all provider contract vendor records associated with the retrieved vendor ID, and:

    • IF:

    THEN:

    No DIAMOND provider contract vendor records found to be associated with the retrieved vendor ID…

    IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report as "provider not found."

    Multiple DIAMOND provider contract vendor records found to be associated with the retrieved vendor ID…

    Try Provider_ShortName_MiddleInitial Search.

    Single DIAMOND provider contract vendor record found to be associated with the retrieved vendor ID…

    Verify the match:

    1. Concatenate provider Lst Nm and Frst Nm, entered during Heads Down Claims entry.

    2. Use first 15 characters of concatenated provider LstNmFrstNm and, based on the single provider contract vendor record, try to match LstNmFrstNm to the corresponding provider Short Name on DIAMOND’s Provider Master file.

    Single verification match found on provider LstNmFrstNm…

    Provider Search is complete:

    1. Store Provider ID.

    2. Process next incoming claim record.

    No verification match found on provider LstNmFrstNm…

    IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

    Provider_ShortName_MiddleInitial Search

    IF:

    THEN:

    Multiple DIAMOND provider contract vendor records found to be associated with the retrieved vendor ID…

    1. Concatenate provider Lst Nm, Frst Nm, and MI entered during Heads Down Claims entry.

    2. Use first 15 characters of concatenated provider LstNmFrstNmMI and, based on the provider contract vendor records, try to match provider LstNmFrstNmMI to the corresponding provider Short Name on DIAMOND’s Provider Master file.

    Multiple DIAMOND provider contract vendor record matches remain after using LstNmFrstNmMI…

    Try Add_Provider_ZIP Search.

    Single DIAMOND provider contract vendor record found after using LstNmFrstNmMI…

    Provider Search is complete:

    1. Store Provider ID.

    2. Process next incoming claim record.

    No DIAMOND provider contract vendor record found after using LstNmFrstNmMI…

    1. Concatenate only the provider Lst Nm and Frst Nm (no MI) entered during Heads Down Claims entry.

    2. Use first 15 characters of concatenated provider LstNmFrstNm and, based on the provider contract vendor record(s), try to match provider LstNmFrstNm to the corresponding provider Short Name on DIAMOND’s Provider Master file.

    Multiple DIAMOND provider contract vendor records matches remain after using LstNmFrstNm…

    Try Add_Provider_ZIP Search.

    Single DIAMOND provider contract vendor record found after using LstNmFrstNm…

    Provider Search is complete:

    1. Store Provider ID.

    2. Process next incoming claim record.

    No DIAMOND provider contract vendor record found after using LstNmFrstNm…

    1. Concatenate first 3 characters of the provider Lst Nm and the first character of the provider Frst Nm entered during Heads Down Claims entry.

    2. Use the concatenated provider LstNm(3)FrstNm(1) and, based on the provider contract vendor records, try to match LstNm(3)FrstNm(1) to a corresponding provider shortened name - Last Name (3), First Name (1) - on DIAMOND’s Provider Master file.

    No DIAMOND provider contract vendor record found after using on provider Short Name using LstNm(3)FrstNm(1)…

    IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

    Multiple DIAMOND provider contract vendor record matches remain after using LstNm(3)FrstNm(1)…

    Try Add_Provider_ZIP Search.

    Single DIAMOND provider contract vendor record found after using LstNm(3)FrstNm(1)…

    Provider Search is complete:

    1. Store Provider ID.

    2. Process next incoming claim record.

    Address Line 1 Search

    The IDE module invokes the Address Line 1 Search when multiple Vendor records or multiple Provider Contract Vendor records (multiple matches) have been found during respective IDE Vendor or IDE Provider searches. The Address Line 1 search:

    • compares the vendor address entered during Heads Down Claims entry to DIAMOND’s Vendor Address records during the IDE Vendor Search.

    • compares the provider address entered during Heads Down Claims entry to DIAMOND’s Provider Contract Vendor address and Provider Address records during the IDE Provider Search.

      • IF:

      THEN:

      Multiple matches found on Vendor Short Name/Vendor State/Vendor ZIP (more than one Vendor Master records match on Vendor Short Name and at least one Vendor Address record matches on State and ZIP)…

      OR

      Multiple DIAMOND provider contract vendor record matches remain after using:

      • LstNmFrstNmMI and ZIP…

      OR

      • LstNmFrstNm and ZIP…

      OR

      • LstNm(3)FrstNm(1) and ZIP…

      Compare only the numeric values (numeric string) from the Vendor or Provider Addr 1 entered during Heads Down Claims entry to the numeric values on the corresponding vendor or provider Addr 1 on DIAMOND’s Vendor or Provider Address file.

      Single match found using numeric string from Addr 1…

      Compare first >2 character word in Addr 1 (not including compass indicators, punctuation or special characters) entered during Heads Down Claims entry to the first >2 character word in the corresponding vendor or provider Addr 1 on DIAMOND’s Vendor or Provider Address file.

      Single match found using Addr 1 >2 character word…

      Return to IDE Vendor Search or IDE Provider Search (assume a match has occurred and vendor or provider is identified).

      No match found using Addr 1 >2 character word…

      IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

      No match or multiple matches found using numeric string Addr 1…

      1. Remove punctuation, special characters, and compass indicators from the Vendor or Provider Addr 1 entered during Heads down Claim entry and from the matching DIAMOND Addr 1 records.

      2. Compare initial numeric strings and the first >2 character word in Addr 1.

      Single match found using numeric string and >2 character word from Addr 1…

      Return to IDE Vendor Search or IDE Provider Search (assume a match has occurred and vendor or provider is identified).

      Multiple matches found using numeric string and <2 character word from Addr 1…

      IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

      No match found using numeric string and >2 character word from Addr 1…

      Compare remaining records using first numeric string from Addr 1 only (don’t include >2 character word).

      Single match found using first numeric string from Addr 1 only.…

      Return to IDE Vendor Search or IDE Provider Search (assume a match has occurred and vendor or provider is identified).

      No match found using first numeric string from Addr 1 only…

      IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

      Multiple matches found using first numeric string from Addr 1 only (during IDE Vendor Search)…

      IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

      Multiple matches found using first numeric string from Addr 1 only (during IDE Provider Search)…

      Try IDE Provider Contract Search.

      Provider Contract Search

      The IDE module invokes the Provider Contract Search when multiple Provider Contract Vendor records (multiple matches) have been found during the IDE Provider search. These multiple matches have the same:

      • Vendor

      • Provider Name

      • ZIP Code

      • Address (Address Line 1)

        • IF:

        THEN:

        Multiple matches found using Addr 1 first numeric string only (during IDE Provider Search, IDE Address Line 1 Search)…

        Use member eligibility data retrieved/stored during the IDE Member Search (for example, Panel + IPA + LOB).

        Member eligibility data has not been stored…

        IDE module rejects the claim (member not eligible). The rejected claim is reported on the Claims Batch Process > Critical Error Report.

        Member eligibility data has been stored…

        1. Retrieve the member’s Line of Business based on the stored eligibility record.

        2. Retrieve the first Contract Search Criteria — Contract Search Type (for example, Provider + LOB + Panel) from the Line of Business record.

        3. Apply the Contract Search Type to each multiple match to find a Provider Contract that matches the stored eligibility record.

        Single contract match found using Contract Search Type…

        Provider Search is complete:

        1. Store Provider ID.

        2. Process next incoming claim record.

        Multiple contract matches found using Contract Search Type…

        IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

        No contract match found using Contract Search Type…

        1. Retrieve the next successive Contract Search Criteria > Contract Search Type (for example, Provider + LOB + Panel) from the Line of Business record.

        2. Apply each successive Contract Search Type to each multiple match to find a Provider Contract that matches the stored eligibility record.

        No contract match or multiple contract matches found using successive Contract Search Types…

        IDE module rejects the claim. The rejected claim is reported on the Claims Batch Process > Critical Error Report.

        Single contract match found using successive Contract Search Types…

        Provider Search is complete:

        1. Store Provider ID.

        2. Process next incoming claim record.
